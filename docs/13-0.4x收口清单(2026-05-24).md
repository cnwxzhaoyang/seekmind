# SeekMind 0.4.x 收口清单

本文档用于收束 `0.4.x` 最后几项还未完全闭合的能力。

当前状态下，混合检索、语义开关、阈值、query 改写、历史扩展、收藏/最近打开偏好加权、结果解释都已经有了雏形或可用版本。剩下的工作不再是“扩功能”，而是把排序、解释和回退收得更稳。

---

## 1. 收口目标

```text
1. 排序权重更细，但仍然可控。
2. rerank 前后变化可见。
3. 分项得分结构化返回。
4. 语义回退有明确终态提示。
```

---

## 2. 剩余项拆解

### 2.1 可配置权重再细一层

#### 目标

```text
把“当前是怎么排出来的”从经验规则，推进到可调的分项权重。
```

#### 需要做什么

```text
1. 把关键词分、语义分、标题分、文件名分、偏好分拆开。
2. 让它们有独立的权重或档位。
3. 默认值保持保守，不能让语义或偏好压过精确命中。
```

#### 验收什么

```text
1. 同一条结果在不同权重下会变化。
2. 精确关键词命中仍然优先。
3. 调整后前端能看见变化，不需要猜。
```

---

### 2.2 rerank 前后名次变化

#### 目标

```text
让用户知道某条结果是“原本就在前面”，还是“被 rerank 拉上来的”。
```

#### 需要做什么

```text
1. 保留原始排序。
2. 保留 rerank 后排序。
3. 返回名次变化值，至少能显示前移 / 后移多少位。
```

#### 验收什么

```text
1. 调试面板能看到 rerank 前后变化。
2. 结果卡能展示“被重排提升”或“被重排压后”。
3. rerank 失败不影响主结果。
```

---

### 2.3 分项得分结构化输出

#### 目标

```text
把“排序解释”从一串文本，变成可拆的结构化字段。
```

#### 需要做什么

```text
1. 返回 keyword_score。
2. 返回 semantic_score。
3. 返回 title_score。
4. 返回 filename_score。
5. 返回 preference_score。
6. 保留 summary 作为给人看的总览。
```

#### 验收什么

```text
1. 前端可以按标签展示各分项。
2. 调试面板能直接看出每一项贡献。
3. 后续再加新的权重项时，不需要重写整段解释文本。
```

---

### 2.4 语义回退终态提示

#### 目标

```text
把“结果有了”与“语义链路真的正常”分开。
```

#### 需要做什么

```text
1. 显示本次语义是否真正参与。
2. 显示是否发生回退。
3. 显示回退原因。
4. 把回退状态做成终态标签，而不是只藏在日志里。
```

#### 验收什么

```text
1. 用户能一眼看出本次搜索有没有走语义链路。
2. 语义异常时，全文仍然可用。
3. 回退发生后，前端明确提示，不再让人误以为语义正常。
```

---

## 3. 推荐执行顺序

```text
1. 分项得分结构化输出
2. rerank 前后名次变化
3. 可配置权重再细一层
4. 语义回退终态提示
```

### 原因

```text
1. 先把解释数据做成结构化，后面 UI 才好画。
2. 再把 rerank 的前后变化补出来，验证重排是否有效。
3. 接着才考虑更细的权重配置，避免调参没有可视化依据。
4. 最后把回退终态补齐，作为收口。
```

---

## 4. 0.4.x 收口标准

当下面这些条件满足时，`0.4.x` 就可以认为收口：

```text
1. 混合检索结果稳定。
2. 语义开关、阈值、历史扩展、收藏/最近打开偏好都可控。
3. 结果排序可以解释。
4. rerank 的影响可以观察。
5. 语义回退可见且不影响全文搜索可用性。
```

---

## 5. 收口后的下一步

如果 `0.4.x` 收口完成，后续更适合进入下面方向，而不是继续扩搜索主链路：

```text
1. 搜索体验抛光
2. 结果解释 UI 优化
3. 文档列表 / 切片页细节优化
4. 更稳定的索引任务体验
5. 面向 1.0 的问答能力预研
```
