# Python 侧任务调用经验规范

这份文档用于沉淀 SeekMind 里所有“调用 Python sidecar 处理耗时任务”的统一做法。目标很简单：以后再增加任何 Python 任务，都默认按这套模式实现，避免再次回到“Rust 同步等待 Python 最终返回”导致的超时问题。

## 1. 总原则

所有 Python 侧耗时任务都遵循同一模式：

1. Rust 侧负责发起任务。
2. Python 侧负责执行任务并持续发出进度 event。
3. Rust 侧负责接收 event，更新数据库状态，再把状态推给 UI。
4. Rust 不依赖 Python 一次性返回最终结果来驱动界面。

这条原则适用于：

- 单文档重新切片
- 目录重新索引
- 失败文件重试
- 语义向量重建
- embedding warmup
- 未来所有新增的 Python 任务

## 2. 为什么必须这样做

同步等待 Python 最终结果有几个明确问题：

- 文档越大、任务越多，越容易超时。
- UI 会长时间停留在“初始化 / 等待中”，用户会误判为卡死。
- Python 侧一旦在中间阶段卡住，Rust 只能在最终超时后才知道，丢失了中间状态。
- 后续想支持暂停、恢复、分阶段回显时，同步模式很难扩展。

因此，新的任务接口默认不再设计成“Rust 一次请求，Python 一次返回”的纯同步模式，而是设计成“事件驱动任务流”。

## 3. 推荐链路

推荐链路固定为：

1. Rust 创建 `job_id`。
2. Rust 通过 sidecar 命令把任务发给 Python。
3. Python 持续输出 JSON line event。
4. Rust 逐条消费 event。
5. Rust 更新 `current_task`、失败项、索引状态等持久化状态。
6. Rust 再把状态发给前端。
7. Python 最终返回一个完成响应，Rust 只把它当作结束信号，而不是唯一进度来源。

## 4. 任务协议约定

### 4.1 请求

建议请求字段至少包含：

- `request_id`
- `command`
- `path`
- `options`
- `texts` 或其他任务输入

示例：

```json
{
  "request_id": "uuid",
  "command": "parse_document_stream",
  "path": "/Users/zhaoyang/Documents/example.docx",
  "options": {
    "include_chunks": true,
    "max_chunk_chars": 800,
    "max_chunks": null
  }
}
```

### 4.2 事件

Python 在处理中持续输出事件，每条事件单独一行 JSON，并立即 flush。

建议字段：

- `kind`
- `request_id`
- `event`
- `stage`
- `message`
- `percent`
- `current`
- `total`
- `processed`
- `parser_source`
- `warning`

示例：

```json
{
  "kind": "event",
  "request_id": "uuid",
  "event": "progress",
  "stage": "chunk",
  "message": "正在生成切片",
  "percent": 68,
  "current": "15/22",
  "total": 22,
  "processed": 15,
  "parser_source": "python",
  "warning": null
}
```

### 4.3 最终响应

最终响应只用于告诉 Rust：

- 任务结束了
- 成功还是失败
- 是否带回结构化结果

最终响应不应该承担实时进度职责。

## 5. Rust 侧实现模式

### 5.1 不要直接阻塞等待最终输出

不要再写成这种风格：

- 启动 Python
- 一直 `wait` 最终结果
- 中间没有任何状态回传

这类写法最容易触发“python parser timed out”或者“看起来卡住了”的问题。

### 5.2 要改成流式消费

Rust 侧应使用流式 client：

- `parse_document_stream`
- `embed_texts_stream`

然后在 event 回调里做三件事：

1. 更新数据库里的当前任务状态。
2. 计算并写入进度。
3. 向前端发送对应的 progress event。

### 5.3 每个 event 都要能驱动 UI

前端不要只等最终完成事件。只要 Python 侧发了 event，Rust 就应该能让 UI 看见当前进度。

这意味着 Rust 侧在处理 event 时最好更新：

- 当前阶段
- 当前文件
- 当前块
- 处理百分比
- 任务描述

## 6. Python 侧实现模式

### 6.1 命令要分两类

Python 入口建议把命令拆成两类：

- 普通命令：返回最终结果
- 流式命令：边做边发 event，最后再返回最终结果

例如当前已经有：

- `parse_document_stream`
- `embed_texts_stream`

### 6.2 事件必须及时 flush

Python 每次输出 event 后都要立刻：

- `stdout.write(...)`
- `stdout.write("\n")`
- `stdout.flush()`

否则 Rust 侧即使逻辑正确，也可能因为缓冲区没刷新而看不到实时进度。

### 6.3 不要在 Python 侧隐藏长耗时阶段

像下面这些阶段都应该主动发 event：

- 开始解析
- 提取内容
- 规范化
- 切片
- embedding 生成
- 写入前准备
- 完成

如果 Python 在某个阶段没有任何输出，Rust 侧的流式超时就会把它当成“卡住了”。

## 7. 超时策略

### 7.1 任务超时应看“活动时间”

流式任务不应该只看总耗时，还要看最近一次 event 的时间。

推荐策略：

- 只要 Python 持续发 event，就认为任务仍然活着。
- 如果长时间没有任何 event，再视为超时。

### 7.2 超时前先保证状态可恢复

如果任务失败或超时，Rust 侧至少要做到：

- 保留已处理进度
- 写入失败原因
- 不把旧状态直接抹掉

这样用户重试时不会丢失上下文。

## 9. 新增 Python 任务时的检查清单

新增一个 Python 任务前，先确认下面几点：

1. 是否必须流式输出进度。
2. 是否会处理很多文件、很多 chunk 或很大的单文件。
3. 是否需要在 UI 上展示阶段状态。
4. 是否需要支持失败回退或部分成功。
5. 是否会因为同步等待而容易超时。

如果以上任一项答案是“是”，就应该直接按流式模式设计。

## 10. 常见反模式

下面这些写法尽量不要再出现：

- Rust 调 Python，然后一直等最终 stdout。
- Python 只在最后输出一次结果，中间没有 event。
- Python 做了很久，但没有 `flush()`。
- Rust 收到 Python event 但没有更新 `current_task`。
- UI 只依赖最终结果，不看任务进度。
- 把所有任务都塞进同一个同步 request/response 接口里。

## 11. 结论

以后凡是 Python 侧的耗时任务，优先遵循这条路线：

**Rust 发命令，Python 发 event，Rust 更新状态。**

这是当前最稳的实现方式，也最适合后续继续扩展到新的解析、embedding 和索引任务。
