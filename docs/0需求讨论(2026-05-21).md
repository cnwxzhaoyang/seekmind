## 1. 产品定位可以这样定义

这个 App 的核心能力不是“管理文档”，而是：

> 用户把 PDF、Word、Markdown、TXT、网页剪藏、代码文档等丢给 App，App 自动解析、分段、建立索引。以后用户输入自然语言关键词，就能找到相关文档、相关段落，并快速跳转到原文件。

例如用户搜索：

```
去年关于离线 Maven 仓库的方案
```

即使文档里没有完全出现这句话，只要某些段落写的是：

```
通过 dependency:go-offline 拉取 parent POM、BOM、plugin 依赖，并生成 .offline-repo
```

也应该能搜到。

这就是 **全文检索 + 语义向量检索** 的组合。

## 2. 不要一上来就做纯 AI，应该做“双索引”

建议采用：

```
文件索引
  ├─ 元数据索引：文件名、路径、类型、修改时间
  ├─ 全文索引：文档正文、标题、段落文本
  └─ 语义索引：段落 embedding 向量
```

搜索时走 **混合检索 Hybrid Search**：

```
用户输入关键词
    ↓
全文检索：Tantivy / BM25 / fuzzy
    ↓
语义检索：Embedding + Vector DB
    ↓
结果融合：关键词命中 + 语义相似度 + 最近访问 + 文件权重
    ↓
返回：文档 + 段落 + 高亮 + 打开入口
```

为什么一定要“双索引”？

因为纯语义检索适合“意思相近”，但对精确词不稳定；全文索引适合精确命中、短关键词、文件名、专有名词。比如 `JwtAuthenticationTokenFilter`、`W-8BEN`、`Tauri v2 capabilities` 这种技术词，全文检索会更靠谱。

Tantivy 是 Rust 生态里比较成熟的全文搜索引擎，支持 BM25、布尔查询、短语查询等能力，适合放在你的 Rust/Tauri 本地应用里。[Docs.rs+1](https://docs.rs/tantivy/latest/tantivy/query/index.html?utm_source=chatgpt.com)  
语义检索可以用本地向量数据库，例如 LanceDB 或 Qdrant；LanceDB 支持本地路径嵌入式使用，Qdrant 则是 Rust 实现的向量搜索引擎，也支持本地部署。[Docs.rs+2docs.lancedb.com+2](https://docs.rs/lancedb?utm_source=chatgpt.com)

核心数据流：

```

    ↓

    ↓

    ↓

    ↓
保存 chunk 元数据
    ↓
写入 Tantivy 全文索引
    ↓
生成 embedding
    ↓
写入向量索引
```

## 4. 数据模型要围绕“段落块 Chunk”设计

不要只对“文件”建索引，必须对“段落块”建索引。

建议核心表：

```SQL
documents
- id
- path
- file_name
- file_type
- size
- modified_at
- content_hash
- indexed_at
- title
- status

chunks
- id
- document_id
- chunk_index
- page_no
- heading
- text
- text_hash
- start_offset
- end_offset
- token_count

chunk_vectors
- chunk_id
- embedding_model
- vector_id
```

搜索结果返回的不是单纯文件，而是：

```JSON
{
  "document": "Maven离线仓库方案.docx",
  "path": "/Users/xxx/Documents/...",
  "chunk": "通过 dependency:go-offline 拉取 parent POM、BOM、plugin 依赖...",
  "pageNo": 4,
  "heading": "三、离线依赖导出方案",
  "score": 0.87,
  "matchType": "semantic"
}
```

这样用户才会感觉“真的定位到了内容”。

## 5. 第一版 MVP 不建议做太复杂

你的第一版可以分 3 个阶段。

### 阶段一：全文索引版

先支持这些格式：

```
.txt
.md
.pdf
.docx
.html
.rtf
```

实现能力：

```
1. 用户选择目录
2. 后台扫描文档
3. 提取文本
4. 按段落切块
5. Tantivy 建全文索引
6. 输入关键词后返回文档 + 段落
```

这一版不做语义 embedding，也已经很有价值。

技术选型：

```
全文搜索：Tantivy
元数据存储：SQLite
文档解析：
  - txt / md：直接读取
  - pdf：pdf-extract / lopdf / poppler
  - docx：zip + xml 解析，或 docx-rs
  - html：scraper / html2text
```

---

### 阶段二：语义搜索版

加入 embedding：

```
段落文本 → embedding model → vector → vector database
```

本地 embedding 可以考虑 `fastembed`。它是一个 Rust crate，定位就是本地生成文本向量，提供 `TextEmbedding` 接口。[Docs.rs+1](https://docs.rs/fastembed?utm_source=chatgpt.com)  
向量库可以先选 LanceDB，因为它支持本地路径作为嵌入式存储，比较适合桌面 App，不一定要起一个独立服务。[Docs.rs+1](https://docs.rs/lancedb?utm_source=chatgpt.com)

搜索流程：

```
用户输入 query
    ↓
query embedding
    ↓
向量检索 topK chunks
    ↓
Tantivy 全文检索 topK chunks
    ↓
合并去重
    ↓
重新打分排序
```

---

### 阶段三：体验增强版

加入这些高级能力：

```
1. PDF 页码定位
2. Word 标题层级定位
3. 搜索结果关键词高亮
4. Quick Look 预览
5. 最近访问排序
6. 文档更新自动增量索引
7. 文件删除后自动清理索引

10. 问答式搜索
```

后面甚至可以做成：

```

  - 相关段落

## 6. 关键难点

### 难点一：文档解析

真正麻烦的不是搜索，而是各种文档格式解析。

PDF 最复杂，因为 PDF 不是天然结构化文本。它可能出现：

```
1. 文本顺序混乱
2. 多栏排版
3. 表格无法还原
4. 扫描版 PDF 没有文本
5. 页眉页脚干扰正文
```

第一版可以明确不支持 OCR，只支持可复制文本的 PDF。

---

### 难点二：段落切分

不要简单按固定长度切。

建议优先级：

```
Markdown：按标题层级切
Word：按段落 + 标题样式切
PDF：按页 + 空行 + 文本块切
TXT：按空行切
HTML：按 h1/h2/h3/p/li 切
```

每个 chunk 不宜太短，也不宜太长。

建议：

```
中文：300 ~ 800 字
英文：200 ~ 500 tokens
```

太短语义不完整，太长搜索结果不精准。

---

### 难点三：embedding 模型体积和性能

本地语义搜索必须考虑：

```
1. 模型文件大小
2. 首次加载时间
3. CPU 推理速度
4. 内存占用
5. 中英文效果
6. App Store 沙盒限制
```

你现在 `aseek` 37 万文件索引内存约 210MB，控制得已经不错。加 embedding 后，内存和磁盘占用会明显上升。

一个粗略估算：

```
100,000 个段落
384 维 float32 向量
单条向量约 1.5KB
总向量裸数据约 150MB
```

如果 768 维，就是约 300MB 起步。

所以不要把所有向量常驻内存，必须落盘，用向量数据库或 mmap 方案。

有，而且比单纯文件名搜索更容易做出差异化。

```

你可以定位成：

## 10. 最推荐的落地顺序

我建议你不要一开始就上语义搜索，按这个顺序最稳：

```
第 1 步：文档解析器
第 2 步：Chunk 分段模型
第 3 步：SQLite 保存文档和段落
第 4 步：Tantivy 全文索引
第 5 步：搜索结果返回段落
第 6 步：UI 展示文档 + 命中段落
第 7 步：增量索引
第 8 步：加入 fastembed
第 9 步：加入 LanceDB
第 10 步：全文 + 语义混合排序
```

一句话总结：

> 先把“定位到段落”的全文搜索做好，再做“语义相近也能搜到”。

这样不会陷入 AI 工程泥潭，而且第一版就能发布。

**You:**

rust有关于文档解析的库吗

* * *

**ChatGPT:**

我会按文档类型拆开说，因为 Rust 里没有一个“全格式通吃且效果很好”的官方库，通常是多库组合：文本类直接解析，Office/PDF 分别处理，扫描版再接 OCR。

## 推荐库按文档类型划分

| 类型 | Rust 库 | 适合程度 | 说明 |
| --- | --- | --- | --- |
| TXT / Markdown | 标准库、`pulldown-cmark` | 高 | 纯文本最好处理，Markdown 可以保留标题层级 |
| HTML | `scraper`、`html2text` | 高 | `scraper` 适合按 DOM 提取，`html2text` 适合直接转纯文本。`scraper` 基于 html5ever 和 CSS selector 查询能力。[Docs.rs](https://docs.rs/scraper?utm_source=chatgpt.com) |
| DOCX | `docx-rs`、`docx-parser`、`docx-rust` | 中 | 可以解析 Word 文档，但复杂样式、表格、页眉页脚要自己处理。`docx-rs` 是常见 DOCX crate。[Docs.rs](https://docs.rs/docx-rs?utm_source=chatgpt.com) |
| DOCX / XLSX / PPTX | `undoc`、`oxdoc-core` | 高 | 更适合你的场景，因为目标是**提取文本/Markdown/结构化 JSON**，不是编辑 Office。`undoc` 支持 DOCX、XLSX、PPTX 提取为 Markdown、纯文本或 JSON。[Docs.rs+1](https://docs.rs/undoc?utm_source=chatgpt.com) |
| PDF | `pdf-extract`、`lopdf`、`pdfium-render` | 中 | 可提取可复制文本的 PDF；扫描版 PDF 需要 OCR。PDF 版面还原会比较麻烦。 |
| EPUB | `epub`、zip + xml | 中 | 本质是压缩包 + HTML，适合后续支持 |
| RTF | 现成库较弱 | 低~中 | 可以考虑调用系统工具或先转文本 |
| 旧版 `.doc` | Rust 原生支持弱 | 低 | 建议通过 LibreOffice / 系统转换 / 外部工具转成 docx 或 pdf 后再处理 |

## 解析层建议这样设计

不要在业务逻辑里直接写：

```Rust
if path.ends_with(".docx") {
   ...
}
```

而是定义统一接口：

```Rust
pub struct ParsedDocument {
    pub title: Option<String>,
    pub file_type: String,
    pub chunks: Vec<DocumentChunk>,
}

pub struct DocumentChunk {
    pub heading: Option<String>,
    pub page_no: Option<u32>,
    pub text: String,
    pub order: usize,
}

pub trait DocumentParser {
    fn supports(&self, ext: &str) -> bool;
    fn parse(&self, path: &std::path::Path) -> anyhow::Result<ParsedDocument>;
}
```

然后实现：

```Rust
TxtParser
MarkdownParser
HtmlParser
PdfParser
DocxParser
OfficeParser
```

这样后面加新格式不会污染主流程。

## 对你的 App 最实用的解析策略

### 1. Markdown

按标题切：

```
# 一级标题
## 二级标题
正文段落
```

转成 chunk：

```JSON
{
  "heading": "二级标题",
  "text": "正文段落",
  "order": 12
}
```

### 2. DOCX

优先提取：

```
标题
普通段落
表格单元格文本
页眉页脚可选
```

DOCX 本质是 zip 包，正文主要在：

```
word/document.xml
```

所以就算不用高级库，也可以用：

```
zip + quick-xml
```

自己提取 `<w:t>` 文本。

### 3. PDF

PDF 建议分两档：

```
基础版：只提取文本
增强版：保留页码
```

你的搜索结果最好能返回：

```
文档名
第几页
命中段落
```

PDF 第一版只要能定位到页码，体验就已经很好。

### 4. HTML

HTML 建议不要只转纯文本，最好保留：

```
h1 / h2 / h3 / p / li / table
```

这样分段质量会明显更好。
