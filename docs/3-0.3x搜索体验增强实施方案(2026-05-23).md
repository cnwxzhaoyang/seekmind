# SeekMind 0.3.x 搜索体验增强实施方案

本文档用于细化 SeekMind `0.3.x` 的实施方案。当前产品已经完成 `0.2.x` 的稳定性和索引质量增强，下一阶段应当把重点放到“搜得到”之后的“看得懂、判得准、用得顺”。

## 1. 阶段目标

`0.3.x` 的目标不是引入新的检索范式，而是把现有关键词检索体验打磨到可长期使用的水平。

核心目标：

```text
用户输入关键词后，能快速看到相关结果，明确判断命中位置，并通过结果聚合和上下文预览迅速决定是否打开文档。
```

阶段关键词：

```text
高亮、摘要、聚合、排序、预览、历史、快捷入口
```

## 2. 当前基线

`0.2.x` 完成后，SeekMind 已具备以下基础：

```text
1. 本地目录索引
2. SQLite 元数据
3. Tantivy 全文索引
4. Python 解析器切片
5. chunk 级结果展示
6. 失败项调试和重试
7. 索引状态和增量更新
```

这意味着 `0.3.x` 可以直接基于现有搜索链路增强体验，而不需要重做索引底座。

## 3. 0.3.x 范围

### 3.1 必做项

```text
1. 命中关键词高亮
2. 更好的 snippet 生成
3. 搜索结果按文档聚合
4. 同一文档多个命中段落折叠展示
5. 文件名、标题、正文分字段权重
6. 最近修改时间排序因子
7. 最近打开文档排序因子
8. 搜索历史
9. 常用目录快捷入口
```

### 3.2 可选项

```text
1. 收藏结果
2. 最近查看结果
3. 搜索结果的键盘快速选择
4. 搜索页局部筛选
```

### 3.3 明确不做

```text
1. 语义搜索
2. 向量库
3. 本地问答
4. 自动总结
5. OCR
6. 复杂知识图谱
7. 团队协作
```

## 4. 设计原则

```text
1. 先让用户看清楚命中位置，再考虑更多排序策略。
2. 所有增强都要建立在现有关键词检索结果之上，不改变检索主链路。
3. 前端体验改进优先于新增复杂数据模型。
4. 任何新增字段都应支持回溯和调试，避免出现“看起来更高级，但无法解释”的结果。
```

## 5. 实施拆分

### 5.1 阶段 A：命中展示基础化

目标：

```text
让搜索结果中的命中内容更清楚，减少用户依赖肉眼在长段落中寻找关键词。
```

要做的事：

```text
1. 为搜索结果增加命中词高亮。
2. 改进 snippet 生成逻辑，优先展示命中词前后若干字符。
3. 在结果卡片中明确区分“命中文本”和“上下文文本”。
4. 对多关键词查询，尽量让 snippet 命中最密集的片段。
```

后端任务：

```text
1. 搜索结果返回 snippet 元信息。
2. 记录命中的字段来源：文件名、标题、正文。
3. 返回命中词片段的位置索引，供前端高亮。
```

前端任务：

```text
1. 搜索结果卡片支持高亮渲染。
2. 右侧预览支持同样的高亮样式。
3. 对长文本 snippet 做截断和展开。
```

验收标准：

```text
用户搜索一个明确存在的关键词后，可以直接在结果列表中看到高亮，而不需要手动逐字寻找。
```

---

### 5.2 阶段 B：结果聚合与折叠

目标：

```text
让同一文档的多个命中段落更容易被整体判断，减少结果列表噪音。
```

要做的事：

```text
1. 按文档聚合搜索结果。
2. 同一文档下的多个 chunk 默认折叠。
3. 展开后可以查看每个 chunk 的具体命中位置。
4. 文档卡片上显示命中 chunk 数。
```

后端任务：

```text
1. 搜索接口输出 chunk 级结果，同时提供文档聚合视图所需字段。
2. 结果中保留文档 id、chunk id、chunk 顺序、页码、标题路径。
3. 提供按文档聚合的辅助统计。
```

前端任务：

```text
1. 搜索列表默认按文档聚合。
2. 每个文档卡片可展开查看 chunk 明细。
3. 聚合卡片显示命中数量、文件类型、最近修改时间。
4. 统一处理“无结果”和“部分命中”两种状态。
```

验收标准：

```text
当一个文档有多个命中段落时，列表不会被重复结果刷屏，用户能先看文档层级，再展开看细节。
```

---

### 5.3 阶段 C：排序策略增强

目标：

```text
让更可能正确的结果排在前面。
```

要做的事：

```text
1. 文件名命中加权。
2. 标题命中加权。
3. 正文命中保持基础相关性。
4. 最近修改时间增加轻量排序因子。
5. 最近打开文档增加轻量排序因子。
6. 允许未来插入文档类型权重，但默认不激进。
```

建议排序公式：

```text
最终得分 = 正文相关性 + 标题命中权重 + 文件名命中权重 + 最近修改权重 + 最近打开权重 + 类型权重
```

实现原则：

```text
1. 权重不要过强，避免相关结果被时间因素冲掉。
2. 排序策略必须可解释，调试页可查看命中来源。
3. 初版优先以“更符合直觉”为目标，而不是追求复杂公式。
```

验收标准：

```text
同一关键词下，最有可能命中的文档通常排在前面，且排序原因基本可解释。
```

---

### 5.4 阶段 D：搜索历史与快捷入口

目标：

```text
降低重复检索成本，让用户更快回到常用搜索和常用目录。
```

要做的事：

```text
1. 保存最近搜索词。
2. 支持点击历史词快速再次搜索。
3. 提供常用目录快捷入口。
4. 支持把某些结果标记为收藏。
```

后端任务：

```text
1. 增加搜索历史存储。
2. 为历史记录设置最大条数和去重策略。
3. 提供常用目录读取和写入接口。
```

前端任务：

```text
1. 搜索框下方展示最近搜索词。
2. 支持点击历史词一键搜索。
3. 目录侧边栏增加快捷入口区。
4. 收藏项在搜索页和切片页都可见。
```

验收标准：

```text
用户重复搜索常见关键词时，不需要每次手动输入完整内容。
```

---

## 6. 数据模型建议

`0.3.x` 阶段建议补充或明确以下数据：

```text
1. search_history
2. favorite_searches
3. recent_documents
4. document_search_stats
```

建议字段：

### 6.1 search_history

```text
id
query
normalized_query
created_at
hit_count
```

### 6.2 recent_documents

```text
id
path
last_opened_at
open_count
```

### 6.3 document_search_stats

```text
document_path
last_hit_at
hit_count
last_query
```

说明：

```text
这部分不是为了做复杂画像，而是为了实现“最近使用权重”和“常用入口”。
```

## 7. 模块拆分建议

### 7.1 Rust 后端

建议继续按模块拆分：

```text
1. search
2. storage
3. parser
4. ranking
5. history
```

职责建议：

```text
search：负责 query 解析、Tantivy 查询和结果组装
ranking：负责排序权重和打分规则
history：负责历史记录、最近文档和收藏
storage：负责 SQLite 持久化
parser：继续保持 Python sidecar 和 Rust fallback
```

### 7.2 Vue 前端

建议按页面拆分：

```text
1. SearchPage：搜索主入口和结果展示
2. ChunkPage：文档切片浏览
3. StatusPage：索引和调试状态
4. SettingsPage：配置和偏好
5. SearchHistoryPanel：历史记录
6. ResultGroupCard：文档聚合结果卡片
```

## 8. 实施顺序

推荐顺序如下：

```text
1. 命中高亮和 snippet 改进
2. 按文档聚合和折叠展示
3. 排序策略增强
4. 搜索历史和常用目录
5. 收藏与最近查看
```

原因：

```text
先把用户看结果的效率提上来，再做行为记忆和快捷入口，最后再做轻量偏好排序。
```

## 9. 验收清单

### 9.1 搜索体验

```text
1. 输入关键词后，结果能高亮显示。
2. snippet 能直接展示命中上下文。
3. 同一文档的多个结果可以折叠和展开。
4. 排序结果更符合直觉。
```

### 9.2 使用效率

```text
1. 最近搜索词可重复点击。
2. 常用目录有快捷入口。
3. 常用结果可收藏。
```

### 9.3 可调试性

```text
1. 调试页可以看命中字段和权重。
2. 出现异常排序时能定位原因。
3. 搜索结果和调试数据保持一致。
```

## 10. 风险与控制

### 10.1 风险：高亮与 snippet 做得太复杂

控制：

```text
先保证正确，再追求漂亮。高亮只做关键词级，不做复杂富文本嵌套。
```

### 10.2 风险：聚合后结果变少

控制：

```text
聚合默认只改变展示方式，不改变真实命中数；用户可以随时展开看所有 chunk。
```

### 10.3 风险：排序调得过猛

控制：

```text
所有额外权重都要偏轻，不能压过正文相关性。
```

### 10.4 风险：历史和收藏变成新负担

控制：

```text
先做最小可用：最近搜索词、最近文档、收藏标记，不急着做复杂推荐。
```

## 11. 阶段验收定义

`0.3.x` 完成时，应满足：

```text
1. 用户能快速看懂是否命中。
2. 用户能从聚合结果里快速定位目标文档。
3. 用户可以通过高亮和 snippet 判断是否打开。
4. 搜索结果排序基本符合直觉。
5. 搜索历史和常用入口能减少重复操作。
```

如果这些目标达成，说明 SeekMind 从“可搜索”进入了“好搜索”的阶段，可以进入下一阶段的语义检索预备工作。
