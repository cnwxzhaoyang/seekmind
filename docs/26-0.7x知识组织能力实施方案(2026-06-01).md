# seekmind 0.7.x 知识组织能力实施方案

本文档用于定义 `0.7.x` 的开发范围和执行顺序。

上游依据：

```text
docs/2MVP后阶段规划(2026-05-21).md
```

阶段定位：

```text
从“搜索和问答工具”扩展为“个人文档知识库”。
```

核心目标：

```text
帮助用户把常用文档、重要段落、问答来源和主题资料沉淀下来，而不是每次重新搜索。
```

---

## 1. 0.7.x 第一原则

`0.7.x` 不做完整笔记软件，只做围绕本地文档来源的轻量知识组织。

产品边界：

```text
1. 收藏对象必须能回到具体文档、chunk 或搜索条件。
2. 主题集合只管理来源和备注，不做复杂富文本编辑器。
3. 标签用于轻量分类，不做复杂知识图谱。
4. 笔记围绕来源补充个人说明，不替代独立笔记软件。
5. 继续保持本地优先，数据默认只保存在本机 SQLite。
```

技术边界：

```text
1. 不重建搜索、语义搜索和问答体系。
2. 不引入团队协作、云同步和账号系统。
3. 不做 Obsidian 式双链、反链和块级编辑。
4. 不要求 PDF / Word 内部精确批注坐标。
5. 不把模型生成内容当作事实来源，来源仍以文档 chunk 为准。
```

---

## 2. 当前基础

当前项目已经具备这些可复用能力：

```text
1. documents / chunks / embeddings 等基础索引数据。
2. 搜索结果包含文档名、路径、标题路径、段落、页码和排序解释。
3. 右侧预览支持打开原文件、Quick Look、复制路径、复制引用、查看切片。
4. 问答答案已经带来源列表，并能回到具体来源。
5. 问答会话已经支持保存、重命名、删除、导出 Markdown。
6. 设置页已经从历史记录中瘦身，问答历史归口到独立问答页。
```

因此 `0.7.x` 应该在现有来源对象之上新增“组织层”：

```text
文档 / chunk / 搜索条件 / 问答来源
  ↓
收藏
  ↓
标签
  ↓
主题集合
  ↓
备注和引用导出
```

---

## 3. 第一批目标

第一批只交付最小知识组织闭环：

```text
用户可以把搜索结果或问答来源加入主题集合，并在之后从集合页快速回顾这些来源。
```

第一批包含：

```text
1. 收藏文档。
2. 收藏段落 chunk。
3. 主题集合的创建、重命名、删除。
4. 将搜索结果加入主题集合。
5. 将问答来源加入主题集合。
6. 主题集合详情页。
7. 主题内来源列表支持打开原文件、Quick Look、复制引用。
8. 主题级 Markdown 引用列表导出。
```

第一批不做：

```text
1. 复杂标签系统。
2. 富文本笔记编辑器。
3. 自动推荐文档关系。
4. 自动相似段落推荐。
5. 多人协作和云同步。
6. 复杂双链、反链和知识图谱视图。
```

## 4. 产品形态

### 4.1 新增入口

建议新增独立页面：

```text
/collections
```

侧边栏入口名称：

```text
知识库
```

页面结构：

```text
左侧：主题集合列表
中间：当前主题的来源列表
右侧：来源详情和备注
```

原因：

```text
1. 收藏和主题集合属于长期沉淀，不应该塞进搜索页或设置页。
2. 三栏结构可以复用问答页的来源详情经验。
3. 搜索页和问答页只负责“添加到集合”，集合页负责“管理和回顾”。
```

### 4.2 搜索页改动

搜索结果卡片增加操作：

```text
加入主题集合
收藏段落
收藏文档
```

交互建议：

```text
1. 右键菜单优先，避免结果卡片按钮过多。
2. 常用操作可以保留一个收藏图标。
3. 加入集合时弹出轻量选择器，支持新建集合。
```

### 4.3 问答页改动

问答来源摘要表增加操作：

```text
加入主题集合
收藏来源
```

交互建议：

```text
1. 来源行右键菜单增加“加入主题集合”。
2. 右侧引用详情操作区增加“加入主题集合”。
3. 加入集合时保存来源 chunk，不保存模型回答作为事实来源。
```

## 5. 数据设计

### 5.1 collections

主题集合表：

```text
id: string
name: string
description: string | null
color: string | null
sort_order: integer
created_at: string
updated_at: string
```

约束：

```text
1. name 非空。
2. 同名允许存在，但 UI 可提示。
3. 删除集合时同步删除集合条目，不删除原始 documents / chunks。
```

### 5.2 collection_items

主题集合条目表：

```text
id: string
collection_id: string
item_type: document | chunk | search | qa_source
document_id: string | null
chunk_id: string | null
qa_session_id: string | null
qa_message_id: string | null
title: string
path: string | null
title_path: string | null
snippet: string | null
note: string | null
source_meta_json: string | null
sort_order: integer
created_at: string
updated_at: string
```

说明：

```text
1. document 收藏只要求 document_id。
2. chunk / qa_source 收藏优先保存 chunk_id。
3. search 收藏保存搜索条件到 source_meta_json。
4. title / path / snippet 保留快照，避免原文档暂时不可访问时列表完全空白。
```

### 5.3 tags

标签第一批可以先不单独做复杂管理。如果要保留扩展位，建议建表但 UI 延后：

```text
tags
  id: string
  name: string
  color: string | null
  created_at: string
  updated_at: string

item_tags
  id: string
  target_type: document | chunk | collection | collection_item
  target_id: string
  tag_id: string
  created_at: string
```

第一批建议：

```text
建表可以做，完整标签管理 UI 放到第二批。
```

### 5.4 recent_views

最近查看表：

```text
id: string
target_type: document | chunk | collection | qa_session
target_id: string
title: string
path: string | null
viewed_at: string
```

用途：

```text
1. 集合页展示最近查看。
2. 后续排序时引入最近使用因子。
```

## 8. Markdown 导出

主题集合导出格式：

```markdown
# 集合名称

> 描述

## 来源 1：文档名

- 路径：/path/to/file.docx
- 位置：标题 A > 标题 B > 第 12 段
- 类型：chunk

摘录：

> snippet

备注：

用户备注
```

导出原则：

```text
1. 导出的是引用列表，不是重新生成总结。
2. 每条来源必须包含文档路径和定位信息。
3. 备注和摘录分开，避免混淆用户观点和原文来源。
```

## 9. 开发顺序

### 9.1 第一批：主题集合闭环

建议顺序：

```text
1. SQLite 新增 collections / collection_items。
2. Rust 新增集合 CRUD 命令。
3. 前端新增 /collections 页面和侧边栏入口。
4. 集合页实现集合列表和来源列表。
5. 搜索结果支持加入集合。
6. 问答来源支持加入集合。
7. 集合来源支持打开文件、Quick Look、复制引用。
8. 集合 Markdown 导出。
```

验收：

```text
1. 用户可以创建一个主题集合。
2. 用户可以把搜索结果加入集合。
3. 用户可以把问答来源加入集合。
4. 用户可以在集合页回顾来源。
5. 用户可以从集合页打开原文件和复制引用。
6. 用户可以导出集合引用列表。
```

### 9.2 第二批：标签和最近查看

建议顺序：

```text
1. 新增 tags / item_tags。
2. 集合和条目支持打标签。
3. 集合页支持按标签过滤。
4. 记录最近查看。
5. 首页或集合页展示最近查看。
```

验收：

```text
1. 用户可以给集合或来源条目打标签。
2. 用户可以按标签快速筛选集合内容。
3. 用户可以快速回到最近查看的文档、集合或问答会话。
```

### 9.3 第三批：推荐能力

建议顺序：

```text
1. 基于 embeddings 查找相似 chunk。
2. 在集合详情中显示相似段落推荐。
3. 基于同文档、同标题路径、同标签推荐相关文档。
4. 推荐项必须可解释：相似度、同目录、同标题、同标签。
```

验收：

```text
1. 用户查看一个来源时，可以看到 3 到 5 个相关段落。
2. 推荐结果必须能打开原文件并查看来源。
3. 推荐不会自动加入集合，必须由用户确认。
```

## 10. 风险和取舍

风险：

```text
1. 功能边界容易膨胀成笔记软件。
2. 主题集合、标签、收藏、最近查看可能概念重叠。
3. 收藏快照和原始文档更新之间可能出现不一致。
4. 来源详情组件如果重复实现，后续维护成本会上升。
```

取舍：

```text
1. 第一批只做主题集合和来源收藏，不做完整标签体系。
2. 快照用于列表稳定展示，打开和引用仍优先回到原始文档和 chunk。
3. 备注只做纯文本或 Markdown 文本，不做富文本编辑器。
4. 推荐能力放到第三批，避免影响第一批闭环。
```

## 11. 0.7.x 完成定义

`0.7.x` 可以收口的标准：

```text
1. 用户可以创建、重命名、删除主题集合。
2. 用户可以收藏文档和段落。
3. 用户可以把搜索结果加入主题集合。
4. 用户可以把问答来源加入主题集合。
5. 用户可以在集合页回顾来源并打开原文件。
6. 用户可以给集合条目添加备注。
7. 用户可以导出集合引用列表。
8. 不影响现有搜索、语义搜索、问答和来源预览能力。
```

一句话目标：

```text
seekmind 0.7.x 不是“做笔记软件”，而是“把可信来源沉淀成可复用的个人知识集合”。
```
