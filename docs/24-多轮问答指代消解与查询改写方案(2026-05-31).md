# 多轮问答指代消解与查询改写方案

## 背景

来源问答已经支持多轮会话，但当前续问主要依赖最近对话上下文和轻量历史词扩展。遇到“这两者有关系吗”“它们有什么区别”这类问题时，直接检索原始问题会丢失真实对象，简单拼接历史回答又容易引入噪声来源。

典型问题：

1. 用户问：什么是WBS
2. 用户问：什么是工作包
3. 用户问：这两者有关系吗

第三轮中的“这两者”应该被解析为“WBS”和“工作包”，检索问题应改写为“WBS 与 工作包 的关系”，而不是直接检索“这两者有关系吗”。

## 目标

- 在检索前完成轻量指代消解。
- 将续问改写成完整、可检索的问题。
- 保留原始用户问题用于展示和记录。
- 将已解析实体作为强强化词参与检索排序。
- 不引入额外模型调用，优先用规则实现稳定的第一版。

## 处理流程

### 1. 会话实体提取

从最近若干轮用户问题中提取核心实体。

规则优先覆盖高频问法：

- `什么是X` -> `X`
- `X是什么` -> `X`
- `介绍一下X` -> `X`
- `解释一下X` -> `X`

示例：

- `什么是WBS` -> `WBS`
- `什么是工作包` -> `工作包`

### 2. 指代识别

识别当前问题中的指代表达：

- `这两者`
- `这二者`
- `二者`
- `两者`
- `它们`
- `前面两个`

只有出现指代表达时才使用最近实体做消解。单纯包含“关系”的显式问题不应被误判为指代问题。

### 3. 查询改写

如果当前问题是指代型续问，并且能从历史问题中提取到两个实体：

原始问题：

```text
这两者有关系吗
```

检索问题：

```text
WBS 与 工作包 的关系
```

上下文提示：

```text
当前问题中的指代表达指代：WBS 与 工作包。
```

### 4. 检索强化词

检索排序除使用全文/语义得分外，还要将以下词作为强化词：

- 改写问题中的关键词
- 已解析实体原文，例如 `WBS`、`工作包`
- 关系型场景词，例如 `关系`、`关联`、`组成`、`分解`、`包含`、`工作分解结构`

命中这些强化词的来源应优先进入最终上下文。

### 5. 生成约束

Prompt 中明确传递：

- 用户原始问题
- 改写后的检索问题
- 指代对象

模型回答时必须围绕指代对象，不能把新检索到的无关来源当作“二者”的对象。

## 第一版实现范围

- 后端规则提取最近两个实体。
- 后端生成 `retrieval_question` 和 `relation_hint`。
- 后端生成 `boost_terms` 参与来源重排。
- 不新增数据库表。
- 不新增前端配置项。

## 后续增强

- 在问答详情中展示“实际检索问题”，便于调试。
- 将实体记忆持久化到会话级 metadata。
- 增加基于小模型/主模型的 query rewrite fallback。
- 对“它的风险”“这个流程”等单实体指代做同样处理。

## Python 侧 RAG 编排评估

RAG 编排放在 Python 侧有明显优势，但不建议把所有能力一次性迁移。

适合放到 Python 侧的能力：

- 查询改写：可以接入轻量 LLM、规则模板或中文 NLP 工具。
- 指代消解：可以做更完整的实体抽取、共指消解和主题记忆。
- 重排：可以使用 cross-encoder、bge-reranker 等模型做精排。
- Prompt 组装：方便做实验、灰度和日志追踪。
- RAG 调试：输出 rewritten query、entities、rerank scores、selected chunks。

仍建议保留在 Rust/Tauri 侧的能力：

- SQLite/Tantivy 索引管理。
- 文档扫描、状态管理、任务调度。
- 文件打开、Quick Look、桌面端权限操作。
- 前端事件流与取消控制。

推荐边界：

```text
Tauri/Rust: 管理会话、索引、文件、命令入口
Python Sidecar: RAG planner/query rewrite/entity memory/rerank/prompt assembly
Model Provider: 只负责最终生成
```

当前阶段先在 Rust 侧实现轻量规则，验证交互闭环；后续如果要提升准确率，再把 RAG planner 抽到 Python sidecar。
