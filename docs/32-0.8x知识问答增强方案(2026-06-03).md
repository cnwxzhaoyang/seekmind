# DocMind 0.8.x 知识问答增强方案

## 1. 背景

当前知识问答已经具备可用闭环：

1. 用户输入问题。
2. 后端检索本地文档 chunk。
3. 组装来源上下文。
4. 调用用户配置的 OpenAI-compatible 模型。
5. 前端展示答案、来源摘要和引用详情。

但它距离“可用、准确性高”还差一层稳定的 RAG 编排。主要问题不是 UI，而是证据链质量。

这里需要明确一个架构原则：RAG 核心能力应收敛到 Python sidecar。Rust/Tauri 侧负责数据库、任务调度、状态持久化和 UI 事件桥接，不应继续承载 query rewrite、证据重排、答案生成、引用校验这类 RAG 编排细节。

原因有三点：

1. Python sidecar 已经承担文档解析和切片，结构化 chunk、图片、表格等语义上下文天然在 Python 侧形成。
2. RAG 后续会引入 query rewrite、reranker、LLM judge、评测集等能力，Python 生态更合适。
3. 如果 Rust 侧先取大量候选、再搬给 Python 做二次处理，会增加数据搬运和两侧状态不一致风险。

因此，本方案中的 Rust 侧增强属于过渡实现，后续应迁移为 Python 侧 `rag_answer_stream` 任务。

当前主要问题：

1. 单一 query 召回容易漏掉正确片段。
2. 搜索排序偏相关性，不一定偏“能支撑答案”。
3. 相邻 chunk 被切散后，模型拿到的上下文不完整。
4. 答案和来源之间还缺少强引用绑定与生成后校验。

## 2. 目标

0.8.x 的目标是把问答从“能回答”提升到“可验证地基于本地证据回答”。

完成后应满足：

1. 正确来源更容易被召回。
2. 进入 prompt 的来源更少但更准。
3. 相邻内容能作为连续证据提供给模型。
4. 证据不足时更稳定地拒答。
5. 用户能从答案快速定位到具体来源。

## 3. 第一批范围

第一批先做后端 RAG 编排增强，不改前端主交互。当前已经在 Rust 侧落了一版过渡实现，但目标形态应调整为 Python sidecar 编排：

1. 多查询召回。
2. 问答专用证据重排。
3. 相邻 chunk 合并为连续 source block。
4. 增加关键日志，便于定位召回和筛选问题。

Rust 过渡实现保留这些能力，用于验证交互和数据结构；Python 侧迁移时复用同一套输入输出契约。

暂不做：

1. LLM query rewrite。
2. cross-encoder reranker。
3. 答案级引用校验。
4. 专用评测集 UI。

## 4. 多查询召回

问答不再只使用一个检索 query。

第一版 query 组成：

1. 当前问题或改写后的检索问题。
2. 当前问题 + 会话强化词。
3. 仅会话强化词。

这些 query 分别走现有混合检索，再按 chunk id 合并去重。

合并时记录：

1. 命中过几个 query。
2. 最高原始得分。
3. 问题词和来源标题、正文的覆盖情况。

## 5. 问答专用重排

问答排序不完全复用搜索排序。

第一版规则：

1. 原搜索分数仍作为基础分。
2. 命中多个 query 的 chunk 加分。
3. 文件名、标题路径、heading 命中问题词加分。
4. 正文 snippet 命中问题词加分。
5. 与会话强化词重合越多，排序越靠前。

这样可以优先选择“更能回答问题”的片段，而不是仅仅相关的片段。

## 6. 相邻 chunk 合并

如果同一文档中多个命中的 chunk 相邻，不再把它们全部作为离散来源塞给模型。

第一版处理：

1. 按文档和 chunk 顺序识别连续命中段。
2. 连续命中段合并成一个 source block。
3. source 仍保留代表 chunk，保证前端引用和详情能定位。
4. prompt 中把连续段落作为同一证据窗口提供给模型。

这能降低上下文碎片化，提升步骤类、流程类、表格附近文本问答的稳定性。

## 7. 后续增强

第二批建议：

1. 答案正文强制引用 `[S1]`。
2. 点击答案引用联动右侧来源详情。
3. 生成后做引用覆盖检查。
4. 建立 30 到 50 条本地问答评测集。
5. 将 query rewrite、entity memory、reranker、答案生成后校验迁到 Python sidecar。

## 8. Python sidecar 目标形态

RAG 核心任务新增为 `rag_answer_stream`。

Rust 侧职责：

1. 接收前端提问。
2. 创建 `job_id` / `request_id`。
3. 读取必要的 QA 设置、会话状态和检索范围。
4. 调用 Python sidecar。
5. 流式消费 Python event，更新数据库和 UI。
6. 持久化最终答案、来源、warning 和错误。

Python 侧职责：

1. 查询改写。
2. 候选召回编排。
3. 证据重排。
4. 相邻 chunk 合并和上下文窗口组装。
5. 调用 OpenAI-compatible 模型生成答案。
6. 引用校验。
7. 必要时修复、重试或拒答。

数据搬运原则：

1. Rust 不把大量 chunk 正文整体搬给 Python。
2. Python 通过轻量查询接口或本地 SQLite 只拉取候选需要的字段。
3. 最终只回传 UI 需要的 `answer`、`sources`、`retrieval`、`warning`、`error` 和进度事件。
4. 大字段如 preview blocks 尽量按需加载，右侧引用详情打开时再补充。

## 9. 完成标准

第一批完成标准：

1. 问答后端能执行多查询召回。
2. 重复 chunk 被去重并按问答证据规则重排。
3. 同文档相邻命中 chunk 能合并进入 prompt。
4. 问答结果仍兼容现有前端来源展示和引用详情。
5. 构建和 Rust 检查通过。
