# DocMind 0.6.x 基于来源问答实施方案

本文档用于定义 `0.6.x` 的开发范围和执行顺序。

上游依据：

```text
docs/2MVP后阶段规划(2026-05-21).md
```

阶段定位：

```text
从“搜索相关段落”升级到“基于本地文档回答问题”。
```

核心约束：

```text
没有来源，不给确定答案。
来源不足，明确说明无法从当前文档判断。
答案必须能回溯到具体文档和段落。
```

---

## 1. 0.6.x 第一原则

`0.6.x` 不做通用聊天机器人，只做“带来源的本地文档问答”。

产品边界：

```text
1. 只回答能被本地索引文档支持的问题。
2. 不联网搜索。
3. 不默认上传用户文档。
4. 不自动操作文件或执行任务。
5. 不追求长篇综合报告，先追求短答案和可靠引用。
```

技术边界：

```text
1. 检索结果由 DocMind 后端确定。
2. 模型只能基于给定上下文回答。
3. 引用来源由后端生成 source id，模型不能凭空创造来源。
4. 证据不足时走拒答分支，而不是强行生成。
5. 第一批先做单轮问答，不做多轮上下文记忆。
```

---

## 2. 当前基础

当前项目已经具备这些可复用能力：

```text
1. SQLite 文档和 chunk 元数据。
2. Tantivy 全文搜索。
3. FastEmbed 语义向量和混合检索。
4. SearchResultView 已包含文件名、路径、标题路径、段落、页码、排序解释。
5. ChunkView 已包含 preview_blocks，可用于构造更完整上下文。
6. 搜索页已有结果列表、右侧预览、打开原文件、Quick Look、收藏等入口。
```

因此 `0.6.x` 不应该重建索引或检索体系，而应该新增一层问答编排：

```text
用户问题
  ↓
混合检索召回 chunk
  ↓
证据筛选和上下文组装
  ↓
证据充足性判断
  ↓
调用用户配置的模型
  ↓
答案 + 来源列表 + 可点击定位
```

---

## 3. 第一批目标

第一批只交付最小可靠闭环：

```text
用户输入一个问题，DocMind 检索本地文档，生成一段简短答案，并列出 2 到 5 个可点击来源。
```

第一批包含：

```text
1. 问答入口。
2. 后端 ask 命令。
3. 基于当前混合检索的候选 chunk 召回。
4. 上下文预算和 source id 组装。
5. 模型配置和一次性调用。
6. 答案来源列表。
7. 证据不足拒答。
8. 问答历史的最小落盘。
```

第一批不做：

```text
1. 多轮对话。
2. 流式输出。
3. Agent 工具调用。
4. 自动总结整篇文档。
5. 跨问题长期记忆。
6. 联网搜索。
7. 默认云端模型。
8. PDF / Word 内部精确高亮。
```

---

## 4. 模型策略

### 4.1 默认策略

默认不内置云端模型，也不自动发送文档内容到任何远端服务。

第一批推荐支持：

```text
1. OpenAI-compatible HTTP endpoint
2. 用户自填 base_url
3. 用户自填 api_key
4. 用户自填 model
5. 本地服务同样走 OpenAI-compatible 协议，例如 Ollama / LM Studio 的兼容接口
```

原因：

```text
1. 可以同时覆盖本地模型和用户自己的云端模型。
2. 后端协议简单，第一批不需要为每个模型供应商写适配器。
3. 隐私边界清晰：只有用户显式配置后才会调用模型。
```

### 4.2 配置项

建议新增 `qa_settings`：

```text
enabled: bool
provider: openai_compatible
base_url: string
api_key: string
model: string
temperature: number
max_output_tokens: number
context_chunk_limit: number
context_token_budget: number
min_evidence_count: number
min_retrieval_score: number
```

第一批默认值：

```text
enabled = false
temperature = 0.2
max_output_tokens = 600
context_chunk_limit = 8
context_token_budget = 6000
min_evidence_count = 2
min_retrieval_score = 0.0
```

## 5. 后端设计

### 5.1 新增模块

建议新增：

```text
src-tauri/src/docmind/qa/mod.rs
src-tauri/src/docmind/qa/commands.rs
src-tauri/src/docmind/qa/models.rs
src-tauri/src/docmind/qa/context.rs
src-tauri/src/docmind/qa/client.rs
src-tauri/src/docmind/qa/store.rs
```

职责：

```text
commands.rs  Tauri 命令入口
models.rs    问答请求、回答、来源、设置、历史视图
context.rs   检索、去重、上下文预算、prompt source block 构造
client.rs    OpenAI-compatible HTTP 调用
store.rs     设置和历史落盘
```

### 5.2 新增命令

第一批建议命令：

```text
ask_question(question, scope_paths, limit) -> QaAnswerView
get_qa_settings() -> QaSettingsView
save_qa_settings(settings) -> QaSettingsView
list_qa_history(limit) -> Vec<QaHistoryView>
remove_qa_history(id) -> ()
```

暂不做流式命令。流式输出放到第二批，避免第一批同时处理取消、事件乱序和部分答案落盘。

### 5.3 数据结构

`QaAnswerView`：

```text
id: string
question: string
answer: string
state: answered | insufficient_evidence | model_not_configured | failed
sources: Vec<QaSourceView>
retrieval: QaRetrievalView
model: string
created_at: string
error: string | null
```

`QaSourceView`：

```text
source_id: string
chunk_id: string
file_name: string
path: string
ext: string
title_path: string
heading: string
paragraph: number | null
page: number | null
snippet: string
score: number
rank_reason: string
```

`QaRetrievalView`：

```text
search_mode: string
candidate_count: number
selected_count: number
semantic_enabled: bool
semantic_fallback: bool
semantic_fallback_reason: string
```

## 6. 检索和上下文策略

### 6.1 召回

第一批直接复用现有混合搜索：

```text
database.search_documents_debug(question, recall_limit)
```

建议：

```text
recall_limit = max(context_chunk_limit * 3, 20)
```

这样可以复用全文、语义、标题、文件名、收藏、最近打开等既有排序信号。

### 6.2 去重

去重规则：

```text
1. 同一个 chunk_id 只保留一次。
2. 同一个文档最多保留 3 个 chunk。
3. 连续相邻 chunk 可以合并展示，但 source id 仍指向主命中 chunk。
4. 过短 chunk 或空 snippet 不进入上下文。
```

### 6.3 上下文预算

第一批不引入 tokenizer，先用字符预算近似控制。

建议：

```text
中文字符预算约等于 token 预算的 1.5 到 2 倍。
单个 source block 最大 1200 字符。
总 source block 最大 12000 字符。
```

source block 格式：

```text
[S1]
文件: Maven 离线构建方案.md
路径: /abs/path/Maven 离线构建方案.md
位置: 三、离线依赖导出方案 / 第 5 段
内容:
...
```

### 6.4 证据不足判断

进入模型调用前先做硬判断：

```text
1. selected_sources < min_evidence_count 时拒答。
2. 所有候选都是极短文本时拒答。
3. 语义检索不可用且关键词召回为空时拒答。
4. 问题明显不是文档问题时拒答，例如“讲个笑话”。
```

拒答文案：

```text
无法从当前已索引文档中找到足够来源来回答这个问题。你可以尝试换一个关键词、扩大索引目录，或先确认相关文档已经被索引。
```

## 7. Prompt 策略

系统提示词目标：

```text
1. 只基于给定 sources 回答。
2. 不使用外部知识补全事实。
3. 每个关键结论必须带来源编号。
4. 来源不足时明确说明无法判断。
5. 输出简短、直接、可执行。
```

建议输出格式：

```text
回答:
...

依据:
- [S1] ...
- [S3] ...

不足:
...
```

后端校验：

```text
1. sources 为空时不接受 answered。
2. 答案中出现不存在的 [Sx] 时标记 failed 或移除非法引用。
3. 模型返回空答案时走 failed。
4. 不让模型决定最终来源列表，来源列表始终来自后端 selected_sources。
```

## 8. 前端设计

### 8.1 页面入口

第一批推荐在现有搜索页增加一个 `问答` 模式，而不是新增独立复杂页面。

入口形态：

```text
搜索 / 问答 segmented control
```

原因：

```text
1. 用户心智连续：都是围绕“找内容”。
2. 可以复用搜索页右侧来源预览。
3. 不需要先设计完整聊天页。
```

### 8.2 问答结果区

需要展示：

```text
1. 问题输入框。
2. 回答状态。
3. 简短答案。
4. 来源卡片列表。
5. 每个来源支持打开原文件、Quick Look、复制引用。
6. 证据不足提示。
7. 模型未配置提示和跳转设置入口。
```

### 8.3 设置页

新增问答模型设置：

```text
1. 启用问答。
2. base_url。
3. api_key。
4. model。
5. temperature。
6. 测试连接。
7. 隐私提示：启用远端模型会发送被选中的文档片段。
```

## 9. 数据落盘

第一批建议新增表：

```sql
CREATE TABLE IF NOT EXISTS qa_settings (
    id INTEGER PRIMARY KEY CHECK (id = 1),
    enabled INTEGER NOT NULL DEFAULT 0,
    provider TEXT NOT NULL DEFAULT 'openai_compatible',
    base_url TEXT NOT NULL DEFAULT '',
    api_key TEXT NOT NULL DEFAULT '',
    model TEXT NOT NULL DEFAULT '',
    temperature REAL NOT NULL DEFAULT 0.2,
    max_output_tokens INTEGER NOT NULL DEFAULT 600,
    context_chunk_limit INTEGER NOT NULL DEFAULT 8,
    context_token_budget INTEGER NOT NULL DEFAULT 6000,
    min_evidence_count INTEGER NOT NULL DEFAULT 2,
    min_retrieval_score REAL NOT NULL DEFAULT 0.0,
    updated_at INTEGER NOT NULL
);
```

```sql
CREATE TABLE IF NOT EXISTS qa_history (
    id TEXT PRIMARY KEY,
    question TEXT NOT NULL,
    answer TEXT NOT NULL,
    state TEXT NOT NULL,
    sources_json TEXT NOT NULL DEFAULT '[]',
    retrieval_json TEXT NOT NULL DEFAULT '{}',
    model TEXT NOT NULL DEFAULT '',
    error TEXT NOT NULL DEFAULT '',
    created_at INTEGER NOT NULL
);
```

注意：

```text
api_key 第一批可以先存在本地 SQLite，但必须在 UI 标注“本机明文保存”。
如果要提高安全性，第二批再接系统 keychain。
```

## 10. 开发顺序

### 10.1 第一批：最小问答闭环

```text
1. 新增 qa 数据模型和设置表。
2. 新增 get/save qa settings 命令。
3. 新增 context builder，复用 search_documents_debug。
4. 新增 OpenAI-compatible client。
5. 新增 ask_question 命令。
6. 前端增加问答模式和设置项。
7. 来源卡片接入打开文件、Quick Look、复制引用。
8. 问答历史最小列表。
```

验收：

```text
1. 未配置模型时，问答入口明确提示需要配置。
2. 配置本地 OpenAI-compatible 服务后，可以基于已索引文档回答。
3. 每次回答都展示 2 到 5 个来源。
4. 点击来源可以回到文档。
5. 无召回或证据不足时不生成答案。
```

### 10.2 第二批：体验增强

```text
1. 流式输出。
2. 请求取消。
3. 更好的历史详情。
4. 指定目录问答。
5. source 去重和相邻 chunk 合并优化。
6. prompt 调试面板。
```

### 10.3 第三批：质量增强

```text
1. 引用一致性校验增强。
2. answer/source 覆盖率检查。
3. 更好的上下文压缩。
4. 多模型供应商适配。
5. API key 迁移到系统 keychain。
```

## 11. 验收用例

基础用例：

```text
1. 提问“这个项目如何启动？”
2. 提问“语义搜索模型如何预热？”
3. 提问“当前支持哪些文档格式？”
4. 提问“PDF 现在是否稳定支持？”
```

拒答用例：

```text
1. 提问“今天上海天气怎么样？”
2. 提问“帮我写一首诗。”
3. 提问一个已索引文档中不存在的业务事实。
```

隐私用例：

```text
1. 问答未启用时，不允许调用模型。
2. base_url / model 为空时，不允许调用模型。
3. UI 明确提示将发送被选中的文档片段到用户配置的模型服务。
```

回归用例：

```text
1. 关键词搜索结果不受问答功能影响。
2. 语义搜索关闭时，问答仍可基于全文搜索尝试召回。
3. 语义模型不可用时，问答返回检索降级信息。
4. 打开原文件、Quick Look、复制引用仍可用。
```

## 12. 风险和取舍

主要风险：

```text
1. 模型配置复杂，用户可能不知道如何选择本地服务。
2. 远端模型涉及隐私，需要明确提示。
3. 第一批不做流式输出，等待时间可能偏长。
4. 字符预算不是精确 token 预算，可能导致部分模型上下文超限。
5. 只靠 prompt 不能完全消除幻觉，所以必须有后端证据门禁。
```

当前取舍：

```text
1. 先支持 OpenAI-compatible，暂不做多供应商 UI。
2. 先单轮问答，暂不做聊天历史上下文。
3. 先用后端固定 sources，暂不让模型自由检索。
4. 先接受非流式调用，保证闭环稳定后再改流式。
```

## 13. 0.6.x 完成定义

`0.6.x` 可以收口的标准：

```text
1. 用户可以显式启用并配置问答模型。
2. 用户可以对本地已索引文档提问。
3. 系统能基于混合检索选择来源 chunk。
4. 答案必须附带可点击来源。
5. 来源不足时稳定拒答。
6. 问答历史可查看。
7. 不影响现有搜索、语义搜索、预览定位能力。
```

一句话目标：

```text
DocMind 0.6.x 不是“会聊天”，而是“能带着证据回答本地文档问题”。
```
