# Windows 适配前平台边界整理

## 背景

SeekMind 当前在 macOS 上可以正常编译和运行。下一步希望支持 Windows 构建，但不建议直接切出 `macOS` / `Windows` 两条长期分支。Windows 适配应优先收敛平台差异，让主线代码具备跨平台构建能力，再通过构建脚本和 CI 区分产物。

推荐分支策略：

- `main`：保持唯一长期主线。
- `feat/windows-build`：Windows 适配短期功能分支。
- 适配完成后合回 `main`，后续通过平台构建脚本生成 macOS / Windows 包。

只有在两个平台需要长期不同发布节奏时，才考虑 `release/macos` 和 `release/windows` 这种长期分支。

## 整理目标

Windows 适配前先完成一轮“平台边界整理”，目标是把平台相关逻辑集中管理，避免后续在业务代码里散落 `cfg(target_os)`、路径拼接和脚本差异。

本轮整理不直接追求 Windows 一次性打包成功，而是先让代码结构具备可维护的跨平台边界。

## 一、抽取 platform / runtime_paths 模块

### 目标

统一管理资源路径、缓存路径、sidecar 路径、模型缓存路径和运行时诊断信息。

### 建议位置

Rust 侧建议新增或整理为：

```text
src-tauri/src/seekmind/platform/
  mod.rs
  runtime_paths.rs
```

或者如果不想新增目录，也可以先使用：

```text
src-tauri/src/seekmind/runtime_paths.rs
```

### 需要收敛的内容

- app resources 目录解析
- parser sidecar 路径解析
- sidecar 可执行文件名
- OCR 资源目录
- fastembed cache 目录
- bundled fastembed 目录
- bundled fastembed 压缩包路径
- 开发态和打包态资源目录差异
- 用户显式环境变量覆盖逻辑

### 设计原则

- 后端文件路径统一使用 `PathBuf` 和 `std::path`，不要手写 `/` 拼接。
- sidecar 名称按平台生成：
  - macOS / Linux：普通二进制名
  - Windows：`.exe`
- 开发态和生产态路径差异集中在一个模块里。
- 业务 command 不直接关心资源具体在哪里，只调用 runtime paths API。

### 预期 API 示例

```rust
pub struct SeekMindRuntimePaths {
    pub app_resources_dir: PathBuf,
    pub parser_sidecar_path: PathBuf,
    pub ocr_resources_dir: PathBuf,
    pub fastembed_cache_dir: PathBuf,
    pub bundled_fastembed_dir: PathBuf,
    pub bundled_fastembed_archive: PathBuf,
}

pub fn resolve_runtime_paths(app: &tauri::AppHandle) -> Result<SeekMindRuntimePaths, String>;
pub fn parser_sidecar_filename() -> &'static str;
pub fn prepare_fastembed_cache(paths: &SeekMindRuntimePaths) -> Result<(), String>;
```

## 二、把 npm 脚本改成跨平台写法

### 当前风险

当前开发脚本里存在 Unix 风格环境变量写法：

```text
SEEKMIND_USE_PY_PARSER=1 SEEKMIND_PARSER_SCRIPT=... tauri dev
```

这类写法在 Windows `cmd.exe` 和 PowerShell 下不可直接运行。

### 建议方案

优先使用 `cross-env`，或者改成 Node 脚本统一设置环境变量。

推荐顺序：

1. 简单脚本使用 `cross-env`
2. 复杂脚本使用 `scripts/*.mjs`
3. 不在 `package.json` 中塞过长命令

### 需要改造的脚本类型

- `tauri:dev:first-launch`
- `tauri:dev`
- `semantic:warmup`
- `semantic:warmup:mirror`
- parser sidecar 构建脚本
- macOS / Windows 专用 tauri build 脚本

### 示例

```json
{
  "scripts": {
    "tauri:dev:first-launch": "cross-env SEEKMIND_USE_PY_PARSER=1 SEEKMIND_FORCE_FIRST_LAUNCH=1 tauri dev",
    "build:check": "npm run build && cargo check --manifest-path src-tauri/Cargo.toml"
  }
}
```

如果涉及动态路径，例如 `$PWD/parser/...`，建议用 Node 脚本生成：

```text
scripts/dev-tauri-first-launch.mjs
```

避免依赖 shell 变量展开。

## 三、保留并扩展 sidecar 资源检测日志

### 目标

启动时直接打印关键资源路径和命中状态，方便定位 Windows 下“模型已下载但运行时不可用”“sidecar 找不到”“OCR 资源缺失”等问题。

### 当前应保留的信息

- fastembed 实际 cache 目录
- fastembed 模型缓存是否命中
- 是否使用显式环境变量覆盖
- bundled fastembed 目录
- bundled fastembed 压缩包路径
- OCR runtime probe 结果
- SQLite 数据库路径
- Tantivy 索引目录

### Windows 需要补充的信息

- 当前平台识别结果：`windows` / `macos` / `linux`
- sidecar 文件名和最终路径
- sidecar 文件是否存在
- sidecar 是否可执行或可启动
- app resources 目录
- OCR 资源目录是否存在
- fastembed cache 回填来源
- Windows 下 `%APPDATA%` / `%LOCALAPPDATA%` 对应路径

### 日志格式建议

保持单行结构化文本，便于复制排查：

```text
[SeekMind] runtime paths platform=windows resources=... parser_sidecar=... sidecar_exists=true fastembed_cache=... model_cache_exists=true
```

关键日志应该在应用启动阶段打印一次；失败时必须输出具体路径和原因。

## 四、准备独立构建与检查脚本

### 目标

把“本地检查”和“平台构建”拆成明确命令，避免每次手动组合命令。

### 建议脚本

```json
{
  "scripts": {
    "check:frontend": "npm run build",
    "check:rust": "cargo check --manifest-path src-tauri/Cargo.toml",
    "check:all": "npm run check:frontend && npm run check:rust",
    "tauri:build:macos": "tauri build",
    "tauri:build:windows": "tauri build --target x86_64-pc-windows-msvc"
  }
}
```

如果需要交叉编译 Windows，需要额外确认：

- Rust target 是否安装
- Tauri Windows bundler 依赖是否可用
- Python parser sidecar 是否能构建为 Windows `.exe`
- OCR 和 fastembed 资源是否进入 Windows bundle
- Windows 签名和安装包策略

### 检查顺序

1. `npm run check:frontend`
2. `npm run check:rust`
3. `npm run check:all`
4. `npm run semantic:warmup:mirror`
5. `npm run tauri:dev:first-launch`
6. `npm run tauri:build:macos`
7. `npm run tauri:build:windows`

Windows target 构建可能无法在 macOS 上完全闭环，最终仍建议在 Windows CI 或 Windows 真机上验证。

## 五、优先级建议

### P0

- 抽取 runtime paths 模块
- 收敛 sidecar 路径、OCR 路径、fastembed cache 路径
- 保留并扩展启动诊断日志
- 把开发脚本改成跨平台写法

### P1

- 增加 `check:frontend`、`check:rust`、`check:all`
- 增加 macOS / Windows 构建脚本
- 梳理 Tauri resources / externalBin 配置
- 明确 Windows parser sidecar 产物名称和打包位置

### P2

- Windows 真机验证拖拽、索引、语义模型、OCR、日志导出
- Windows 安装包、图标、签名、自动更新链路
- CI 增加平台矩阵构建

## 六、完成标准

本轮平台边界整理完成后，应满足：

- 业务代码不直接拼接平台资源路径。
- sidecar、OCR、fastembed cache 路径都由统一模块解析。
- macOS 开发态和打包态仍可正常启动。
- 启动日志能明确说明当前使用的资源路径和缓存命中情况。
- npm 脚本不依赖 Unix 专属环境变量写法。
- 至少具备 `cargo check`、`npm run build`、`tauri build --target ...` 的独立脚本入口。

完成这些整理后，再进入 Windows 构建适配会更可控，也更适合后续合回 `main` 长期维护。
