# DocMind 0.5.x 文档预览与定位增强开发计划

本文档用于定义 DocMind 在 `0.4.x` 收口之后的下一阶段路线。

`0.5.x` 不再继续扩搜索主链路，而是把能力从“找到段落”推进到“快速回到原文上下文并确认来源位置”。

当前产品已经具备部分雏形：

```text
1. 搜索结果可展示命中段落和 snippet。
2. 可打开原文件。
3. 可展示上下文预览和高亮。
4. 搜索页已经有右侧预览区和文档详情雏形。
```

但这些能力还不够系统，仍存在几个明显缺口：

```text
1. 上下文预览与来源信息不够完整。
2. 不同文档类型的定位信息还不统一。
3. 复制引用、复制路径、Quick Look 等“回到原文”的动作还不完整。
4. 预览区仍偏搜索页附属能力，没有成为稳定的定位体验。
```

`0.5.x` 的目标，就是把这些雏形收口成一个完整的“文档预览与定位”阶段。

---

## 1. 阶段定位

建议版本：

```text
0.5.x
```

阶段定位：

```text
让用户从“找到段落”进一步到“快速确认原文、回到上下文、拿到可引用定位”。
```

长期价值：

```text
这个阶段会显著提升搜索结果可信度，也为后续基于来源的问答打基础。
```

---

## 2. 阶段目标

### 2.1 核心目标

```text
搜索结果不仅告诉用户命中了哪里，还要让用户迅速理解“这段内容来自哪、前后讲了什么、如何回到原文”。
```

### 2.2 用户体验目标

```text
1. 用户点击结果后，可以立即看到上下文。
2. 用户可以看到命中段落前后相邻 chunk。
3. 用户可以根据文档类型看到更明确的定位信息。
4. 用户可以一键打开原文件或使用系统预览快速确认。
5. 用户可以复制引用和文件路径，方便在其他地方继续使用。
```

### 2.3 成功标准

```text
1. 用户看到一个搜索结果后，能在极短时间内判断它是否就是目标文档。
2. 用户无需手动翻文件，就能明确命中位置和上下文。
3. 预览、引用、定位、打开文件这几条链路都能从同一个结果卡闭环。
```

---

## 3. 当前基线

当前已经存在的基础能力：

```text
1. 搜索结果卡可以显示 snippet 和高亮。
2. 右侧已有文档详情和上下文预览雏形。
3. 可以打开原文件。
4. 文档列表页和切片页已经能展示 chunk。
5. 搜索结果按文档聚合、卡片化展示已经比较稳定。
```

当前还缺的关键能力：

```text
1. 更完整的 chunk 上下文前后联动。
2. 更准确的页码 / 标题路径 / 层级路径。
3. Quick Look / 系统级预览。
4. 复制引用、复制路径、复制定位信息。
5. 文档类型差异化定位展示。
```

---

## 4. 明确不做

`0.5.x` 只做“定位增强”，不越界到下一阶段。

```text
1. 不做复杂文档编辑器。
2. 不做 Word 内部精确高亮和批注编辑。
3. 不做 PDF 坐标级文字精确选择。
4. 不做 OCR。
5. 不做基于来源的自动问答。
6. 不做团队协作和云同步。
```

---

## 5. 功能拆解

### 5.1 App 内上下文预览闭环

#### 目标

```text
把“命中的段落”扩展成“命中的段落 + 前后上下文 + 来源信息”。
```

#### 要做什么

```text
1. 结果详情区展示当前命中 chunk。
2. 展示命中段落前后相邻 chunk。
3. 展示当前 chunk 在文档中的位置。
4. 展示文件名、路径、格式、更新时间等基础信息。
5. 让用户能快速确认这条结果是不是目标内容。
```

#### 当前状态

```text
已有 snippet 和部分上下文预览雏形，但还不够系统化。
```

#### 验收什么

```text
1. 用户点击搜索结果后，右侧可以看到命中 chunk 的上下文。
2. 上下文不只是当前段落，还能看到相邻段落。
3. 结果详情里能明确看到来源文件信息。
```

---

### 5.2 文档类型定位增强

#### 目标

```text
让不同文档类型都有更贴近原文结构的定位信息。
```

#### 要做什么

```text
1. PDF 显示页码，并支持按页定位。
2. DOCX 显示标题路径或标题层级路径。
3. Markdown 显示标题路径。
4. HTML 显示标题路径和结构块来源。
5. 统一把“定位信息”作为搜索结果的一等信息。
```

#### 当前状态

```text
当前结果里已经有部分 chunk 顺序和 snippet 信息，但页码、标题路径和类型差异化展示还不完整。
```

#### 验收什么

```text
1. PDF 结果能看到页码。
2. DOCX / MD / HTML 结果能看到来源标题路径。
3. 用户不需要翻开原文件就能知道命中内容出自哪里。
```

---

### 5.3 Quick Look / 系统级预览

#### 目标

```text
让用户用系统能力快速确认文件内容，不必每次都跳到主应用内阅读。
```

#### 要做什么

```text
1. 提供系统级 Quick Look 入口。
2. 在结果卡和文件侧栏提供快速预览按钮。
3. 预览和“打开原文件”并存，但预览优先用于快速确认。
```

#### 当前状态

```text
当前已有打开原文件，但系统级预览还没有成为稳定能力。
```

#### 验收什么

```text
1. 用户可以在不离开 DocMind 的情况下快速查看文件。
2. 预览和打开原文件是两个明确不同的动作。
```

---

### 5.4 引用与路径复制

#### 目标

```text
让用户可以把文档定位结果直接带到其他工作流里。
```

#### 要做什么

```text
1. 复制段落引用。
2. 复制文件路径。
3. 复制定位信息。
4. 引用格式尽量可读，并包含文档名、标题路径、chunk 序号或页码。
```

#### 建议引用格式

```text
《Maven 离线构建方案.md》 / 三、离线依赖导出方案 / 第 5 段
```

#### 验收什么

```text
1. 用户能一键复制定位信息。
2. 复制内容能直接粘贴到笔记、聊天或文档里。
3. 引用信息不丢文档层级。
```

---

### 5.5 文件信息侧栏

#### 目标

```text
把文档元信息从“附属展示”变成“辅助判断结果是否可靠”的一部分。
```

#### 要做什么

```text
1. 展示文件名、路径、格式、大小、修改时间。
2. 展示 chunk 数、命中 chunk 数。
3. 展示必要的来源状态信息。
4. 与上下文预览并列，形成稳定的右侧信息区。
```

#### 验收什么

```text
1. 用户能快速判断文档是否最新。
2. 用户能判断这份文档的结果是不是来自正确位置。
```

---

## 6. 推荐执行顺序

### 第一批：上下文与来源信息闭环

```text
1. App 内段落上下文预览
2. 展示命中段落前后相邻 chunk
3. 文件信息侧栏
```

原因：

```text
先把“看懂结果”的核心闭环做稳，再谈更高级的定位动作。
```

### 第二批：文档类型定位增强

```text
1. PDF 页码定位
2. DOCX 标题路径展示
3. Markdown 标题路径展示
4. HTML 标题路径展示
```

原因：

```text
不同格式的定位体验必须一致，否则用户会在多格式混合文档库里反复猜。
```

### 第三批：快速确认与复用

```text
1. Quick Look 预览
2. 复制段落引用
3. 复制文件路径
4. 复制定位信息
```

原因：

```text
这部分属于“最后一公里”，能把预览结果接入用户的笔记、会议和协作工作流。
```

---

## 7. 实施建议

### 7.1 数据层

```text
1. 统一补齐 chunk 的来源信息。
2. 保留页码、标题路径、chunk 顺序、来源文件元信息。
3. 为引用生成提供稳定字段，而不是只靠前端拼字符串。
```

### 7.2 服务层

```text
1. 给搜索结果补更完整的 context payload。
2. 提供复制引用所需的标准化输出。
3. 让预览和打开文件共用同一条来源数据。
```

### 7.3 前端层

```text
1. 结果详情和预览区分层展示。
2. 文档类型定位信息放在醒目的位置。
3. 系统预览和打开原文件保持明确区分。
```

---

## 8. 风险与注意点

```text
1. PDF、DOCX、MD、HTML 的定位粒度不一致，不能强行统一成一张假图。
2. 预览区信息不能过多，否则会把搜索页重新做重。
3. 引用格式要稳定，后续不要频繁改动，否则会影响用户习惯。
4. Quick Look 依赖平台能力，需保留降级路径。
```

---

## 9. 收口标准

当以下条件都满足时，`0.5.x` 可以认为收口：

```text
1. 用户能在 App 内快速确认命中文档的上下文。
2. 用户能明确知道命中来自哪个文件、哪一页或哪一层标题。
3. 用户能一键打开原文件或使用系统预览。
4. 用户能复制段落引用和文件路径。
5. 预览体验在主要文档类型上是稳定一致的。
```

---

## 10. 与后续阶段的关系

`0.5.x` 完成后，DocMind 会更适合进入 `0.6.x` 的“基于来源的问答”阶段，因为：

```text
1. 预览与定位已经足够稳定。
2. 引用信息已经标准化。
3. 后续问答回答可以直接复用这些来源字段。
```

换句话说，`0.5.x` 不是终点，而是把“引用和定位”这层打牢，为 `0.6.x` 做铺垫。
