# 0.8x 格式扩展缺口清单

本文用于承接 `docs/2MVP后阶段规划(2026-05-21).md` 中 `0.8.x` 的阶段目标，并结合当前实现状态，梳理还需要补齐的能力、建议的实现顺序以及收口标准。

对照时间：2026-06-05

---

## 1. 0.8x 阶段目标

`0.8x` 的定位是：

```text
扩大可索引资料范围
```

阶段目标是让更多现实中的文档格式进入 SeekMind 的索引范围，尤其是：

```text
扫描版 PDF
图片
PPTX
XLSX
EPUB
RTF
旧版 doc
```

---

## 2. 当前已完成情况

结合当前代码实现，`0.8x` 里已经基本完成或可视为完成的能力有：

```text
1. OCR 已引入 Python 侧处理链路。
2. 扫描版 PDF chunk 已支持。
3. PPTX 文本提取已支持。
4. 旧版 doc 转换已支持。
```

因此，`0.8x` 当前不再是“从零开始做格式扩展”，而是：

```text
把 OCR 和多格式支持补成完整闭环。
```

---

## 3. 0.8x 缺口清单

### 3.1 OCR 闭环收口

当前缺口：

```text
1. 扫描版 PDF 的检测与触发流程还需要进一步收紧。
2. OCR 任务队列还需要更明确的状态流转。
3. OCR 结果保存和复用策略还需要统一。
4. OCR 的可见状态和说明还没有形成完整的产品表达。
```

建议实现：

```text
1. 增加扫描版 PDF 检测逻辑，识别哪些页面需要 OCR。
2. 将 OCR 任务放入低优先级后台队列。
3. OCR 结果落库，避免重复识别。
4. 在索引状态或格式支持面板中展示 OCR 状态、耗时和结果可用性。
5. 让用户能手动对指定目录或指定文件触发 OCR。
```

完成标准：

```text
用户可以对扫描版 PDF 触发 OCR，OCR 结果能进入搜索索引，并且状态可见、结果可复用。
```

---

### 3.2 图片文字识别

当前缺口：

```text
图片文字识别还没有形成完整闭环。
```

建议实现：

```text
1. 识别文档中的图片块是否需要 OCR。
2. 对图片内容提取 OCR 文本并与原 chunk 关联。
3. 图片 OCR 结果在预览和搜索结果中可追溯。
```

完成标准：

```text
文档中的图片如果包含文字，可以被识别并参与搜索。
```

---

### 3.3 XLSX 支持

当前缺口：

```text
XLSX 还没有进入完整的解析闭环。
```

建议实现：

```text
1. 提取工作表文本内容。
2. 保留 sheet 名、行列范围、单元格上下文。
3. 将表格型内容按可搜索 chunk 写入索引。
4. 对超大工作簿做基础截断和降级说明。
```

完成标准：

```text
用户可以把常见的 Excel 表格文件纳入搜索范围，并检索到表格中的文本。
```

---

### 3.4 EPUB 支持

当前缺口：

```text
EPUB 还没有形成稳定支持。
```

建议实现：

```text
1. 解析 EPUB 的章节结构。
2. 提取章节正文、标题和基础元数据。
3. 保留章节路径，方便搜索结果定位。
```

完成标准：

```text
用户可以索引常见 EPUB 电子书，并按章节内容搜索。
```

---

### 3.5 RTF 支持

当前缺口：

```text
RTF 还没有形成可用支持。
```

建议实现：

```text
1. 先走文本抽取路线。
2. 如果结构信息不足，优先保证可搜索内容而不是版面还原。
3. 对失败样本做明确降级提示。
```

完成标准：

```text
用户可以索引常见 RTF 文件，最少做到正文可搜索。
```

---

### 3.6 格式支持状态说明

当前缺口：

```text
缺少统一的“哪些格式已支持、哪些格式在处理中、哪些格式暂不支持”的状态说明。
```

建议实现：

```text
1. 在设置页或索引状态页增加格式支持说明。
2. 按“已支持 / 部分支持 / 计划支持 / 暂不支持”分类。
3. 对 OCR、PPTX、XLSX、EPUB、RTF、旧版 doc 给出清晰提示。
```

完成标准：

```text
用户可以一眼看出 SeekMind 当前支持哪些格式，以及这些格式的处理边界。
```

---

## 4. 建议实施顺序

按当前产品成熟度和实现依赖关系，建议按以下顺序推进：

```text
1. OCR 闭环收口
2. 图片文字识别
3. XLSX 支持
4. EPUB 支持
5. RTF 支持
6. 格式支持状态说明
```

原因是：

```text
1. OCR 是 0.8x 的核心收口点，优先级最高。
2. 图片 OCR 会影响预览和搜索结果表达，最好在 OCR 主链路稳定后再做。
3. XLSX / EPUB / RTF 属于相对独立的格式扩展，适合作为后续补齐项。
4. 格式支持状态说明需要建立在实际能力边界明确之后。
```

---

## 5. 0.8x 收口完成定义

当以下条件都满足时，可以把 `0.8x` 标记为完成：

```text
1. 扫描版 PDF 可以识别并进入 OCR 流程。
2. OCR 结果可保存、可复用、可追溯。
3. 图片中的文字可以被识别并参与搜索。
4. XLSX / EPUB / RTF 至少具备可用的基础支持。
5. PPTX 和旧版 doc 支持稳定。
6. 用户能清楚看到格式支持状态和处理边界。
7. 不破坏已有索引、搜索、预览和问答链路。
```

---

## 6. 不纳入 0.8x 的内容

以下能力不应该在 `0.8x` 作为主线展开：

```text
1. 团队协作。
2. 云同步。
3. 企业权限管理。
4. 自动 Agent 工作流。
5. 完整文档编辑器。
6. 复杂版面完全还原。
7. 全量 OCR 默认开启。
```

这些能力会显著扩大复杂度，应该继续留到更后面的阶段。
