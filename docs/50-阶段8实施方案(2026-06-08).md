# 阶段 8 实施方案：性能、打包和商业化准备

本文档用于指导阶段 8 的推进，目标是把 SeekMind 从“本地可用工具”推进到“可发布、可试用、可排障”的产品状态。

## 1. 实施目标

阶段目标：
- 建立性能基线
- 补齐发布基础设施
- 为后续升级、恢复和商业化留出结构

核心原则：
- 先可观测，再优化
- 先可恢复，再扩能力
- 先做最小商业化骨架，不急于完整授权系统

## 2. 推荐拆分

建议把阶段 8 拆成四条主线：

### 2.1 性能与稳定性主线

包含：
- 性能基线
- 大文档库压测
- 崩溃恢复补强

### 2.2 数据与索引演进主线

包含：
- 数据库迁移
- 索引版本迁移
- 自动重建策略

当前进展：
- 数据库迁移已具备持续演进能力。
- 索引版本迁移机制已落地，全文索引与语义向量索引都能在版本不兼容时触发重建。

### 2.3 发布与排障主线

包含：
- 日志导出
- 自动更新
- 首次启动引导
- 隐私说明

### 2.4 商业化主线

包含：
- License / Pro 开关
- 本地 feature gating
- UI 锁定态和提示文案

## 3. 推荐落地顺序

### 3.1 第一批：先补发布底座

优先实现：
- 性能基线
- 大文档库压测
- 日志导出

原因：
- 这是发布前最缺的底层基础设施。
- 这些能力会直接影响后续问题定位和升级安全性。

### 3.2 第二批：补首次使用和升级体验

优先实现：
- 首次启动引导
- 自动更新
- 隐私说明

原因：
- 这些能力直接影响真实用户试用体验。

### 3.3 第三批：补商业化最小骨架

优先实现：
- License / Pro 功能开关

原因：
- 没有 feature gate，就无法做最基础的试用/付费区分。

## 4. 当前代码基础

从当前仓库现状看，阶段 8 已经具备一些基础：
- SQLite schema 已有持续演进迁移逻辑
- 索引任务已有 checkpoint 和 resume
- 已有底部日志面板
- 已有日志导出能力
- 设置页已有部分隐私说明片段

但这些还不足以构成完整阶段 8 交付，因为还缺：
- 分档 benchmark 与压测的完整回归体系
- 更真实的大文档库压测样本和持续压测模式
- 日志导出
- 自动更新
- 首次引导
- 授权骨架

## 5. 关键风险

1. 没有基线数据时，性能优化会失焦。
2. 没有索引版本迁移时，后续结构演进容易破坏旧数据。
3. 没有日志导出时，真实用户问题很难复现。
4. 没有首次引导时，用户第一次试用成本偏高。
5. 没有 feature gate 时，商业化路径无法落地。

## 6. 最低验收标准

阶段 8 完成的最低标准建议为：
- 有可重复执行的性能基线与压测结果
- 用户可以导出日志用于排障
- 新用户有首次启动引导
- 应用具备最小的更新入口与 Pro 开关骨架
- 索引与数据库升级具备可解释的迁移机制

## 7. 结论

阶段 8 的重点不是继续扩功能，而是建立“发布能力”。  
如果阶段 7 解决的是“能不能搜索和理解”，阶段 8 解决的就是：
- 能不能稳定发布
- 能不能定位问题
- 能不能持续演进
- 能不能开始收费
