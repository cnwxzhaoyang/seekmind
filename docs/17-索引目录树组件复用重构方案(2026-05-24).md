# 索引目录树组件复用重构方案

## 背景

当前系统里有 3 处都在展示索引目录树：

1. `文档切片` 页中的索引目录
2. `文档目录` 页中的目录列表
3. `索引状态` 页中的索引目录区域

这三处的底层数据都来自同一份 `IndexDirView[]`，但现在各自实现了自己的树构造、展开状态、选中态和展示逻辑，导致：

- 结构重复，维护成本高
- 树行为不一致，容易出现状态偏差
- 新增虚拟目录、tooltip、树节点动作时需要改三份代码

因此建议把“索引目录树”拆成可复用的公共层，再由页面注入各自差异化能力。

## 目标

1. 复用同一套目录树骨架和树构造逻辑。
2. 保留三页各自不同的交互和动作，不把业务逻辑硬塞进树组件。
3. 统一展开 / 收起、tooltip、选中态、虚拟目录展示规则。
4. 为后续目录拖拽导入、虚拟目录、目录树统计扩展留出接口。

## 不做什么

1. 不把三页强行合并成一个大页面。
2. 不把 API 调用写进树组件内部。
3. 不把 `文档切片`、`文档目录`、`索引状态` 的业务动作做成完全相同的一套按钮。

## 复用边界

建议拆成两层复用：

### 1. 公共 composable

负责目录树数据和状态，不负责 UI 渲染：

- 扁平 `IndexDirView[]` 构树
- 目录路径归一化
- 父子节点计算
- 展开 / 收起状态
- 默认展开策略
- 选中目录后自动展开祖先节点
- 虚拟目录节点兼容

建议命名：

- `useIndexDirTree`

### 2. 公共树组件

负责目录树外观和节点交互，不负责业务动作：

- 树节点渲染
- 缩进层级
- 展开 / 收起图标
- tooltip
- 节点高亮
- badge / meta / actions 的插槽

建议命名：

- `SeekMindIndexTree`

## 组件拆分建议

### `useIndexDirTree`

输入：

- `IndexDirView[]`
- `selectedPath`
- `virtualDirPath`

输出：

- `treeNodes`
- `visibleRows`
- `isExpanded(path)`
- `toggleExpanded(path)`
- `expandAncestors(path)`
- `findParentPath(path)`
- `getDisplayName(path)`

### `SeekMindIndexTree`

Props：

- `rows`
- `selectedPath`
- `mode`
- `showPathTooltip`
- `virtualDirPath`

Slots：

- `node-label`
- `node-meta`
- `node-status`
- `node-actions`
- `empty`

建议支持的模式：

- `chunks`：用于 `文档切片`
- `library`：用于 `文档目录`
- `status`：用于 `索引状态`

## 三页接入方式

### 1. `文档切片`

职责：

- 选择目录
- 过滤目录
- 显示目录节点状态

树节点展示：

- 目录名
- 文档数 / 切片数
- 启用状态

树节点动作：

- 点击选择目录
- 展开 / 收起
- tooltip 显示完整路径

### 2. `文档目录`

职责：

- 目录管理
- 添加目录
- 拖拽导入
- 启停 / 删除 / 重索引

树节点展示：

- 目录名
- 文档数 / 切片数
- 启用状态
- 虚拟目录标记

树节点动作：

- 启用 / 禁用
- 重索引
- 删除
- 虚拟目录节点只展示，不执行“删除真实目录”动作

### 3. `索引状态`

职责：

- 展示当前索引目录概览
- 与任务状态联动
- 失败文件与目录关联查看

树节点展示：

- 目录名
- 文档数 / 切片数
- 状态

树节点动作：

- 点击切换查看某个目录的状态
- 原则上不提供破坏性操作

## 落地顺序

### 第 1 步：抽公共树数据层

先从 `ChunksPage.vue` 抽出树构造和展开逻辑到 `useIndexDirTree`，保证：

- 不改页面业务行为
- 只抽纯逻辑
- 先让 `文档切片` 继续正常工作

验收：

- `文档切片` 的树行为不变
- `npm run build` 通过

### 第 2 步：抽公共树展示组件

将树节点行抽成 `SeekMindIndexTree`，先支持：

- 基础节点
- tooltip
- 展开按钮
- badge
- 高亮

验收：

- `文档切片` 视觉不退化
- 节点可正常展开 / 收起
- 节点可正常选中

### 第 3 步：接入 `文档目录`

把 `LibraryPage` 的目录列表改为复用公共树组件。

保留 `LibraryPage` 自己的：

- 添加目录
- 拖拽导入
- 重索引
- 启停 / 删除按钮

验收：

- 目录树可以正常显示
- 拖拽导入后能刷新树
- 虚拟目录能正确显示

### 第 4 步：接入 `索引状态`

把 `StatusPage` 的目录区域也改成复用公共树组件。

保留 `StatusPage` 自己的：

- 任务摘要
- 进度条
- 失败文件列表

验收：

- 目录树展示与前两页一致
- 状态页不会出现不同步的目录渲染逻辑

## 数据兼容

### 虚拟目录

虚拟目录不应当被当成真实文件系统路径。

建议前端只通过以下字段识别：

- `path`
- `is_virtual`

如果后端暂时还没有 `is_virtual`，前端可先用固定路径前缀识别，例如：

- `virtual://`

### 路径展示

树节点主文本展示目录名。

完整路径只在：

- tooltip
- 调试信息
- 右侧详情

里出现。

## 验收标准

1. 三个页面都只保留一套目录树逻辑来源。
2. 目录树展开 / 收起规则一致。
3. tooltip 与完整路径展示一致。
4. 虚拟目录在三页中的表现一致。
5. 三页分别保留自己的业务动作，不互相污染。
6. 构建通过：

```bash
npm run build
cargo check --manifest-path src-tauri/Cargo.toml
```

## 推荐实现清单

1. 抽 `useIndexDirTree`
2. 抽 `SeekMindIndexTree`
3. 让 `ChunksPage` 先迁移
4. 让 `LibraryPage` 再迁移
5. 让 `StatusPage` 最后迁移
6. 统一三页的目录 tooltip、节点高亮和 badge 规则
