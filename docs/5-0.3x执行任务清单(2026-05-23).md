# DocMind 0.3.x 执行任务清单

本文档把 [0.3.x 搜索体验增强实施方案](./3-0.3x搜索体验增强实施方案(2026-05-23).md) 拆成更偏执行层的任务清单，目标是方便直接排期、分工和验收。

`0.3.x` 的边界保持不变：

```text
只增强关键词搜索体验，不引入语义搜索、向量库和问答。
```

---

## 1. 里程碑顺序

建议按下面顺序推进：

```text
M1：命中高亮 + snippet 改进
M2：按文档聚合 + 折叠展示
M3：排序策略增强
M4：搜索历史 + 常用入口
M5：收藏与最近查看
```

每个里程碑都尽量做到“前后兼容、可单独验收”。

---

## 2. 后端任务

### 2.1 命中高亮基础数据

目标：

```text
让前端可以稳定渲染高亮，而不是自己猜测命中位置。
```

要做的事：

```text
1. 搜索结果增加命中字段来源。
2. 返回命中文本片段的起止位置。
3. 返回 snippet 的候选片段集合。
4. 标记命中来源是文件名、标题还是正文。
```

建议接口字段：

```text
matched_field
match_spans
snippet
snippet_start
snippet_end
```

验收点：

```text
后端返回的数据足以让前端高亮，不依赖额外二次解析。
```

---

### 2.2 聚合查询支持

目标：

```text
让搜索接口同时支持 chunk 级结果和文档级聚合视图。
```

要做的事：

```text
1. 保留 chunk 级命中结果。
2. 增加文档级聚合统计。
3. 输出每个文档的命中 chunk 数。
4. 输出文档层级排序信息。
```

建议返回字段：

```text
document_id
document_title
document_path
matched_chunks
matched_count
top_chunk
```

验收点：

```text
同一个文档的多个 chunk 可以在一次查询里被聚合出来。
```

---

### 2.3 排序权重

目标：

```text
让更可能正确的结果排在前面，且排序逻辑可解释。
```

要做的事：

```text
1. 文件名命中加权。
2. 标题命中加权。
3. 正文命中保持基础相关性。
4. 最近修改时间轻量加权。
5. 最近打开时间轻量加权。
6. 给出最终排序分值和分值组成。
```

建议记录字段：

```text
score_total
score_content
score_title
score_filename
score_recent_modified
score_recent_opened
```

验收点：

```text
调试页可以解释“为什么这个结果排前面”。
```

---

### 2.4 搜索历史和常用入口

目标：

```text
减少重复检索成本，让用户能快速回到常用搜索和常用目录。
```

要做的事：

```text
1. 保存搜索词历史。
2. 保存最近打开的文档。
3. 保存常用目录。
4. 给收藏接口留后门。
```

建议接口：

```text
save_search_history(query)
list_search_history(limit)
record_recent_document(path)
list_recent_documents(limit)
save_favorite_query(query)
list_favorite_queries()
```

验收点：

```text
用户不需要每次手动输入完整关键词，也能快速回到常用目录和常用结果。
```

---

### 2.5 最小收藏能力

目标：

```text
只做轻量收藏，不做复杂标签体系。
```

要做的事：

```text
1. 收藏搜索词。
2. 收藏文档。
3. 收藏结果项。
```

验收点：

```text
收藏数据可持久化，重启后不丢。
```

---

## 3. 前端任务

### 3.1 搜索结果高亮

目标：

```text
让用户一眼看到命中位置。
```

要做的事：

```text
1. 搜索结果卡片支持关键词高亮。
2. 右侧预览支持同样的高亮。
3. 支持多关键词高亮。
4. 支持长 snippet 截断和展开。
```

页面建议：

```text
SearchPage
SearchResultCard
ResultPreviewPanel
```

验收点：

```text
结果卡片里命中的词必须明显可见，不需要靠肉眼在整段文本里找。
```

---

### 3.2 文档聚合展示

目标：

```text
同一文档多个命中段落先聚合，再展开查看细节。
```

要做的事：

```text
1. 默认展示文档聚合卡片。
2. 展开后查看 chunk 明细。
3. 展示每个文档的命中数。
4. 展示命中来源的标题路径和页码。
```

页面建议：

```text
SearchResultGroupCard
SearchResultChunkList
```

验收点：

```text
同一文档不会在列表中刷屏出现多次，默认先看文档，再看 chunk。
```

---

### 3.3 排序信息可视化

目标：

```text
让用户知道排序结果为什么会这样排。
```

要做的事：

```text
1. 显示命中字段标签。
2. 显示最近修改 / 最近打开的影响标记。
3. 在调试面板里展示分值拆解。
```

验收点：

```text
用户或开发者可以大致看懂每条结果的排序来源。
```

---

### 3.4 搜索历史和快捷入口

目标：

```text
把“重复输入”变成“一键再次搜索”。
```

要做的事：

```text
1. 搜索框下方展示最近搜索词。
2. 点击历史词即可重新搜索。
3. 侧边栏展示常用目录。
4. 最近查看文档支持快速打开。
```

验收点：

```text
高频操作尽量不再需要重新输入和重新找入口。
```

---

### 3.5 收藏入口

目标：

```text
让用户把高频搜索和高频结果留下来。
```

要做的事：

```text
1. 在结果卡片上加收藏按钮。
2. 在历史列表上加收藏按钮。
3. 在文档切片页显示收藏状态。
```

验收点：

```text
收藏态在多个页面展示一致。
```

---

## 4. 数据库迁移任务

### 4.1 新增搜索历史表

建议表：

```text
search_history
```

建议字段：

```text
id
query
normalized_query
created_at
hit_count
last_hit_at
```

---

### 4.2 新增最近文档表

建议表：

```text
recent_documents
```

建议字段：

```text
id
path
title
last_opened_at
open_count
```

---

### 4.3 新增收藏表

建议表：

```text
favorites
```

建议字段：

```text
id
favorite_type
target
title
created_at
updated_at
```

其中：

```text
favorite_type = query | document | result
target = 具体关键词或文件路径或 chunk id
```

---

### 4.4 搜索统计表

建议表：

```text
document_search_stats
```

建议字段：

```text
document_path
last_hit_at
hit_count
last_query
last_score
```

---

### 4.5 迁移原则

```text
1. 新增表优先，不破坏已有 documents / chunks / index_dirs。
2. 老数据默认可用，缺省值可回填。
3. 搜索历史和收藏都应支持幂等写入。
```

## 6. 验收测试点

### 6.1 高亮

```text
1. 搜索结果中关键词高亮可见。
2. 右侧预览高亮一致。
3. 多关键词查询高亮正常。
```

### 6.2 聚合

```text
1. 同一文档多个结果默认聚合。
2. 展开后能看到每个 chunk。
3. 命中数与实际结果一致。
```

### 6.3 排序

```text
1. 文件名命中的结果优先度更高。
2. 标题命中的结果优先度更高。
3. 排序分值可调试。
```

### 6.4 历史与收藏

```text
1. 历史词可点击重搜。
2. 最近文档可快速打开。
3. 收藏在重启后仍然存在。
```

---

## 7. 推荐节奏

如果按小步快跑推进，建议拆成 3 个 sprint：

```text
Sprint 1：高亮 + snippet
Sprint 2：聚合 + 排序
Sprint 3：历史 + 收藏 + 常用入口
```

每个 sprint 都应单独可发布，不要等全部做完再合并。

---

## 8. 结束标准

`0.3.x` 执行完成时，DocMind 应达到：

```text
1. 结果一眼能看出是否命中。
2. 同一文档的结果不会太吵。
3. 排序更符合直觉。
4. 重复搜索和重复打开更省事。
5. 搜索调试能力仍然保留。
```

达到这个标准后，产品才适合进入 `0.4.x` 的语义搜索阶段。
