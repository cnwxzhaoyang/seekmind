# 结构化 Chunk 预览第一阶段实施方案

本文档用于把当前“纯文本 chunk 预览”升级为“结构化文本预览”的第一阶段落地方案。目标是先解决“chunk 预览是一团长串、图标和结构信息丢失”的问题，同时不改动搜索主链路、Tantivy 索引链路和语义检索链路。

## 1. 目标

第一阶段只做结构化文本预览，不做图片、OCR、复杂 PDF 表格还原。

本阶段目标：

1. 预览页能区分标题、段落、表格、列表、代码、引用。
2. Markdown、HTML、DOCX 文档的表格不再压平成一行长文本。
3. chunk 预览能够回到结构化 block，而不是只显示纯文本 snippet。
4. 兼容旧数据，老索引没有 blocks 时仍然可以正常展示 snippet。

不做的内容：

1. 不改 Tantivy 搜索结构。
2. 不改语义 embedding 的检索逻辑。
3. 不做图片抽取、OCR、asset protocol。
4. 不做 PDF 图片和复杂表格还原。
5. 不改变现有 `chunks.snippet` 的用途，搜索结果仍然可以继续用它。

## 2. 当前现状

当前项目里已经有部分基础能力，但还没有形成完整的结构化预览链路。

1. Python 侧已经有内部 `Block` 模型，能够在解析过程中识别标题、段落、表格等结构。
2. Python 返回给 Rust 的仍然是 `ParsedDocument + ParsedChunk[]`，没有对外暴露 `blocks`。
3. Rust SQLite 里目前主要只有 `documents` 和 `chunks`，chunk 的展示字段仍是纯文本。
4. 前端 `ChunksPage` 直接渲染 `chunk.snippet`，没有 block 级渲染器。

因此，本阶段需要做的是把“内部已经有的结构化信息”正式传递到 Rust 和前端。

## 3. 总体方案

建议采用下面的数据流：

`Parser -> DocumentBlocks -> SearchChunks -> index`

其中：

1. `DocumentBlocks` 负责保留原始结构，用于预览。
2. `SearchChunks` 负责检索和 embedding，继续保留现有搜索主链路。
3. 搜索结果命中 chunk 后，再根据 chunk 关联的 block 范围回查 blocks，用于预览展示。

这意味着：

1. 搜索仍然只认 chunk。
2. 预览不再只认 snippet。
3. 预览能力可以逐步增强，不会影响搜索稳定性。

## 4. 数据模型改造

### 4.1 Python 输出模型

建议在 Python 侧新增 `ParsedBlock`，并让 `ParsedDocument` 返回 `blocks`。

第一阶段建议保留如下字段：

```json
{
  "title": "文档标题",
  "file_type": "docx",
  "content": "...",
  "chunks": [
    {
      "heading": "章节路径",
      "page_no": 1,
      "text": "用于检索的纯文本",
      "order": 1,
      "score": 1.0,
      "block_indexes": [1, 2, 3]
    }
  ],
  "blocks": [
    {
      "block_index": 1,
      "type": "heading",
      "text": "一、项目背景",
      "heading": "一、项目背景",
      "level": 1,
      "page_no": null,
      "markdown": null,
      "html": null
    }
  ]
}
```

推荐字段：

1. `block_index`
2. `type`
3. `text`
4. `heading`
5. `level`
6. `page_no`
7. `markdown`
8. `html`

### 4.2 Rust 侧模型

建议在 Rust 侧新增：

1. `ParsedBlock`
2. `PreviewBlockView`

并让 `ParsedDocument` 和 `ChunkView` 追加对应字段。

建议结构：

```rust
pub struct ParsedBlock {
    pub block_index: usize,
    pub block_type: String,
    pub text: String,
    pub heading: Option<String>,
    pub level: Option<u32>,
    pub page_no: Option<u32>,
    pub markdown: Option<String>,
    pub html: Option<String>,
}
```

```rust
pub struct PreviewBlockView {
    pub block_index: usize,
    pub block_type: String,
    pub text: String,
    pub heading: String,
    pub level: Option<u32>,
    pub page: Option<u32>,
    pub markdown: String,
    pub html: String,
}
```

`ChunkView` 增加：

```rust
pub preview_blocks: Vec<PreviewBlockView>
```

## 5. Python 侧实施

### 5.1 文件范围

主要修改：

1. `parser/seekmind_parser/models.py`
2. `parser/seekmind_parser/parser.py`

### 5.2 需要做的事

1. 新增 `ParsedBlock` 数据类。
2. `ParsedDocument.to_dict()` 输出 `blocks`。
3. 内部 `Block` 增加必要字段：
   1. `level`
   2. `markdown`
   3. `html`
4. Markdown 解析补齐 block 类型：
   1. 标题 -> `heading`
   2. 普通文本 -> `paragraph`
   3. 表格 -> `table`
   4. fenced code -> `code`
   5. 引用 -> `quote`
   6. 列表 -> `list`
5. HTML 解析补齐 block 类型：
   1. `h1-h6` -> `heading`
   2. `p` -> `paragraph`
   3. `table` -> `table`
   4. `pre/code` -> `code`
   5. `blockquote` -> `quote`
   6. `li/ul/ol` -> `list`
6. DOCX 解析补齐 block 类型：
   1. 标题样式 -> `heading`
   2. 段落 -> `paragraph`
   3. 表格 -> `table`
   4. 第一阶段先输出表格 markdown，不强求 html
7. `build_chunks()` 在生成 chunk 时记录 `block_indexes`。
8. PDF 第一阶段仍按纯文本段落 block 处理。

### 5.3 解析规则建议

1. `heading` 只负责层级和章节路径，不参与正文展示。
2. `paragraph` 作为默认正文 block。
3. `table` 需要保留 `text` 和 `markdown`，预览时优先渲染 markdown。
4. `code` 需要保留原始代码块文本，前端使用等宽字体展示。
5. `quote` 和 `list` 先做基础结构化展示，不做复杂富文本。

### 5.4 Python 验收点

1. Markdown 文档能区分标题、表格、代码块、引用。
2. DOCX 表格不再被压平成一整行长文本。
3. 解析结果里有 `blocks`，并且 chunk 能记录 `block_indexes`。
4. 老文档即使没有完整结构字段，也能保持可解析、可回退。

## 6. Rust 侧实施

### 6.1 文件范围

主要修改：

1. `src-tauri/src/seekmind/parser/types.rs`
2. `src-tauri/src/seekmind/models.rs`
3. `src-tauri/src/seekmind/storage/db.rs`
4. `src-tauri/src/seekmind/commands.rs`

### 6.2 需要做的事

1. 扩展 parser 协议类型，支持 `ParsedBlock`。
2. `ParsedDocument` 新增 `blocks`。
3. `ParsedChunk` 新增 `block_indexes`。
4. 新增 `PreviewBlockView`。
5. `ChunkView` 新增 `preview_blocks`。
6. `store_document()` 同时写入 blocks 和 chunks。
7. `list_document_chunks()` 回查 blocks 并填充 `preview_blocks`。

### 6.3 数据库存储建议

新增表：

```sql
CREATE TABLE IF NOT EXISTS document_blocks (
    id TEXT PRIMARY KEY,
    document_id TEXT NOT NULL,
    block_index INTEGER NOT NULL,
    block_type TEXT NOT NULL,
    text TEXT NOT NULL,
    heading TEXT NOT NULL DEFAULT '',
    level INTEGER,
    page INTEGER,
    markdown TEXT NOT NULL DEFAULT '',
    html TEXT NOT NULL DEFAULT '',
    metadata_json TEXT NOT NULL DEFAULT '',
    FOREIGN KEY(document_id) REFERENCES documents(id) ON DELETE CASCADE
);
```

给 `chunks` 增加字段：

```sql
ALTER TABLE chunks ADD COLUMN block_indexes_json TEXT NOT NULL DEFAULT '[]';
```

存储顺序建议：

1. 写 `documents`
2. 写 `document_blocks`
3. 写 `chunks`
4. `chunks` 中记录 `block_indexes_json`

### 6.4 兼容策略

1. 老数据没有 `document_blocks` 时，仍可以直接显示 `chunk.snippet`。
2. `preview_blocks` 为空时，前端自动回退到纯文本展示。
3. 不强制老索引立即重建。
4. 后续可以通过重新切片逐步补齐结构化信息。

### 6.5 Rust 验收点

1. Python 新协议能正常反序列化。
2. 文档入库时 blocks 和 chunks 都能保存。
3. `list_document_chunks()` 能返回 `preview_blocks`。
4. 老数据不会因为缺少新字段而崩溃。

## 7. 前端实施

### 7.1 文件范围

主要修改：

1. `src/types/SeekMind.ts`
2. `src/pages/ChunksPage.vue`
3. 新增 `src/components/seekmind/seekmindPreviewBlockRenderer.vue`

### 7.2 需要做的事

1. `ChunkView` 扩展 `preview_blocks`。
2. 新增 `PreviewBlockView` 类型。
3. `ChunksPage` 优先渲染结构化 blocks。
4. `preview_blocks` 为空时回退到原有 `snippet`。
5. 新增 block renderer 组件，按 block 类型分发渲染。

### 7.3 建议的渲染规则

1. `heading`：按 level 显示层级标题。
2. `paragraph`：普通正文。
3. `table`：优先渲染 markdown 表格，没有 markdown 时显示文本。
4. `code`：使用等宽字体和代码块背景。
5. `quote`：左边框引用样式。
6. `list`：列表样式展示。
7. 其他类型：回退显示原始文本。

### 7.4 前端验收点

1. Markdown 表格不再是单行长串。
2. 代码块有明确的代码块视觉样式。
3. 标题、段落、引用、列表有明显区分。
4. 旧数据仍可正常显示。

## 8. 推荐开发顺序

建议按下面顺序推进：

1. Python：补 `ParsedBlock` 和 `blocks` 输出。
2. Rust：补解析协议类型。
3. SQLite：新增 `document_blocks` 和 `chunks.block_indexes_json`。
4. Rust：`store_document()` 写入 blocks。
5. Rust：`list_document_chunks()` 返回 `preview_blocks`。
6. 前端：新增 `seekmindPreviewBlockRenderer.vue`。
7. 前端：`ChunksPage` 接入结构化预览。
8. 回归搜索、重切片、重新索引。

## 10. 结论

这个第一阶段方案是可实施的，且适合当前项目状态。

原因很直接：

1. Python 侧已经有结构化解析基础，不是从零开始。
2. Rust 侧可以通过兼容性字段逐步接入，不需要推翻现有索引。
3. 前端可以先通过 fallback 保证旧数据可用，风险可控。

如果按这个方案推进，先能解决“chunk 预览不成结构”的核心问题，再为后续图片、OCR、PDF 结构化预览留下空间。
