# SeekMind RAG 回归评测集

## 1. 目的

这份清单用于验证 Python 侧 RAG Graph 的三个核心目标：

1. 能先检索，再组织回答。
2. 答案能稳定流式输出，且终态不跳变。
3. 证据不足时能够明确拒答，而不是硬编。

## 2. 评测方式

建议每次修改 `parser/SeekMind_parser/rag/` 后，至少抽查以下样本：

1. 在本地桌面应用里发起问答。
2. 观察 `progress -> answer_delta -> final` 事件链是否完整。
3. 检查最终 `state` 是否符合预期。
4. 检查答案正文是否只依赖当前来源，不出现明显无源句子。
5. 检查拒答样本是否落到 `insufficient_evidence`。

## 3. 回归样本

| 编号 | 问题 | 预期状态 | 重点检查项 |
|---|---|---|---|
| RAG-001 | 什么是 WBS | answered | 答案应包含引用编号，且正文不应先出现一版再整体跳变 |
| RAG-002 | 在 Oracle 中创建表空间 | answered | 应给出可执行步骤，引用应落在事实句后 |
| RAG-003 | 什么是工作流引擎 | answered | 解释类问题应保持结构化，不要把概念写散 |
| RAG-004 | 这两者有什么区别 | answered | 依赖最近上下文做指代消解，不能丢失指代对象 |
| RAG-005 | 随便问一个当前库里没有证据的问题 | insufficient_evidence | 必须进入拒答，不要编造答案 |
| RAG-006 | 只给一个很窄的目录范围后问跨目录问题 | insufficient_evidence | 范围太窄时应拒答或提示缩小范围 |

## 4. 通过标准

1. `answered` 样本应能流式输出最终正文。
2. 最终正文中的核心事实句应带引用编号。
3. `insufficient_evidence` 样本应明确拒答，且状态不能误记为 `answered`。
4. 流式过程中的 UI 不应出现“先一版、后整体跳变”的二次重放。
5. 任何样本若出现空答、引用大面积缺失、或修复后仍错误，都视为回归失败。

## 5. 后续建议

下一步可以把这份清单接成一个 Python CLI 或单元测试入口，自动跑一遍：

1. `retrieve`
2. `rank`
3. `pack`
4. `draft`
5. `verify`
6. `repair / refuse`
7. `finalize`

这样就能把“看起来能用”变成“每次改动都可回归”。
