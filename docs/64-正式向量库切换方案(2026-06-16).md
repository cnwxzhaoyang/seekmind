# 正式向量库切换方案

## 1. 背景

当前 SeekMind 的语义向量能力已经可用，但实现方式仍是：

- 向量写入 SQLite 的 `chunk_embeddings` 表
- 检索时把所有向量读出
- Rust 侧按余弦相似度全量扫描排序

这套方案适合早期验证，但不是正式的向量库形态。随着文档量增加，它会遇到两个问题：

1. 检索性能随向量数量线性退化。
2. 向量存储、查询和生命周期管理都耦合在 SQLite 表结构里，后续扩展成本高。

因此需要引入正式向量库，并把当前实现收口成可切换的存储层。

## 2. 选型结论

### 2.1 推荐顺序

建议优先接入 `sqlite-vec`，再视规模考虑 `LanceDB`。

推荐顺序：

1. `sqlite-vec`
2. `LanceDB`

### 2.2 选择理由

#### sqlite-vec

适合当前阶段的原因：

- 项目主存储已经是 SQLite。
- 文档、任务、状态、语义元数据都围绕 SQLite 管理。
- 向量直接落在 SQLite 生态里，迁移成本最低。
- 对桌面端打包和数据目录管理更简单。

适合的场景：

- 当前 SeekMind 的中小规模知识库
- 需要尽快从全量扫描升级到真正的向量检索
- 希望尽量少改现有数据模型和清理流程

#### LanceDB

适合后续扩展阶段：

- 向量规模明显增大
- 需要更完整的向量索引能力
- 希望向量库与 SQLite 元数据进一步解耦
- 可以接受新增独立数据目录与更重的依赖

## 3. 目标架构

### 3.1 统一抽象层

先在语义层抽一个统一接口，不让业务代码直接依赖具体存储实现。

建议接口职责：

- 写入或更新向量
- 删除单文档向量
- 删除单目录向量
- 清空所有向量
- 按 query vector 检索 topK
- 统计当前向量数量

业务层只调用这一层，不再直接操作 `chunk_embeddings` 的 JSON 字段或全量扫描逻辑。

### 3.2 保留元数据表

即使切到正式向量库，也建议继续保留 SQLite 里的元数据表，用来存：

- `chunk_id`
- `document_id`
- `model_id`
- `text_hash`
- `status`
- `created_at`
- `updated_at`
- 索引状态

这样可以继续复用现有：

- 清空索引
- 删除文档
- 删除目录
- 语义重建进度
- 状态栏展示

## 4. sqlite-vec 落地方案

### 4.1 数据分层

推荐把现有向量数据拆成两层：

1. 向量层：`sqlite-vec` 虚拟表
2. 元数据层：SQLite 普通表

向量层只负责向量检索；元数据层负责业务关联和状态管理。

### 4.2 推荐表结构

可以新增两张表：

```sql
CREATE VIRTUAL TABLE IF NOT EXISTS chunk_embedding_vec
USING vec0(embedding float[512]);
```

```sql
CREATE TABLE IF NOT EXISTS chunk_embedding_meta (
    rowid INTEGER PRIMARY KEY,
    chunk_id TEXT NOT NULL UNIQUE,
    document_id TEXT NOT NULL,
    model_id TEXT NOT NULL,
    text_hash TEXT NOT NULL,
    status TEXT NOT NULL DEFAULT 'ready',
    created_at INTEGER NOT NULL DEFAULT 0,
    updated_at INTEGER NOT NULL DEFAULT 0
);
```

说明：

- `vec0` 存向量。
- `chunk_embedding_meta` 存业务字段。
- 两者通过 `rowid` 或 `chunk_id` 关联。

### 4.3 查询方式

检索时不再全量读 JSON，而是直接做向量近邻查询。

查询流程：

1. Python 侧生成 query embedding。
2. Rust 侧调用 vector store 搜索。
3. 返回 `chunk_id + score`。
4. 再回表补充 `file_name / path / heading / snippet` 等业务信息。

结果分数可以继续保留当前搜索链路的融合逻辑。

## 5. LanceDB 落地方案

### 5.1 适用场景

如果后续知识库规模继续增长，或者需要更成熟的向量索引和过滤能力，可以把 `VectorStore` 实现切到 `LanceDB`。

### 5.2 主要差异

与 `sqlite-vec` 相比，`LanceDB` 的变化更大：

- 需要独立数据目录
- 向量和元数据不再天然和 SQLite 同库
- 需要额外考虑初始化、迁移、备份和清理

因此不建议当前直接跳过 `sqlite-vec` 上 `LanceDB`。

## 6. 迁移步骤

### 6.1 第一阶段：抽象接口

先把当前实现包成统一接口：

- 当前 JSON SQLite 实现作为 `legacy` 或 `fallback`
- 新实现挂在 `VectorStore` trait 下

### 6.2 第二阶段：接入 sqlite-vec

实现：

- 向量写入
- 向量检索
- 删除文档
- 删除目录
- 清空
- 统计数量

同时保留一套开关，方便在开发期切回旧实现。

### 6.3 第三阶段：并行验证

在一段时间内可以做双轨验证：

- 主路径走 `sqlite-vec`
- 旁路保留旧实现比对结果

验证重点：

- topK 是否稳定
- 召回是否一致
- 性能是否显著优于全量扫描
- 清空/删除/重建是否行为一致

### 6.4 第四阶段：切换默认实现

确认稳定后，将默认实现切为 `sqlite-vec`，保留旧实现作为回退选项。

### 6.5 第五阶段：再评估 LanceDB

只有在向量规模和场景明确需要时，再评估是否切到 `LanceDB`。

## 7. 兼容与回退

### 7.1 兼容要求

切换时必须保留：

- 既有 SQLite 元数据
- 索引状态
- 清空索引逻辑
- 文档删除和目录删除逻辑
- 现有语义状态栏展示

### 7.2 回退要求

如果新向量库初始化失败，系统应能回退到旧实现，至少保证：

- 可继续搜索
- 可继续重建向量
- 可继续查看状态

## 8. 验收标准

切换完成后至少满足：

1. 语义检索不再依赖全量扫描 JSON 向量。
2. 清空、删除、重建仍然可用。
3. 语义状态栏可正确展示向量数量和重建状态。
4. 混合检索的 RRF 融合逻辑不受存储切换影响。
5. 开发期 trace 仍可输出 `keyword_top_k`、`vector_top_k`、`final_ranked` 等信息。

## 9. 建议实施顺序

1. 抽 `VectorStore` 接口。
2. 保留旧实现并命名为 fallback。
3. 接入 `sqlite-vec`。
4. 重新跑重建和搜索回归。
5. 如有需要，再评估 `LanceDB`。

## 10. 结论

SeekMind 现在不适合直接切到重量级向量库。

最稳妥的路径是：

1. 先抽象向量存储接口。
2. 先接 `sqlite-vec`，把“全量扫描 JSON”替换掉。
3. 等规模和需求上来后，再评估是否迁到 `LanceDB`。
