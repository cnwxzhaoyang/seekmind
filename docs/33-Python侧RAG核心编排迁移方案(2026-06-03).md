# Python 侧 RAG 核心编排迁移方案

## 1. 结论

DocMind 的 RAG 核心能力应放在 Python sidecar。

Rust/Tauri 侧继续负责：

1. 前端命令入口。
2. 数据库路径和配置读取。
3. 任务创建、取消和状态持久化。
4. Python event 消费。
5. UI progress event 转发。

Python sidecar 负责：

1. query rewrite。
2. 多路召回。
3. 候选合并。
4. 证据重排。
5. 相邻 chunk 合并。
6. prompt 组装。
7. 模型调用。
8. 引用校验。
9. 修复、重试或拒答。

这符合 [20-Python侧任务调用经验规范(2026-05-24).md](/Users/zhaoyang/Desktop/enjoy/docmind/docs/20-Python侧任务调用经验规范(2026-05-24).md) 里的任务流模式：Rust 发起任务，Python 执行任务并持续输出 event，Rust 消费 event 并驱动 UI。

## 2. 当前问题

当前 QA 链路里，Rust 侧已经开始承载太多 RAG 逻辑：

1. `src-tauri/src/docmind/qa/context.rs` 负责多查询召回、候选合并、证据重排和相邻 chunk 合并。
2. `src-tauri/src/docmind/qa/commands.rs` 负责 prompt 组装、答案生成、引用校验、重试和失败降级。
3. Python sidecar 只承担解析和 embedding，没有进入问答编排主链路。

这会带来三个问题：

1. RAG 策略会继续在 Rust 中膨胀，后续接 reranker、LLM judge、评测集都不顺手。
2. Rust 和 Python 两侧都会理解 chunk、source、citation，边界会变模糊。
3. 如果后续再把候选搬给 Python 做二次处理，会产生不必要的大文本搬运。

## 4. 减少数据搬运原则

不要让 Rust 先查出大量候选正文，再交给 Python 做 RAG。

推荐做法：

1. Rust 只把 `db_path`、`question`、`scope_paths`、`settings` 和必要会话摘要传给 Python。
2. Python 使用 SQLite 只查询 RAG 需要的字段。
3. Python 只加载进入候选集的 chunk 正文。
4. Python 只回传最终选中的 sources。
5. `preview_blocks` 默认不随 RAG 回传，右侧引用详情需要时仍走现有按 source 加载逻辑。

这样数据流是：

```text
UI -> Rust command -> Python rag_answer_stream -> SQLite / model -> Rust event bridge -> UI
```

而不是：

```text
UI -> Rust -> SQLite 大量候选 -> Python rerank -> Rust -> Python generate -> Rust -> UI
```

## 5. Python 侧模块建议

不要继续把 RAG 逻辑塞进 `parser.py`。`parser.py` 已经偏大，RAG 应拆成新模块：

```text
parser/docmind_parser/
  rag/
    __init__.py
    models.py
    db.py
    retrieval.py
    rerank.py
    prompt.py
    verifier.py
    pipeline.py
```

建议职责：

1. `models.py`：RAG request、event、response、source 数据结构。
2. `db.py`：只封装 RAG 所需 SQLite 查询。
3. `retrieval.py`：query rewrite、多查询召回、scope 过滤。
4. `rerank.py`：规则重排，后续接 reranker。
5. `prompt.py`：source block 和 system prompt 组装。
6. `verifier.py`：引用覆盖校验，区分事实句和辅助句。
7. `pipeline.py`：串起 retrieve -> rerank -> generate -> verify -> repair/refuse。

## 6. Rust 侧迁移步骤

第一步：新增 Python RAG client。

文件建议：

```text
src-tauri/src/docmind/rag/client.rs
src-tauri/src/docmind/rag/types.rs
src-tauri/src/docmind/rag/mod.rs
```

职责：

1. 按 `parse_document_stream` 的方式启动 sidecar。
2. 发送 `rag_answer_stream` 请求。
3. 流式消费 progress event。
4. 解析最终 response。

第二步：`qa/commands.rs` 改成编排入口。

保留：

1. 权限和配置检查。
2. 会话创建。
3. job 注册和取消。
4. progress event 转发。
5. qa_history / qa_messages 持久化。

移走：

1. prompt 组装。
2. 模型请求。
3. 引用校验。
4. 重试逻辑。

第三步：保留 Rust fallback。

迁移初期可以保留当前 Rust RAG 链路作为 fallback：

1. Python sidecar 不可用时使用 Rust fallback。
2. Python RAG 返回协议错误时使用 Rust fallback。
3. 日志明确标记 `rag_source=python` 或 `rag_source=rust_fallback`。

## 7. 校验策略调整

当前“逐句硬校验”过于僵硬。Python 侧应改成软硬结合：

1. 事实句必须有有效来源标注。
2. 辅助句、过渡句、结构句允许无引用。
3. 如果核心事实无引用，先进入 repair。
4. repair 后仍无有效引用，降级为拒答。
5. 如果只是少量辅助句无引用，只给 warning，不打 failed。

事实句识别第一版可以用规则：

1. 包含数字、版本、路径、命令、配置项。
2. 包含“需要、必须、配置、执行、创建、修改、删除、支持、不支持”等动作判断。
3. 包含定义、因果、比较、步骤结论。

后续再接 LLM judge。

## 8. 实施顺序

1. 新增 Python `rag` 模块和 `rag_answer_stream` 协议。
2. Python 先复刻当前 Rust 的多查询召回、重排、chunk 合并。
3. Rust 新增 RAG sidecar client，并保留 Rust fallback。
4. QA 命令切到 Python RAG 优先。
5. 把引用校验改成软硬结合。
6. 加本地评测集，量化迁移前后效果。
7. 确认稳定后，逐步删除 Rust 侧 RAG 编排细节。

## 9. 完成标准

1. 默认问答链路走 Python `rag_answer_stream`。
2. Rust 日志能明确看到 `rag_source=python`。
3. UI 仍能看到检索、生成、校验等阶段进度。
4. 问答历史、来源摘要、引用详情不退化。
5. Python RAG 不回传大量未选中候选正文。
6. Python 不可用时 Rust fallback 可用。
7. `npm run build` 和 `cargo check --manifest-path src-tauri/Cargo.toml` 通过。
