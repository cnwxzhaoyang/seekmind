# DocMind 文档目录树形改造方案

本文档用于解决当前 `文档切片` 页中“文档目录”栏信息密度不足、路径不可读、目录层级丢失的问题。

当前目录区虽然能展示索引目录列表，但在实际使用中暴露出几个明显问题：

```text
1. 路径过长，无法看清具体目录。
2. 没有完整路径的悬停提示。
3. 子目录层级没有被表达出来。
4. 当目录很多时，当前平铺列表不利于浏览和定位。
```

---

## 1. 现状问题

### 1.1 路径过长且不可读

当前目录区直接展示完整路径作为主文本，但栏宽有限，导致路径只显示前半段，用户无法判断具体目录。

### 1.2 缺少完整路径提示

目录项没有 `tooltip`，鼠标移上去也看不到完整路径，只能靠猜测。

### 1.3 目录层级被打平

实际目录可能包含子目录，但当前展示方式把它们当作同级列表处理，丢失了父子关系。

### 1.4 不适合目录很多的场景

当索引目录数量增加后，当前列表会变得越来越长，浏览成本高，定位某个目录也不直观。

---

## 2. 改造目标

```text
1. 看得清目录名。
2. 看得见目录层级。
3. 鼠标悬停能看到完整路径。
4. 目录多时依然易于浏览和切换。
5. 不改变现有索引/切片业务逻辑。
```

---

## 3. 推荐方案

### 3.1 改成树形目录

目录区改为树结构，而不是纯平铺列表。

```text
索引目录
├─ 父目录 A
│  ├─ 子目录 A-1
│  └─ 子目录 A-2
└─ 父目录 B
   └─ 子目录 B-1
```

树形展示的好处：

```text
1. 保留目录层级。
2. 更适合查看父目录下的子目录。
3. 更容易做展开 / 折叠。
4. 更符合文件管理器的认知。
```

### 3.2 目录节点主文本改为“可读摘要”

目录节点不再把完整路径当主文本，而是显示更短、更可读的摘要：

```text
1. 优先展示最后一级目录名。
2. 如果需要，再展示简化后的层级片段。
3. 完整绝对路径放到 hover tip 里。
```

示例：

```text
主文本：AI / 面向agent编程
副文本：16 文档 · 359 切片
tooltip：/Users/zhaoyang/Documents/MarkdownHome/zhaoyang-markdown/AI/面向agent编程
```

### 3.3 节点补充信息只保留核心项

每个目录节点建议只展示以下信息：

```text
1. 目录名或简化路径。
2. 文档数。
3. 切片数。
4. 启用 / 禁用状态。
```

避免在树节点里塞太多操作，保持可扫视性。

---

## 4. 节点结构建议

### 4.1 父目录节点

父目录节点建议显示：

```text
1. 折叠 / 展开箭头。
2. 目录名。
3. 文档数 / 切片数。
4. 启用状态。
```

### 4.2 子目录节点

子目录节点建议显示：

```text
1. 更明显的缩进。
2. 同样保留文档数 / 切片数。
3. 鼠标悬停显示完整路径。
```

### 4.3 叶子节点

叶子节点可直接作为当前可索引目录或实际索引单位。

```text
1. 维持与当前目录逻辑一致。
2. 仍然支持启用 / 禁用。
3. 仍然支持重新索引。
```

---

## 5. 交互设计

### 5.1 默认展开策略

建议默认只展开当前选中的目录分支，其余分支保持折叠。

```text
1. 降低初始噪音。
2. 减少目录树首次加载的视觉压力。
3. 方便直接定位当前工作目录。
```

### 5.2 点击行为拆分

建议把“选中目录”和“展开 / 折叠”分开：

```text
1. 点击箭头：只展开 / 折叠。
2. 点击目录行：选中目录并加载文档列表。
```

### 5.3 tooltip

目录节点需要支持悬停提示：

```text
1. tooltip 显示完整绝对路径。
2. 父目录和子目录都显示。
3. 触控板和鼠标都能触发。
```

### 5.4 目录多时的辅助操作

如果后续目录很多，可以再考虑增加：

```text
1. 全部展开。
2. 全部收起。
3. 搜索目录。
```

但这些不作为第一版必做项。

---

## 6. 数据层建议

### 6.1 第一版可前端构树

如果后端已经能返回目录 `path`，第一版可以先由前端根据路径前缀自行构树。

```text
优点：
1. 不需要立即改后端协议。
2. 改造快。
3. 可以先验证树形交互是否有效。
```

### 6.2 后续更稳的后端字段

如果后续需要更稳定的结构，建议后端补充：

```text
1. parent_path
2. depth
3. is_leaf
4. children_count
```

这样前端可以减少路径推导逻辑，树结构也更可靠。

---

## 7. 推荐落地顺序

```text
1. 先改成树形 UI。
2. 再补完整路径 tooltip。
3. 再优化默认展开策略。
4. 最后再考虑目录搜索和全局折叠控制。
```

### 为什么这样排

```text
1. 树形是核心改造。
2. tooltip 是路径可读性的最低保障。
3. 默认展开策略决定初始体验。
4. 附加能力可以后补，不应阻塞主改造。
```

---

## 8. 验收标准

当以下条件满足时，可以认为目录区改造完成：

```text
1. 目录层级被正确表达。
2. 目录名可读，不再依赖完整路径。
3. 鼠标悬停能看到完整路径。
4. 目录多时仍能快速定位和切换。
5. 不影响现有文档列表和切片详情联动。
```

---

## 9. 备注

这次改造只针对目录区展示，不改变现有索引、切片和重索引逻辑。

如果后续要继续优化，可以再单独补一份：
