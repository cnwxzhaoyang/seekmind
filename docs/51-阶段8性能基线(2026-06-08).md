# 阶段 8 性能基线（2026-06-08）

本文档对应阶段 8 的第 1 项任务：建立性能基线。当前仓库已经补了一个最小可执行基线工具 `seekmind-benchmark`，用于输出本地索引规模、运行时能力和搜索延迟报告。

## 1. 目标

当前这版基线先覆盖三类数据：

- 索引规模：文档数、切片数、失败文件数、待 OCR 任务数
- 运行时能力：Python 解析、Office 转换、PDF OCR、图片 OCR、语义模型状态
- 搜索时延：按多组 query 统计 `min / avg / p50 / p95 / p99 / max`

这版还没有覆盖完整压测，不负责：

- 1,000 / 5,000 / 10,000 文档造数
- 全量索引耗时压测
- 增量索引 / 恢复索引耗时对比
- 更细粒度内存 / CPU 采样

这些内容进入阶段 8 的第 2 项“大文档库压测”。

## 2. 命令入口

### 2.1 npm 入口

```bash
npm run benchmark:baseline -- --output docs/reports/seekmind-baseline.json
```

### 2.2 cargo 入口

```bash
cargo run --manifest-path src-tauri/Cargo.toml --bin seekmind-benchmark -- --output docs/reports/seekmind-baseline.json
```

## 3. 常用参数

```bash
--output <path>         可选，输出 JSON 报告文件
--limit <number>        搜索结果上限，默认 20
--repeat <number>       每个 query 重复次数，默认 3
--history-limit <num>   无显式 query 时，回退读取搜索历史条数，默认 8
--queries-file <path>   从文本文件按行读取 query
--query <text>          指定 query，可重复传入
```

## 4. Query 选择顺序

工具会按下面顺序选择查询词：

1. `--query`
2. `--queries-file`
3. 本地搜索历史
4. 默认内置词：`设计 / 系统 / WBS / 测试 / 集成`

这样做的目的是保证：

- 新环境可以直接跑
- 已使用环境可以优先反映真实查询习惯

## 5. 输出内容

报告为 JSON，核心字段包括：

- `index_status`
- `parser_runtime`
- `semantic_status`
- `queries`
- `latency_ms`
- `max_rss_mb`

其中：

- `queries` 会保留每个 query 的单次样本
- `latency_ms` 会输出整批查询的聚合分位数
- `max_rss_mb` 当前为进程峰值 RSS 近似值

## 6. 当前定位

这不是最终压测体系，而是阶段 8 的第一层基线。

它解决的是：

- 当前机器上到底有多少索引数据
- 当前运行时能力是否可用
- 当前搜索响应速度处于什么量级

它还没有解决：

- 索引构建性能
- 超大目录极限表现
- 不同版本之间的回归对比自动汇总

## 7. 下一步

在这份基线之上，下一步应该直接接：

1. 大文档库压测脚本
2. 压测目录生成器或固定样本目录
3. 基线报告归档和回归对比
