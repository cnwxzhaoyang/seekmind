# SeekMind MVP 后阶段规划

本文档用于定义 SeekMind 在第一阶段 MVP 之后的产品演进路线，目标是让产品能力稳步推进，避免过早进入语义搜索、RAG、OCR、团队协作等高复杂度方向。

核心原则：

```text
先稳定，再增强。
先搜索，再理解。
先本地单机闭环，再考虑团队和云端。
先解决真实高频问题，再做 AI 感知能力。
```

---

## 0. 产品长期方向

SeekMind 的长期目标是成为一个本地优先的个人文档知识检索工具。

用户价值：

```text
用户不需要记住文件名，只要记得文档里讲过什么，就能快速找到对应文档和具体段落。
```

长期能力可以分为四层：

```text
第一层：本地文档内容搜索
第二层：语义相似检索
第三层：基于来源的问答和总结
第四层：个人知识库组织和持续沉淀
```

MVP 之后的所有阶段都应围绕这四层递进，不应跳跃式开发。

## 1. 阶段 0：MVP 基线版本

建议版本：

```text
0.1.x
```

阶段定位：

```text
本地文档全文搜索 MVP
```

核心目标：

```text
目录 -> 文档 -> 文本 -> 段落 -> 索引 -> 搜索 -> 定位
```

必须完成：

```text
1. 添加本地索引目录
2. 扫描 txt / md / html / docx / pdf
3. 提取正文文本
4. 按段落生成 chunk
5. 保存 documents / chunks 元数据
6. 使用 Tantivy 建立全文索引
7. 搜索返回命中文档和命中段落
8. 打开原文件
9. 在 Finder 中显示原文件
10. 展示索引进度和失败文件
```

明确不做：

```text
不做语义搜索
不做本地问答
不做 OCR
不做团队协作
不做云同步
不做复杂文件格式
不做 PDF 精确坐标跳转
```

验收标准：

```text
用户添加一个文档目录后，可以通过关键词搜索到文档中的具体段落，并能打开原文件。
```

## 2. 阶段 1：稳定性和索引质量增强

建议版本：

```text
0.2.x
```

阶段定位：

```text
让 MVP 从“能用”变成“稳定可长期使用”
```

核心目标：

```text
降低解析失败率，提升索引可靠性，让用户敢把常用文档目录交给 SeekMind。
```

关键功能：

```text
1. 增量索引
2. 文件修改检测
3. 文件删除后的索引清理
4. 重新索引单个文件
5. 重新索引单个目录
6. 清空并重建索引
7. 索引任务可暂停 / 可恢复
8. 失败文件列表和失败原因
9. 排除目录和排除扩展名配置
10. 最大文件大小限制配置
```

解析质量改进：

```text
1. Markdown 保留标题层级
2. HTML 按 h1 / h2 / h3 / p / li / table 分块
3. DOCX 提取标题、段落和表格文本
4. PDF 至少保留页码
5. 过滤页眉、页脚和重复空白
```

明确不做：

```text
不做语义搜索
不做 OCR
不做问答
不做复杂知识图谱
不做团队功能
```

验收标准：

```text
1. 用户修改、删除、新增文件后，索引结果能正确更新。
2. 用户可以知道哪些文件失败，以及为什么失败。
3. 大目录索引中断后，不需要从零开始。
```

阶段价值：

```text
这个阶段决定 SeekMind 是否能从 Demo 变成日常工具。
```

## 3. 阶段 2：搜索体验增强

建议版本：

```text
0.3.x
```

阶段定位：

```text
提升关键词搜索体验，让结果更容易判断和使用
```

核心目标：

```text
让用户不仅能搜到，还能快速判断哪个结果是正确的。
```

关键功能：

```text
1. 命中关键词高亮
2. 更好的 snippet 生成
3. 搜索结果按文档聚合
4. 同一文档多个命中段落折叠展示
5. 文件名、标题、正文分字段权重
6. 最近修改时间排序因子
7. 最近打开文档排序因子
8. 搜索历史
9. 收藏结果
10. 常用目录快捷入口
```

结果排序策略：

```text
最终得分 = 正文相关性 + 标题命中权重 + 文件名命中权重 + 最近使用权重 + 文档类型权重
```

UI 体验目标：

```text
1. 搜索结果列表清晰展示文档名、路径、命中段落和时间
2. 右侧预览展示完整 chunk 和上下文
3. 支持键盘上下选择结果
4. 支持 Enter 打开文件
5. 支持快捷键聚焦搜索框
```

明确不做：

```text
不做语义搜索
不做问答
不做自动总结
不做复杂标签系统
```

验收标准：

```text
用户输入关键词后，可以在 1 秒内看到结果，并能通过摘要和高亮快速判断是否命中目标。
```

阶段价值：

```text
这个阶段直接影响用户是否愿意持续使用 SeekMind。
```

## 4. 阶段 3：语义搜索

建议版本：

```text
0.4.x
```

阶段定位：

```text
从“关键词匹配”升级到“语义相似搜索”
```

核心目标：

```text
用户输入的搜索词不需要和文档原文严格一致，只要语义相近，也能找到相关段落。
```

关键功能：

```text
1. 本地 embedding 模型管理
2. 对 chunk 生成向量
3. 向量落盘存储
4. query embedding
5. 语义 TopK 检索
6. 全文结果和语义结果合并
7. 混合排序
8. 语义搜索开关
9. 模型下载和初始化状态
10. 向量索引重建
```

推荐技术路线：

```text
Embedding：fastembed
向量存储：LanceDB 或其他本地向量索引
搜索策略：Tantivy BM25 + Vector Similarity Hybrid Search
```

混合搜索流程：

```text
用户输入 query
  ↓
Tantivy 全文检索 topK
  ↓
Embedding 模型生成 query vector
  ↓
向量检索 topK
  ↓
去重合并
  ↓
重新打分
  ↓
返回文档和段落
```

明确不做：

```text
不做自动问答
不做长文总结
不做云端模型
不做多轮对话
不做 OCR
```

验收标准：

```text
1. 搜索“离线依赖方案”可以找到包含“dependency:go-offline、Maven 仓库、本地构建”等相关表达的段落。
2. 精确关键词仍然能稳定命中，不能因为语义搜索引入结果退化。
3. 用户可以关闭语义搜索，只使用全文搜索。
```

阶段风险：

```text
1. 模型体积
2. 首次索引耗时
3. CPU 推理速度
4. 内存占用
5. 中英文效果
6. 向量数据迁移
```

阶段价值：

```text
这个阶段是 SeekMind 从普通全文搜索工具形成差异化的关键。
```

## 5. 阶段 4：文档预览和定位增强

建议版本：

```text
0.5.x
```

阶段定位：

```text
让用户从“找到段落”进一步到“快速回到原文上下文”
```

核心目标：

```text
搜索结果不仅告诉用户命中了哪里，还能让用户快速查看上下文和原始文档位置。
```

关键功能：

```text
1. App 内段落上下文预览
2. 展示命中段落前后相邻 chunk
3. PDF 页码定位
4. DOCX 标题路径展示
5. Markdown 标题路径展示
6. HTML 标题路径展示
7. Quick Look 预览
8. 文件信息侧栏
9. 复制段落引用
10. 复制文件路径
```

引用格式示例：

```text
《Maven 离线构建方案.md》 / 三、离线依赖导出方案 / 第 5 段
```

明确不做：

```text
不做复杂文档编辑器
不做 PDF 坐标级文字高亮
不做 Word 内部精确高亮
不做批注编辑
```

验收标准：

```text
用户点击一个搜索结果后，可以看到命中段落、上下文、来源文件、页码或标题路径，并能快速打开原文件。
```

阶段价值：

```text
这个阶段提升搜索结果可信度，也为后续问答引用来源打基础。
```

## 6. 阶段 5：基于来源的问答

建议版本：

```text
0.6.x
```

阶段定位：

```text
从“搜索相关段落”升级到“基于本地文档回答问题”
```

核心目标：

```text
用户可以用自然语言提问，SeekMind 基于检索到的本地文档片段生成回答，并明确引用来源。
```

关键功能：

```text
1. 问答式搜索入口
2. 检索相关 chunk
3. 构造上下文
4. 调用本地或用户配置的模型
5. 生成答案
6. 每条答案附带来源段落
7. 来源可点击回到文档
8. 回答失败或证据不足时明确提示
9. 问答历史
10. 仅基于选定目录问答
```

回答原则：

```text
没有来源，不给确定答案。
来源不足，明确说明无法从当前文档判断。
答案必须能回溯到具体文档和段落。
```

明确不做：

```text
不做无来源闲聊
不做联网搜索
不默认上传用户文档
不做复杂 Agent 自动操作
不做团队共享知识库
```

验收标准：

```text
用户提问后，SeekMind 能给出简短回答，并列出 2 到 5 个来源段落。用户点击来源可以回到对应文档。
```

阶段风险：

```text
1. 本地模型质量和速度
2. 上下文截断
3. 幻觉控制
4. 来源引用准确性
5. 用户隐私和模型配置
```

阶段价值：

```text
这个阶段让 SeekMind 具备 AI 产品属性，但必须建立在前面稳定检索和来源定位的基础上。
```

## 7. 阶段 6：知识组织能力

建议版本：

```text
0.7.x
```

阶段定位：

```text
从搜索工具扩展为个人文档知识库
```

核心目标：

```text
帮助用户把常用结果、重要段落和主题沉淀下来，而不只是每次重新搜索。
```

关键功能：

```text
1. 收藏文档
2. 收藏段落
3. 标签
4. 笔记
5. 保存搜索
6. 主题集合
7. 文档关系推荐
8. 相似段落推荐
9. 最近查看
10. 导出引用列表
```

主题集合示例：

```text
Maven 离线构建
Tauri v2 权限配置
客户项目交付资料
合同风险条款
```

明确不做：

```text
不做完整笔记软件
不做 Obsidian 替代品
不做复杂双链编辑
不做多人协作
```

验收标准：

```text
用户可以把搜索到的重要段落加入一个主题集合，并在之后快速回顾这些来源。
```

阶段价值：

```text
这个阶段提升留存，让 SeekMind 从一次性搜索工具变成长期知识入口。
```

## 8. 阶段 7：OCR 和更多格式支持

建议版本：

```text
0.8.x
```

阶段定位：

```text
扩大可索引资料范围
```

核心目标：

```text
让扫描版 PDF、图片、PPT、Excel 等资料也能逐步进入 SeekMind 的搜索范围。
```

关键功能：

```text
1. 扫描版 PDF 检测
2. OCR 任务队列
3. OCR 结果保存
4. 图片文字识别
5. PPTX 文本提取
6. XLSX 文本提取
7. EPUB 支持
8. RTF 支持
9. 旧版 doc 转换策略
10. 格式支持状态说明
```

OCR 策略：

```text
默认关闭 OCR。
用户手动开启指定目录或指定文件。
OCR 任务低优先级后台执行。
必须显示耗时和磁盘占用预估。
```

明确不做：

```text
不追求版面完全还原
不做表格结构深度理解
不做图片内容语义理解
不默认扫描所有图片
```

验收标准：

```text
用户可以手动对扫描版 PDF 执行 OCR，完成后能通过文字搜索到对应页面。
```

阶段风险：

```text
1. OCR 准确率
2. 本地计算成本
3. 语言模型包体积
4. PDF 页面图像提取复杂度
5. 用户对处理时间的预期
```

阶段价值：

```text
这个阶段扩大 SeekMind 的适用范围，但不应早于语义搜索和预览定位能力。
```

## 11. 阶段推进总览

```text
0.1.x  MVP：全文搜索闭环
0.2.x  稳定性：增量索引、失败恢复、索引质量
0.3.x  搜索体验：高亮、排序、聚合、预览
0.4.x  语义搜索：embedding、向量索引、混合检索
0.5.x  定位增强：上下文、页码、标题路径、Quick Look
0.6.x  来源问答：基于检索结果生成答案
0.7.x  知识组织：收藏、标签、主题集合
0.8.x  格式扩展：OCR、PPTX、XLSX、EPUB
0.9.x  发布准备：性能、迁移、更新、日志、商业化
1.0.0  稳定发布：本地文档智能搜索产品
```

## 12. 优先级规则

当多个需求冲突时，按以下顺序决策：

```text
1. 搜索结果正确性优先于视觉效果
2. 索引稳定性优先于新格式支持
3. 来源可追溯优先于回答流畅度
4. 本地隐私优先于云端便利
5. 可解释失败优先于静默失败
6. 可控范围优先于功能完整
```

不建议提前做的能力：

```text
1. 团队协作
2. 云同步
3. 企业权限管理
4. 多端同步
5. 插件市场
6. 自动 Agent 工作流
7. 全量 OCR
8. 完整文档编辑器
```

这些能力会显著改变产品复杂度，不应在 1.0 之前作为主线目标。

## 13. 每阶段通用验收模板

每个阶段进入开发前，都应回答：

```text
1. 这个阶段解决哪个用户问题？
2. 这个问题是否高频？
3. 是否依赖前一阶段的稳定能力？
4. 有没有更小的实现方式？
5. 是否会破坏已有搜索体验？
6. 是否会增加不可控的本地资源消耗？
7. 失败时用户能否理解和恢复？
```

每个阶段完成后，都应验证：

```text
1. 搜索结果是否更准确？
2. 定位是否更清晰？
3. 索引是否更稳定？
4. 用户是否更容易理解结果？
5. 是否引入明显性能退化？
6. 是否有可观测日志和错误信息？
```

---

## 14. 最终建议

SeekMind 的正确推进方式不是快速堆 AI 功能，而是先把本地文档搜索这件事做到可靠。

推荐主线：

```text
全文搜索稳定
  ↓
搜索体验变好
  ↓
语义检索增强召回
  ↓
来源定位增强可信度
  ↓
基于来源生成答案
  ↓
知识组织提高留存
```

只要每个阶段都围绕“更快找到可信来源”这个目标推进，SeekMind 的产品方向就不会偏。
