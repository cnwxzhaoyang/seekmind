# DocMind 0.5.x 第一批开发顺序

本文档把 [18-0.5x文档预览与定位增强开发计划(2026-05-24).md](./18-0.5x文档预览与定位增强开发计划(2026-05-24).md) 再压缩成第一批可执行顺序。

这一批只做一件事：

```text
把“命中段落”变成“命中段落 + 上下文 + 来源信息 + 快速确认入口”。
```

不提前扩展到 `Quick Look`、页码全量体系化、复制引用格式标准化等后续能力。  
第一批的目标是先把“看懂这条结果到底来自哪里”做稳。

---

## 1. 第一批目标

```text
用户点击任意搜索结果后，能在右侧快速看到命中段落、前后上下文、文件基础信息，并能一键打开原文件确认。
```

第一批能力范围：

```text
1. 命中段落展示
2. 上下文预览联动
3. 前后相邻 chunk 展示
4. 文件信息侧栏
5. 打开原文件与视图切换的闭环
```

第一批不做：

```text
1. Quick Look
2. 复制引用
3. 复制路径
4. PDF 页码定位体系化
5. DOCX / MD / HTML 标题路径统一标准化
```

---

## 2. 第一批执行顺序

### 2.1 先补“当前命中 chunk + 前后上下文”

#### 目标

```text
右侧详情区先稳定展示当前命中的 chunk，再把前后相邻 chunk 作为上下文补上。
```

#### 需要做什么

```text
1. 结果选中后，详情区始终显示当前命中 chunk。
2. 同时展示前一段和后一段的文本摘要。
3. 上下文区不要只放固定文案，要能真正跟随 chunk 切换。
4. 命中段落要保持高亮。
```

#### 当前状态

```text
已有 snippet、高亮和部分上下文预览雏形，但展示方式还偏附属，不够稳定。
```

#### 验收什么

```text
1. 切换搜索结果后，右侧内容会实时变化。
2. 能看到当前 chunk 和相邻上下文。
3. 命中高亮不丢失。
```

---

### 2.2 再补“文件信息侧栏”

#### 目标

```text
把来源文件的基础信息稳定挂在结果详情里，帮助用户快速判断结果是否可信。
```

#### 需要做什么

```text
1. 展示文件名。
2. 展示绝对路径。
3. 展示文件格式。
4. 展示更新时间。
5. 展示 chunk 数或命中 chunk 数。
```

#### 当前状态

```text
当前已经有文件名、路径、扩展名和部分统计信息，但侧栏结构还不够统一。
```

#### 验收什么

```text
1. 文件基础信息在右侧固定区域稳定展示。
2. 用户不用回到列表页，也能确认这是哪一个文件。
```

---

### 2.3 然后补“结果详情信息块收口”

#### 目标

```text
让命中段落、上下文、来源元信息、排序解释形成一个稳定的右侧详情结构。
```

#### 需要做什么

```text
1. 把命中段落和上下文区分层展示。
2. 把来源元信息和排序解释分开。
3. 统一详情区的标题层级和间距。
4. 保持右侧详情在不同结果之间切换时视觉稳定。
```

#### 当前状态

```text
当前详情区已经有命中段落、上下文预览、排序理由，但层级还可以再收口。
```

#### 验收什么

```text
1. 右侧详情读起来像一条完整的“定位卡片”。
2. 用户能快速区分哪些是原文内容，哪些是结果解释。
```

---

### 2.4 最后补“打开原文件”闭环

#### 目标

```text
确保用户从搜索结果能一步回到原文件，形成最小可用闭环。
```

#### 需要做什么

```text
1. 结果详情里保留打开原文件入口。
2. 打开后保持当前结果上下文不丢失。
3. 如果后续还有预览动作，也要与打开原文件明确区分。
```

#### 当前状态

```text
打开原文件能力已存在，但第一批要把它放回到“上下文定位闭环”的中心位置。
```

#### 验收什么

```text
1. 用户看到结果后可以直接打开原文件。
2. 打开前后，定位信息仍然可回看。
```

---

## 3. 推荐实施拆分

### 3.1 后端任务

```text
1. 补齐搜索结果返回的上下文字段。
2. 确保当前 chunk 与前后 chunk 的顺序关系稳定。
3. 保持文件元信息返回一致。
4. 不新增新的定位存储模型，先复用现有 chunk / document 数据。
```

### 3.2 前端任务

```text
1. 统一右侧详情区结构。
2. 把命中段落、上下文、来源信息、打开文件入口分层。
3. 确保切换结果时详情刷新稳定。
4. 保留当前已有的高亮和聚合结果展示。
```

### 3.3 数据任务

```text
1. 确保 chunk 顺序、文件名、路径、更新时间等字段可稳定读取。
2. 如有必要，补齐前后 chunk 获取的最小查询能力。
```

---

## 4. 验收方式

第一批完成后，应该满足下面的验收：

```text
1. 用户在搜索页点击一个结果，右侧立即展示命中 chunk。
2. 同时能看到前后上下文。
3. 用户能清楚看到来源文件信息。
4. 用户能一键打开原文件。
5. 不需要依赖额外的 Quick Look 或复制动作，也能完成“快速确认结果是否正确”的任务。
```

---

## 5. 第一批完成后的状态

如果第一批完成，`0.5.x` 就已经具备一个稳定的最小闭环：

```text
搜索结果 -> 上下文预览 -> 来源确认 -> 打开原文件
```

这时第二批才继续做：

```text
1. PDF 页码定位
2. DOCX / Markdown / HTML 标题路径展示
3. Quick Look
4. 复制引用 / 复制路径
```
