# Python 解析异步任务化方案

## 背景
当前 docMind 的 Python 解析链路已经能够承担 `md/docx/html/txt/pdf` 的主要切片工作，但它仍然是通过同步调用进入 Rust command，再由 Rust 进程同步等待 Python sidecar 返回。

这会带来几个直接问题：

- 复杂文档解析时间不可控，容易触发超时
- UI 会把“等待中”误判成“失败”
- Python 成功回退 Rust 时，用户很难区分“真 Python 结果”和“兜底结果”
- 单文档重切、目录重索引、语义向量重建都容易卡在同步等待上

因此，下一步应把 Python 侧处理统一改成“任务制 + 事件驱动”，而不是继续在同步请求里硬等结果。

## 目标

1. 让耗时的 Python 解析不再阻塞 UI 和 Rust command。
2. 让“正在处理 / 已完成 / 失败 / 回退 Rust”成为可见状态。
3. 让单文档重切、目录重索引、语义向量重建都可排队执行。
4. 避免因为超时导致的误报失败。

## 适用范围

优先改造以下链路：

- 单文档重新切片
- 目录重索引
- 失败文件重试
- 语义向量重建
- PDF 文本提取
- DOCX 复杂结构解析

暂不优先改造：

- 简单文本文件的快速解析
- 纯 Rust fallback 的保底路径

## 核心原则

### 1. Rust 负责编排，Python 负责执行

Rust 不再同步等待 Python 返回最终结果，而是：

- 创建任务
- 写入任务状态
- 交给后台 worker 执行
- 通过事件/状态表反馈进度

Python sidecar 只负责：

- 接收任务输入
- 完成解析或 embedding
- 输出结构化结果

### 2. UI 只看状态，不看同步返回

前端不再依赖 command 是否“立刻返回成功”来判断任务结果，而是通过：

- 任务状态
- 进度事件
- 最终结果
- 回退警告

来展示真实状态。

### 3. 允许回退，但必须显式告知

如果 Python 解析失败并回退到 Rust，任务仍然可以完成，但前端必须明确提示：

- 本次由 Python 尝试处理
- 最终回退到 Rust
- 回退原因是什么

这样用户不会误以为当前文档确实由 Python 成功解析。

## 任务模型

建议统一使用 `job_id` 管理所有耗时任务。

### 任务状态

- `queued`
- `running`
- `paused`
- `done`
- `failed`
- `fallback_rust`

### 任务字段

- `job_id`
- `task_type`
- `target_path`
- `target_dir`
- `state`
- `progress`
- `current_stage`
- `current_file`
- `message`
- `warning`
- `error_code`
- `error_message`
- `created_at`
- `updated_at`

### 任务类型

- `refresh_document`
- `refresh_directory`
- `rebuild_semantic`
- `retry_failed_file`

## 通信方式

### 推荐方案

继续使用当前的 stdin/stdout JSON 协议，但把“同步返回结果”升级为“任务命令 + 异步轮询/事件”。

### 分层设计

#### Rust -> Python

Rust 通过 sidecar 命令发起任务：

```json
{
  "request_id": "uuid",
  "command": "parse_document_async",
  "path": "/path/to/file.docx",
  "options": {
    "include_chunks": true,
    "max_chunk_chars": 800
  }
}
```

#### Python -> Rust

Python 不再只返回最终文档，也可以返回：

```json
{
  "request_id": "uuid",
  "ok": true,
  "job_id": "job-uuid",
  "state": "queued",
  "message": "任务已进入队列"
}
```

#### 进度事件

Rust 后台 worker 在处理过程中通过事件推送：

```json
{
  "job_id": "job-uuid",
  "state": "running",
  "progress": 35,
  "current_stage": "parsing",
  "current_file": "xxx.docx",
  "message": "正在解析 DOCX"
}
```

#### 最终结果

```json
{
  "job_id": "job-uuid",
  "state": "done",
  "warning": "Python 解析失败，已回退 Rust：xxx",
  "result": {
    "documents": 1,
    "chunks": 25
  }
}
```

## 前端表现

### 文档切片页

在“文档列表”里，每个文档重切状态建议显示：

- `切片中`
- `已排队`
- `已完成`
- `回退 Rust`
- `失败`

### 顶部提示

页面顶部继续显示当前解析器：

- `Python 解析`
- `Rust 回退`

但需要补一条说明：

> 当前解析器显示为 Python，只代表默认走 Python 链路；如果 Python 任务异常，最终仍可能回退到 Rust，并在完成后显示警告。

### 结果提示

当任务回退时，必须在结果卡片或详情区提示：

- 本次切片已从 Python 回退到 Rust
- 原因：xxx

## 数据库建议

建议补一张任务表，至少包含：

- `job_id`
- `task_type`
- `state`
- `target_path`
- `target_dir`
- `progress`
- `current_stage`
- `current_file`
- `message`
- `warning`
- `error_code`
- `error_message`
- `created_at`
- `updated_at`

如果不想新建表，也可以先复用 `current_task`，但后续会受限于：

- 单任务模型不够通用
- 无法同时管理多个任务
- 任务历史难以保留

因此建议最终还是拆成独立任务表。

## 实施顺序

### 第一阶段

1. 单文档重切改成异步任务。
2. 保留当前 Python/Rust 结果提示。
3. 前端显示任务状态与回退警告。

### 第二阶段

1. 目录重索引改成任务队列。
2. 失败文件重试纳入同一任务系统。
3. 语义向量重建接入事件驱动。

### 第三阶段

1. Python sidecar 常驻化。
2. 任务持久化和断点续跑统一。
3. 任务历史、失败统计、告警分级完善。

## 验收标准

1. 复杂 DOCX 解析不再因为同步超时直接误报失败。
2. 单文档重切时，UI 能明确看到任务状态。
3. Python 解析异常回退到 Rust 时，用户能看到警告。
4. 同一时间可以排队多个文档重切任务，不会互相覆盖。
5. 目录重索引和语义重建不会因为等待 Python 返回而卡死 UI。

## 风险

- 任务系统复杂度会上升，需要统一状态模型。
- 事件驱动后调试成本会增加，需要保留足够的日志。
- 若 Python sidecar 本身仍频繁崩溃，异步只能降低误报，不能代替修复解析器。

## 结论

Python 侧处理应该尽快从“同步等待返回”切到“异步任务 + 状态事件”。

对于 docMind 来说，真正重要的是：

- 任务能持续推进
- UI 能看到真实状态
- Python 失败时有明确回退提示
- 用户不会再把回退结果误认为 Python 成功结果
