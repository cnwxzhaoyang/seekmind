# DocMind 0.4.x 语义搜索实施拆解

本文档用于把 `0.4.x` 的语义搜索阶段拆成可执行方案。当前 `0.3.x` 已完成关键词检索体验增强，因此 `0.4.x` 的核心不是替换全文检索，而是在不退化原有精确命中的前提下，引入本地语义相似检索能力。

## 1. 阶段目标

`0.4.x` 的目标是：

```text
用户输入不完全匹配原文的描述时，DocMind 也能找到语义相关的文档段落，并且保留关键词检索的稳定性。
```

核心原则：

```text
1. 关键词检索仍然是基础。
2. 语义检索是增强，不是替代。
3. 结果必须可解释、可调试、可关闭。
4. 所有语义能力都必须本地优先。
```

## 2. 当前基线

在进入 `0.4.x` 之前，DocMind 已具备：

```text
1. 本地目录索引
2. SQLite 元数据
3. Tantivy 全文检索
4. Python 文档解析与切片
5. 搜索高亮、摘要和结果聚合
6. 搜索历史、最近文档和收藏
7. 增量索引、暂停 / 恢复、失败重试
```

这意味着 `0.4.x` 可以直接在现有 chunk 体系上增加语义层，不需要重做文档解析和段落切片。

## 3. 阶段范围

### 3.1 必做项

```text
1. 本地 embedding 模型管理
2. 文档 chunk embedding 生成
3. 向量索引存储
4. query embedding 生成
5. 混合检索（关键词 + 语义）
6. 语义结果与全文结果融合排序
7. 语义搜索开关
8. 语义调试面板
9. embedding 重建与失效检测
10. 本地模型状态展示
```

### 3.2 可选项

```text
1. 语义相似度过滤阈值
2. 多 query 改写
3. 轻量 rerank
4. 收藏语义结果
5. 语义搜索历史
```

### 3.3 明确不做

```text
1. 本地问答
2. 自动总结
3. 联网检索
4. 多轮对话
5. 复杂 Agent 编排
6. OCR
7. 团队协作
```

## 4. 架构建议

### 4.1 推荐分层

```text
前端 Vue：展示搜索开关、结果、调试信息、模型状态
Rust 后端：任务编排、SQLite、Tantivy、向量索引、混合检索
Python sidecar：继续负责文档解析，不承担 embedding 主流程
本地 embedding 引擎：可以是 Rust 原生方案，也可以是独立本地进程方案
```

### 4.2 推荐策略

优先采用以下思路：

```text
1. 先把 embedding 生成和缓存做通。
2. 再把向量检索接到搜索链路中。
3. 最后再做融合排序和阈值调参。
```

### 4.3 技术选择原则

```text
1. 必须离线可用。
2. 必须支持本地持久化。
3. 必须能和现有 chunk id 对齐。
4. 必须能在 UI 上解释“为何命中”。
5. 必须允许关闭语义搜索，回退到纯全文检索。
```

### 4.4 本地 embedding 模型选择

语义搜索第一版建议走轻量化模型路线，优先选择可本地离线运行、CPU 可跑、向量维度较小的模型。

推荐优先级：

```text
1. 中文文档为主：BAAI/bge-small-zh-v1.5
2. 英文技术文档为主：BAAI/bge-small-en-v1.5
3. 轻量英文入门：sentence-transformers/all-MiniLM-L6-v2
```

选择原则：

```text
1. 第一版优先 384 维向量，控制存储和索引体积。
2. CPU 可跑优先，不依赖 PyTorch。
3. 先用小模型跑通语义检索链路，再考虑更大模型。
4. 中英混合场景可先用中文模型，再根据真实数据调整。
```

实现建议：

```text
1. 优先使用 FastEmbed 作为本地 embedding 生成层。
2. embedding 生成可以放在 Python sidecar 或独立本地 worker 中。
3. Rust 侧只负责任务编排、缓存、持久化和检索融合。
```

不建议第一版直接采用：

```text
1. 大型 embedding 模型
2. 768 维以上模型作为默认方案
3. 云端 embedding API
4. 依赖 PyTorch 的重型方案
```

向量存储粗估：

```text
384 维 float32 向量约 1.5 KB / chunk
10 万 chunk 约 150 MB 裸向量数据
```

这意味着 docMind 的第一版语义搜索应当尽量控制 chunk 粒度和向量维度，避免索引体积过快膨胀。

## 5. 数据模型拆解

`0.4.x` 建议增加或明确以下数据层：

### 5.1 模型表

建议表：`embedding_models`

建议字段：

```text
id
name
provider
model_path
dimension
enabled
created_at
updated_at
```

用途：

```text
记录当前可用的本地 embedding 模型，以及用户选择的默认模型。
```

### 5.2 chunk embedding 表

建议表：`chunk_embeddings`

建议字段：

```text
chunk_id
document_id
model_id
vector_blob
dimension
text_hash
status
created_at
updated_at
```

用途：

```text
记录每个 chunk 的语义向量，支持按 chunk 失效重建。
```

### 5.3 向量索引元数据表

建议表：`vector_index_meta`

建议字段：

```text
id
model_id
chunk_count
last_indexed_at
last_error
status
```

用途：

```text
记录向量索引是否可用、是否需要重建、是否和当前模型一致。
```

### 5.4 query 语义调试表

可选表：`semantic_search_log`

建议字段：

```text
query
normalized_query
model_id
top_k
hit_count
semantic_hit_count
keyword_hit_count
created_at
```

用途：

```text
方便后期调权和排查命中退化问题。
```

## 6. 功能拆解

### 6.1 本地 embedding 模型管理

目标：

```text
让用户明确知道当前语义搜索用的是哪个模型，并且能看到模型是否可用。
```

要做的事：

```text
1. 模型安装状态展示。
2. 默认模型选择。
3. 模型路径与维度展示。
4. 模型可用性检查。
5. 模型切换后触发向量重建提醒。
```

验收标准：

```text
用户能在设置页看到当前 embedding 模型是否可用，并能切换默认模型。
```

---

### 6.2 chunk embedding 生成

目标：

```text
为每个文档 chunk 生成稳定的语义向量，并与 chunk id 一一对应。
```

要做的事：

```text
1. 从已有 chunk 表读取待编码文本。
2. 对 chunk 内容做 embedding。
3. 按 document + chunk 维度缓存。
4. 记录 text_hash，避免重复计算。
5. 支持失败重试和单 chunk 重建。
```

验收标准：

```text
相同内容不会重复生成 embedding；改动 chunk 内容后能重新生成。
```

---

### 6.3 向量索引存储

目标：

```text
让语义向量可检索、可更新、可删除。
```

要做的事：

```text
1. 选择向量索引存储方案。
2. 提供 add / upsert / delete / search 接口。
3. 和 chunk id 绑定。
4. 目录删除、文档删除时同步删除向量。
5. 支持整库重建。
```

验收标准：

```text
删除文档后，向量索引里不会残留旧 chunk。
```

---

### 6.4 query embedding 生成

目标：

```text
用户查询时即时生成 query 向量，用于语义召回。
```

要做的事：

```text
1. 复用同一 embedding 模型。
2. query 先归一化，再编码。
3. query embedding 与 chunk embedding 采用同一维度。
4. 支持 query 失败回退到全文检索。
```

验收标准：

```text
查询失败时不会阻断全文搜索，语义层只是加分项。
```

---

### 6.5 混合检索

目标：

```text
在关键词检索基础上，叠加语义召回，形成混合排序。
```

推荐流程：

```text
1. 先跑全文检索，得到 keyword hits。
2. 再跑向量检索，得到 semantic hits。
3. 合并两个结果集。
4. 按统一规则重新排序。
5. 输出最终前 N 个结果。
```

建议融合策略：

```text
最终分数 = 关键词相关性权重 + 语义相似度权重 + 标题 / 文件名命中权重 + 最近使用权重
```

关键原则：

```text
1. 关键词结果优先保底。
2. 语义结果只在足够相关时进入前排。
3. 不让语义检索压掉明确命中的精确结果。
```

验收标准：

```text
用户输入“离线依赖导出”之类的近义词，也能命中“dependency:go-offline”“本地仓库”“Maven 离线构建”等相关段落。
```

---

### 6.6 语义搜索开关

目标：

```text
让用户可以明确决定是否启用语义搜索。
```

要做的事：

```text
1. 设置页增加语义搜索开关。
2. 支持语义权重滑块或简单强度档位。
3. 支持只用全文检索。
4. 支持混合检索。
```

建议模式：

```text
1. 仅全文
2. 混合检索（默认）
3. 仅语义（仅调试，不建议默认）
```

验收标准：

```text
关闭语义搜索后，搜索行为恢复到纯关键词模式，结果不退化。
```

---

### 6.7 语义调试面板

目标：

```text
让用户和开发者能看懂语义层到底发生了什么。
```

建议显示内容：

```text
1. 当前模型名称
2. 模型维度
3. query 是否成功编码
4. semantic hits 数量
5. keyword hits 数量
6. 最终融合后的结果数
7. 最高相似度结果
8. 语义检索是否启用
```

验收标准：

```text
当结果不符合预期时，可以判断是 query 编码、向量召回，还是融合排序的问题。
```

## 7. 页面拆分建议

### 7.1 设置页

增加区域：

```text
1. embedding 模型状态
2. 语义搜索开关
3. 语义权重或模式选择
4. 向量重建按钮
```

### 7.2 搜索页

增加区域：

```text
1. 语义结果标记
2. 语义调试卡片
3. 命中来源说明
4. 混合检索开关状态
```

### 7.3 索引状态页

增加区域：

```text
1. 向量索引状态
2. embedding 重建进度
3. 模型可用性
4. 语义索引失败项
```

## 8. 实施顺序

建议按以下顺序推进：

```text
M1：模型管理与状态展示
M2：chunk embedding 生成与缓存
M3：向量索引存储
M4：query embedding 与语义召回
M5：混合检索与排序融合
M6：语义开关与调试面板
```

## 9. 验收清单

### 9.1 功能验收

```text
1. 用户可开启 / 关闭语义搜索。
2. 用户查询近义表达时，能命中相关文档段落。
3. 关闭语义搜索时，结果回到纯全文模式。
4. 文档删除后，向量索引同步清理。
```

### 9.2 质量验收

```text
1. 精确关键词命中不退化。
2. 语义结果不应压过明显的关键词命中。
3. 调试面板可解释命中来源。
4. 模型异常时能自动回退到全文检索。
```

### 9.3 工程验收

```text
1. chunk id 与向量索引一一对应。
2. 向量重建可重复执行。
3. 模型切换后能提示是否需要重建。
4. 语义层数据可持久化。
```

## 10. 风险与控制

### 10.1 风险：精确检索退化

控制：

```text
关键词检索始终是第一层，语义召回只能加分，不能替代。
```

### 10.2 风险：模型体积和性能

控制：

```text
优先小体积本地模型，控制 chunk 大小，控制 top_k。
```

### 10.3 风险：向量数据与文档内容不同步

控制：

```text
用 text_hash 和 model_id 做失效判断，改文档就重建向量。
```

### 10.4 风险：结果不可解释

控制：

```text
调试面板必须展示 keyword / semantic / fused 三层信息。
```

## 11. 阶段结束标准

`0.4.x` 完成时，应满足：

```text
1. 用户可以用近义表达找到文档。
2. 关键词结果不会退化。
3. 语义搜索可以随时关闭。
4. 模型、向量和融合排序都能调试。
5. 本地离线使用体验稳定。
```

达到这个标准后，才适合进入 `0.5.x` 的预览和定位增强阶段。
