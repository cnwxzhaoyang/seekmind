# 1. 有几种集成方式？

主要有 3 种。

## 方案一：Rust 直接嵌入 Python，使用 PyO3

Rust 通过 PyO3 调用 Python 模块。PyO3 支持在 Rust 二进制中嵌入 Python，也支持把 Rust 写成 Python 扩展模块。[GitHub+1](https://github.com/PyO3/pyo3?utm_source=chatgpt.com)

调用形式大概是：

```Rust
use pyo3::prelude::*;

fn parse_with_python(path: &str) -> PyResult<String> {
    Python::with_gil(|py| {
        let parser = PyModule::import(py, "docmind_parser")?;
        let result = parser
            .getattr("parse_document")?
            .call1((path,))?
            .extract::<String>()?;
        Ok(result)
    })
}
```

Python 侧：

```Python
# docmind_parser.py

def parse_document(path: str) -> str:
    # 调用 pymupdf / unstructured / docling / python-docx 等
    return "extracted text"
```

优点：

```
Rust 调用直接
接口统一
不用启动额外进程
```

缺点：

```
打包复杂
Python 环境、依赖、动态库处理麻烦
App Store 沙盒下风险更高
Python GIL 会影响并发
一旦 Python 解析崩溃，可能影响主进程稳定性
```

这个方案适合技术验证，但我不建议作为 macOS 上架版的第一选择。

---

## 方案二：Python 做 Sidecar 独立进程，Rust 通过 stdin/stdout 调用

这是我最推荐的方案。

架构是：

```
Tauri / Rust 主进程
    ↓ JSON-RPC / stdin stdout
Python Parser Worker
    ↓
PyMuPDF / python-docx / docling / unstructured / beautifulsoup
    ↓
返回统一 JSON
```

Tauri v2 支持把外部二进制作为 sidecar 打包进应用，官方文档说明可以通过 `externalBin` 把外部二进制一起 bundle。[Tauri](https://v2.tauri.app/develop/sidecar/?utm_source=chatgpt.com)

你可以把 Python 解析服务用 PyInstaller / Nuitka 打包成一个可执行文件，例如：

```
docmind-parser-aarch64-apple-darwin
docmind-parser-x86_64-apple-darwin
```

Rust 启动它：

```Rust
use std::process::{Command, Stdio};

fn parse_document(path: &str) -> anyhow::Result<String> {
    let output = Command::new("./docmind-parser")
        .arg("parse")
        .arg(path)
        .output()?;

    if !output.status.success() {
        anyhow::bail!("parser failed: {}", String::from_utf8_lossy(&output.stderr));
    }

    Ok(String::from_utf8(output.stdout)?)
}
```

Python 输出统一 JSON：

```JSON
{
  "title": "Maven 离线仓库方案",
  "file_type": "pdf",
  "chunks": [
    {
      "heading": "三、离线依赖导出方案",
      "page_no": 4,
      "text": "通过 dependency:go-offline 拉取 parent POM、BOM、plugin 依赖...",
      "order": 1
    }
  ]
}
```

优点：

```
主进程稳定，Python 崩了不影响 App
可以给解析任务设置超时
Python 依赖独立打包
便于后续替换解析引擎
更适合 Tauri 桌面应用
后续也可以单独升级 parser
```

缺点：

```
首次启动 Python worker 有开销
打包体积会变大
需要处理进程通信和任务队列
macOS 签名、公证、沙盒权限要额外注意
```

对 docMind 来说，这个方案最好。

## 方案三：Python 做本地 HTTP 服务

Rust 启动一个本地 FastAPI 服务：

```
Rust App
  → http://127.0.0.1:xxxxx/parse
  → Python FastAPI Parser Service
```

优点：

```
开发方便
调试方便
接口清晰
后续可扩展为远程解析服务
```

缺点：

```
桌面 App 里启动本地端口不够优雅
可能被防火墙/安全软件拦截
端口冲突
App Store 审核解释成本更高
```

这个适合开发期，不建议第一版正式发布用。

# 2. 我推荐 docMind 的实际架构

建议你把解析层设计成“多后端 Parser Engine”。

```
docMind
├─ Rust 主应用
│  ├─ scanner       文件扫描
│  ├─ indexer       索引调度
│  ├─ storage       SQLite
│  ├─ fulltext      Tantivy
│  ├─ search        搜索服务
│  └─ parser_client 调用 Python sidecar
│
└─ Python Parser Sidecar
   ├─ parse_pdf
   ├─ parse_docx
   ├─ parse_html
   ├─ parse_markdown
   ├─ parse_txt
   └─ output JSON
```

整体流程：

```
文件路径
  ↓
Rust 判断文件是否需要索引
  ↓
Rust 调用 Python Parser Worker
  ↓
Python 提取结构化文本
  ↓
Python 返回 ParsedDocument JSON
  ↓
Rust 校验 JSON
  ↓
Rust 保存 chunks
  ↓
Rust 写入 Tantivy
  ↓
前端搜索展示
```

# 3. Python 侧可以用哪些库？

第一阶段建议这样选：

| 格式 | Python 库 | 用途 |
| --- | --- | --- |
| PDF | `PyMuPDF` / `pypdf` | 提取文本、页码、文本块 |
| DOCX | `python-docx` | 提取段落、标题、表格 |
| HTML | `beautifulsoup4` / `readability-lxml` | 提取正文 |
| Markdown | `markdown-it-py` / 直接文本处理 | 按标题切块 |
| TXT | Python 标准库 | 读取文本 |
| OCR 后续 | `pytesseract` / `paddleocr` | 扫描版 PDF |
| 高级文档解析 | `docling` / `unstructured` | 复杂 PDF、表格、版面结构 |

其中 `Docling` 是一个 Python 包，也提供 CLI，目标就是把 PDF、DOCX、PPTX、HTML、图像等文档转换成统一结构化表示；论文介绍它使用版面分析、表格结构识别等模型，并且可以作为 Python API 或 CLI 使用。[arXiv](https://arxiv.org/abs/2501.17887?utm_source=chatgpt.com)

但第一阶段不要一上来就引入 `docling/unstructured`，因为依赖重、模型大、打包复杂。建议分层：

```
MVP：
  PyMuPDF + python-docx + BeautifulSoup

增强版：
  docling / unstructured

高级版：
  OCR + 表格结构识别 + 版面分析
```

# 4. 推荐通信协议：JSON Lines

不要每解析一个文件就启动一次 Python。那样太慢。

建议 Rust 启动一个长期运行的 Python worker，然后通过 stdin/stdout 用 JSON Lines 通信。

Rust 发：

```JSON
{"id":"task-001","type":"parse","path":"/Users/zhaoyang/Documents/a.pdf"}
```

Python 回：

```JSON
{"id":"task-001","ok":true,"data":{"title":"a.pdf","file_type":"pdf","chunks":[{"heading":null,"page_no":1,"text":"...","order":0}]}}
```

失败时：

```JSON
{"id":"task-001","ok":false,"error":{"code":"UNSUPPORTED_FILE","message":"unsupported file type: .doc"}}
```

这样有几个好处：

```
可以复用 Python 进程
可以做任务队列
可以做超时控制
可以做错误隔离
可以逐步返回进度
```

# 5. Python Worker 示例

## parser_worker.py

```Python
import sys
import json
import traceback
from pathlib import Path

def parse_txt(path: Path):
    text = path.read_text(encoding="utf-8", errors="ignore")
    paragraphs = [p.strip() for p in text.split("\n\n") if p.strip()]
    return {
        "title": path.name,
        "file_type": "txt",
        "chunks": [
            {
                "heading": None,
                "page_no": None,
                "text": p,
                "order": i
            }
            for i, p in enumerate(paragraphs)
        ]
    }

def parse_document(path_str: str):
    path = Path(path_str)
    ext = path.suffix.lower()

    if ext == ".txt":
        return parse_txt(path)

    if ext == ".md":
        return parse_txt(path)

    raise ValueError(f"unsupported file type: {ext}")

def handle_request(req):
    task_id = req.get("id")
    try:
        if req.get("type") != "parse":
            raise ValueError("unsupported request type")

        data = parse_document(req["path"])

        return {
            "id": task_id,
            "ok": True,
            "data": data
        }
    except Exception as e:
        return {
            "id": task_id,
            "ok": False,
            "error": {
                "code": type(e).__name__,
                "message": str(e),
                "trace": traceback.format_exc()
            }
        }

def main():
    for line in sys.stdin:
        line = line.strip()
        if not line:
            continue

        req = json.loads(line)
        resp = handle_request(req)

        print(json.dumps(resp, ensure_ascii=False), flush=True)

if __name__ == "__main__":
    main()
```

# 6. Rust 调用 Worker 示例

```Rust
use serde::{Deserialize, Serialize};
use std::io::{BufRead, BufReader, Write};
use std::process::{Child, ChildStdin, ChildStdout, Command, Stdio};
use uuid::Uuid;

#[derive(Debug, Serialize)]
struct ParseRequest {
    id: String,
    #[serde(rename = "type")]
    request_type: String,
    path: String,
}

#[derive(Debug, Deserialize)]
struct ParseResponse {
    id: String,
    ok: bool,
    data: Option<ParsedDocument>,
    error: Option<ParserError>,
}

#[derive(Debug, Deserialize)]
struct ParsedDocument {
    title: Option<String>,
    file_type: String,
    chunks: Vec<DocumentChunk>,
}

#[derive(Debug, Deserialize)]
struct DocumentChunk {
    heading: Option<String>,
    page_no: Option<u32>,
    text: String,
    order: usize,
}

#[derive(Debug, Deserialize)]
struct ParserError {
    code: String,
    message: String,
    trace: Option<String>,
}

pub struct PythonParserWorker {
    child: Child,
    stdin: ChildStdin,
    stdout: BufReader<ChildStdout>,
}

impl PythonParserWorker {
    pub fn start(binary_path: &str) -> anyhow::Result<Self> {
        let mut child = Command::new(binary_path)
            .stdin(Stdio::piped())
            .stdout(Stdio::piped())
            .stderr(Stdio::piped())
            .spawn()?;

        let stdin = child.stdin.take().unwrap();
        let stdout = BufReader::new(child.stdout.take().unwrap());

        Ok(Self {
            child,
            stdin,
            stdout,
        })
    }

    pub fn parse(&mut self, path: &str) -> anyhow::Result<ParsedDocument> {
        let id = Uuid::new_v4().to_string();

        let req = ParseRequest {
            id: id.clone(),
            request_type: "parse".to_string(),
            path: path.to_string(),
        };

        let line = serde_json::to_string(&req)?;
        writeln!(self.stdin, "{}", line)?;
        self.stdin.flush()?;

        let mut resp_line = String::new();
        self.stdout.read_line(&mut resp_line)?;

        let resp: ParseResponse = serde_json::from_str(&resp_line)?;

        if !resp.ok {
            let err = resp.error
                .map(|e| e.message)
                .unwrap_or_else(|| "unknown parser error".to_string());
            anyhow::bail!(err);
        }

        resp.data.ok_or_else(|| anyhow::anyhow!("empty parser response"))
    }
}
```

实际生产版还需要：

```
超时控制
stderr 日志读取
worker 崩溃自动重启
任务并发队列
大文件限制
返回 JSON 大小限制
```

# 8. 对 docMind 的最终建议

你的架构应该改成这样：

```
第一阶段 MVP：
  Rust 原生解析 txt / md
  Python sidecar 解析 pdf / docx / html
  Rust 负责 chunk 二次清洗、SQLite、Tantivy

第二阶段：
  Python sidecar 引入 PyMuPDF 更精细提取 PDF 页码和文本块
  引入 python-docx 解析标题、表格
  引入 BeautifulSoup/readability 提取 HTML 正文

第三阶段：
  引入 docling / unstructured
  支持复杂 PDF 表格
  支持扫描版 OCR
  支持版面结构识别

第四阶段：
  对解析结果做质量评分
  同一文件支持多种解析器 fallback
```

核心原则：

> **Python 只负责“把复杂文档变成结构化文本 JSON”，Rust 负责“稳定、快速、可控地索引和搜索”。**

这样可以同时拿到：

```
Python 生态的解析准确度
Rust/Tauri 的桌面体验和性能
进程隔离带来的稳定性
后续解析引擎可替换能力
```

所以，docMind 的 MVP 我建议把原来的 `parser` 模块改成：

```
parser
├─ native
│  ├─ txt_parser.rs
│  └─ markdown_parser.rs
├─ python
│  ├─ python_worker.rs
│  ├─ parser_client.rs
│  └─ parser_protocol.rs
└─ parser_service.rs
```

这会比纯 Rust 解析路线更适合你这个产品。
