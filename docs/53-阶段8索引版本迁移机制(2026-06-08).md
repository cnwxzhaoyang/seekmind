# 阶段 8 索引版本迁移机制

本文档记录阶段 8 中“索引版本迁移机制”的实现结果，作为 [《阶段 8 对照表》](./47-阶段8对照表(2026-06-08).md) 的补充说明。

## 1. 已实现内容

### 1.1 全文索引版本化

- Tantivy 全文索引目录增加了 `schema-version.txt` 标记文件。
- 启动时会检查标记版本是否与当前 `CURRENT_FULLTEXT_INDEX_SCHEMA_VERSION` 一致。
- 版本不一致且存在旧标记时，会删除旧全文索引目录并重建。
- 重建完成后会重新写入版本标记，避免下次启动再次误判。

### 1.2 语义向量索引版本化

- `vector_index_meta` 表新增 `schema_version` 字段。
- 启动时会检查向量索引版本是否与当前 `CURRENT_VECTOR_INDEX_SCHEMA_VERSION` 一致。
- 版本不一致时，会清空 `chunk_embeddings` 并把 `vector_index_meta` 标记为 `needs_rebuild`。
- 语义模型状态接口会把版本失配后的重建需求同步给前端。

### 1.3 统一迁移日志

- 启动时会打印全文索引目录、迁移方向和标记写入日志。
- 向量索引版本不兼容时会打印迁移日志，便于排查首次启动或升级后的重建过程。

## 2. 设计取舍

- 当前实现采用“版本号 + 自动重建”的策略，而不是复杂的多级结构迁移。
- 这对本地知识库场景更直接，也更容易保证升级安全性。
- 当未来新增全文字段、向量维度或 chunk 结构时，可以继续沿用同一套版本判断与重建入口。

## 3. 验证结果

- `cargo check --manifest-path src-tauri/Cargo.toml` 通过
- 索引版本迁移逻辑可以在启动时自动执行
- 旧版本索引在版本不兼容时会走重建路径，而不会静默沿用

## 4. 结论

- 阶段 8 的“索引版本迁移机制”已完成。
- 当前阶段 8 的缺口重心转向日志导出、首次引导、自动更新和商业化骨架。
