# SeekMind 0.4.x 第二批开发顺序

本文档承接 [0.4.x 第一批开发顺序](./8-0.4x第一批开发顺序(2026-05-23).md)，专门细化 `0.4.x` 的第二批推进路径。

这一批不再做语义底座本身，而是把已经有雏形、但还不够完整的能力，补成可上线的混合检索闭环：

```text
关键词检索 + 语义召回 + 开关控制 + 调试解释
```

---

## 1. 第二批目标

第二批只解决一个问题：

```text
用户能明确选择是否启用语义能力，搜索结果能混合排序，而且一眼能看懂“为什么命中”。
```

对应能力：

```text
1. 语义搜索开关
2. 关键词 + 语义混合检索
3. 结果融合排序
4. 语义命中解释
5. 调试面板完善
6. 失败回退与可见性
```

这一批不再优先做向量底座搭建，因为第一批已经负责：

```text
1. 模型状态可见
2. chunk embedding 可生成
3. 向量可持久化
4. query 可调试
5. 语义层可观测
```

---

## 2. 当前前提

进入第二批前，SeekMind 应该已经具备以下基础，但不要求全部完美：

```text
1. 本地 embedding 模型状态可查
2. chunk embedding 已经能生成和重建
3. 向量数据已落库
4. query embedding 已有调试入口
5. 语义调试面板已存在雏形
```

但这些能力里，有些可能只是“能用”，还不够“能解释”或“能稳定联动”，第二批要做的就是把这些补齐。

---

## 3. 第二批范围

### 3.1 必做项

```text
1. 搜索入口的语义开关
2. 混合检索主链路
3. 结果融合排序
4. 命中来源解释
5. 调试面板完善
6. 语义失效时的全文回退
7. 开关状态持久化
```

### 3.2 已有但不够完善的部分

这一类功能第二批要优先“补强”，而不是重新造轮子：

```text
1. 语义调试面板已存在，但还不够解释结果排序。
2. 模型状态已存在，但还没有和搜索行为强绑定。
3. query embedding 已能调试，但还没有进入主搜索链路。
4. 日志里能看到事件，但还不足以说明最终为什么命中。
5. 语义开关可能已在设置页有雏形，但还没有完整控制搜索行为。
```

### 3.3 明确不做

```text
1. rerank
2. query 改写
3. 多模型自动切换
4. 仅语义模式默认开启
5. 云端 embedding API
6. 本地问答
7. 复杂对话式检索
```

---

## 4. 实施顺序

### 4.1 任务 A：语义开关先闭环

目标：

```text
先让用户明确决定语义检索是否参与搜索，避免语义能力默默改变结果。
```

后端任务：

```text
1. 读取并持久化语义搜索开关。
2. 暴露当前启用模式：仅全文 / 混合检索。
3. 在检索入口根据开关决定是否触发语义召回。
```

前端任务：

```text
1. 设置页增加语义开关。
2. 显示当前模式。
3. 保存后立即生效。
```

验收标准：

```text
关闭语义开关后，搜索完全回到纯全文模式。
```

---

### 4.2 任务 B：把 query embedding 接进主搜索

目标：

```text
让搜索请求真正生成 query 向量，而不是只在调试面板里展示。
```

后端任务：

```text
1. 搜索时生成 query embedding。
2. 语义召回只作为补充召回，不替代全文命中。
3. query 失败时直接回退到全文检索。
```

前端任务：

```text
1. 搜索页显示当前是否启用语义召回。
2. 调试面板展示 query 是否成功编码。
3. 显示 query 维度和基础状态。
```

验收标准：

```text
同一个查询在开启语义时和关闭语义时，结果集有差异但不退化。
```

---

### 4.3 任务 C：混合检索与排序融合

目标：

```text
让关键词结果和语义结果合并后，排序稳定且可解释。
```

后端任务：

```text
1. 先取全文召回结果。
2. 再取语义召回结果。
3. 对同一 chunk 做去重合并。
4. 根据统一分数公式排序。
5. 保留命中来源信息。
```

建议融合规则：

```text
1. 关键词命中作为基础分。
2. 语义相似度作为加分项。
3. 标题 / 文件名命中给予额外加权。
4. 明确关键词命中不应被弱语义结果压过。
```

前端任务：

```text
1. 搜索结果显示“来自全文 / 来自语义 / 混合命中”。
2. 保留高亮和摘要。
3. 如果结果排序出现明显变化，要能在调试面板中看到原因。
```

验收标准：

```text
输入近义表达时能召回相关内容，且精确关键词命中仍然优先稳定显示。
```

---

### 4.4 任务 D：命中解释补全

目标：

```text
让用户能看懂“为什么这条结果排在前面”。
```

后端任务：

```text
1. 返回 matched_field / match_origin / semantic_score / keyword_score。
2. 返回标题命中、文件名命中、正文命中的区分。
3. 记录最终排序的简单解释字段。
```

前端任务：

```text
1. 结果卡片展示命中来源。
2. 调试面板展示融合得分。
3. 标明是全文命中、语义命中，还是两者叠加。
```

验收标准：

```text
用户可以看出某条结果是因为标题命中、正文命中，还是语义相似被拉上来的。
```

---

### 4.5 任务 E：调试面板完善

目标：

```text
把调试面板从“状态展示”补成“排错工具”。
```

后端任务：

```text
1. 返回当前模型、query 编码状态、全文命中数、语义命中数、最终结果数。
2. 返回 top hit 的命中来源和分数构成。
3. 对失败状态给出可读错误，而不是只有空结果。
```

前端任务：

```text
1. 调试面板区分全文和语义两条链路。
2. 显示当前开关模式。
3. 显示结果融合前后的变化。
4. 显示失败回退原因。
```

验收标准：

```text
当结果不符合预期时，能判断是 query、召回、排序还是回退导致的。
```

---

### 4.6 任务 F：回退策略收口

目标：

```text
语义能力出问题时，全文搜索仍然稳定可用，且用户能知道发生了回退。
```

后端任务：

```text
1. query embedding 失败时只降级语义层。
2. 向量召回失败时保留全文结果。
3. 模型不可用时明确标记回退。
```

前端任务：

```text
1. 设置页或搜索页显示语义回退状态。
2. 调试面板显示失败原因。
3. 不把“有结果”误当成“语义链路正常”。
```

验收标准：

```text
语义层异常时，搜索仍可用，并且用户能看见回退发生了。
```

---

## 5. 推荐执行顺序

建议按以下顺序推进：

```text
1. 语义开关先闭环
2. query embedding 接进主搜索
3. 混合检索与排序融合
4. 命中解释补全
5. 调试面板完善
6. 回退策略收口
```

这个顺序的原因：

```text
1. 先控制是否启用，避免影响主搜索体验。
2. 再让语义真正参与搜索。
3. 再做排序与解释，不然调试没有意义。
4. 最后把失败和回退收口，避免线上不可见问题。
```

---

## 6. 第二批完成标准

第二批完成后，至少应满足：

```text
1. 用户可以开启 / 关闭语义检索。
2. 开启后，关键词 + 语义结果能稳定混合。
3. 搜索结果能说明命中来源。
4. 调试面板能区分全文、语义、回退。
5. 语义异常时全文搜索仍然可用。
```

---

## 7. 阶段收口判断

如果第二批做到下面这些，就可以进入第三批：

```text
1. 混合检索稳定。
2. 结果解释可见。
3. 开关控制生效。
4. 回退策略清晰。
5. 调试面板能定位主要问题。
```

进入第三批后，再考虑：

```text
1. 更细的评分权重调参
2. 语义结果过滤阈值
3. 多 query 改写
4. 轻量 rerank
5. 搜索历史与收藏联动
```
