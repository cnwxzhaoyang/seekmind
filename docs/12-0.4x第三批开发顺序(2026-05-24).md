# SeekMind 0.4.x 第三批开发顺序

本文档承接 [0.4.x 第二批开发顺序](./11-0.4x第二批开发顺序(2026-05-24).md)，用于把 `0.4.x` 的最后一批能力拆成可执行顺序。

这一批不再做底座接入，也不再优先做主搜索链路改造，而是围绕下面这些已经“有了雏形，但还不够完整”的能力做收口：

```text
1. 评分权重调参
2. 语义阈值控制
3. 多 query 改写
4. 轻量 rerank
5. 搜索历史与收藏联动
6. 结果解释继续补细
```

---

## 1. 第三批目标

第三批只解决一个问题：

```text
语义检索已经能用了，现在要把结果排得更稳、更准、更可理解。
```

对应能力：

```text
1. 排序权重可调
2. 语义命中有阈值边界
3. query 能做轻量改写
4. 候选结果能做轻量 rerank
5. 历史、收藏、最近打开能参与结果偏好
6. 调试解释继续补全
```

这一批的核心不是“增加新能力”，而是“把现有能力调到可用的产品状态”。

---

## 2. 当前前提

进入第三批前，前两批应当已经具备：

```text
1. 语义开关可用
2. 混合检索可用
3. 命中来源可见
4. 调试面板可用
5. 语义失败能回退到全文检索
```

第三批默认这些底座已经存在，但不要求它们全部足够精细。第三批要做的是把“能用”推到“好用”。

---

## 3. 第三批范围

### 3.1 必做项

```text
1. 评分权重调参
2. 语义相似度阈值
3. query 改写的最小版本
4. 轻量 rerank
5. 搜索历史 / 收藏 / 最近文档的偏好加权
6. 调试面板继续细化
```

### 3.2 已有但不够完善的部分

这一类能力第三批优先补强：

```text
1. 结果已经能看见命中来源，但解释还不够细。
2. 混合检索已经能跑，但排序权重还没有稳定边界。
3. 语义开关已经存在，但不同模式下的体验还可再细化。
4. 历史、收藏、最近打开已经有数据，但还没真正参与排序。
5. 调试面板能看状态，但还不能直接回答“为什么这条排第一”。
```

### 3.3 明确不做

```text
1. 云端 embedding API
2. 大模型 rerank
3. 复杂 query planning
4. 多轮对话
5. 本地问答
6. OCR
7. 团队协作
```

---

## 4. 实施顺序

### 4.1 任务 A：先把评分权重调成可控

目标：

```text
让混合检索不只是“能合并”，而是“结果排序稳定、偏好明确”。
```

后端任务：

```text
1. 拆分关键词分、语义分、标题分、文件名分。
2. 引入可配置权重。
3. 让默认权重保持保守，避免语义压过精确命中。
```

前端任务：

```text
1. 设置页增加权重展示。
2. 展示当前权重档位或滑块。
3. 保存后可立即影响搜索结果。
```

验收标准：

```text
同一组结果在不同权重下排序会变化，但不会破坏精确关键词优先级。
```

---

### 4.2 任务 B：语义阈值收口

目标：

```text
把低相关语义结果挡在前排之外，避免“看起来像命中，其实不相关”。
```

后端任务：

```text
1. 为 semantic hit 增加阈值过滤。
2. 支持对不同查询强度使用不同阈值。
3. 低于阈值的结果仍可保留在调试层，但不进入主结果。
```

前端任务：

```text
1. 设置页展示阈值。
2. 调试面板说明哪些结果被阈值过滤。
3. 搜索结果能体现“未进入前排”的原因。
```

验收标准：

```text
明显不相关的语义结果不会挤进前排，召回更稳。
```

---

### 4.3 任务 C：query 改写先做最小版

目标：

```text
不要做复杂改写，先做轻量的查询规范化和同义补充。
```

后端任务：

```text
1. 对 query 做标准化清洗。
2. 对常见技术词做最小同义扩展。
3. 保留原 query 和改写后 query 的可见性。
```

前端任务：

```text
1. 调试面板展示原 query 与改写后 query。
2. 标明改写是否生效。
3. 不把改写结果直接隐藏掉。
```

验收标准：

```text
输入更口语化的表达时，系统能做最小增强，但不会让结果变得不可解释。
```

---

### 4.4 任务 D：轻量 rerank

目标：

```text
对前 N 个候选做更精细的重排，但不引入重模型。
```

后端任务：

```text
1. 只对前若干候选做 rerank。
2. rerank 只使用轻量规则或轻量模型。
3. rerank 失败时不影响主结果。
```

前端任务：

```text
1. 调试面板显示 rerank 是否启用。
2. 显示 rerank 前后名次变化。
3. 标出哪些结果是被 rerank 推上来的。
```

验收标准：

```text
前排结果更稳，但 rerank 失败不影响整体搜索。
```

---

### 4.5 任务 E：历史、收藏、最近文档参与偏好

目标：

```text
让结果排序更贴合用户真实使用习惯。
```

后端任务：

```text
1. 引入最近打开权重。
2. 引入收藏权重。
3. 引入搜索历史偏好权重。
4. 保证这些偏好只是加分，不覆盖核心相关性。
```

前端任务：

```text
1. 搜索结果显示偏好命中的标识。
2. 调试面板展示偏好来源。
3. 让用户知道某条结果是因为“常看 / 收藏 / 最近打开”被抬高。
```

验收标准：

```text
收藏和常用文档在相关查询里更容易靠前，但不会把无关结果硬顶上来。
```

---

### 4.6 任务 F：结果解释继续补细

目标：

```text
把“命中了”补成“为什么命中、为什么排前、为什么没进前排”。
```

后端任务：

```text
1. 返回关键词分、语义分、标题分、文件名分、偏好分。
2. 返回阈值过滤状态。
3. 返回 rerank 前后名次变化。
```

前端任务：

```text
1. 结果卡片展示得分构成。
2. 调试面板展示过滤与重排过程。
3. 对命中来源做更清晰的标签化展示。
```

验收标准：

```text
用户能看懂结果是怎么排出来的，不再只看到最终列表。
```

---

## 5. 推荐执行顺序

建议按下面顺序执行：

```text
1. 评分权重调参
2. 语义阈值收口
3. query 改写最小版
4. 轻量 rerank
5. 历史 / 收藏 / 最近文档偏好加权
6. 结果解释补细
```

这个顺序的原因：

```text
1. 先稳定基础排序。
2. 再控制语义噪声。
3. 然后做最小 query 增强。
4. 再补轻量重排。
5. 最后把用户偏好加进去。
6. 解释层放在最后收口。
```

---

## 6. 第三批完成标准

第三批完成后，至少应满足：

```text
1. 混合检索排序稳定。
2. 语义阈值能抑制噪声。
3. query 改写可见且可控。
4. 轻量 rerank 不影响主链路稳定性。
5. 历史 / 收藏 / 最近文档可影响排序但不越权。
6. 调试面板能解释排序构成。
```

---

## 7. 阶段收口判断

如果第三批做到下面这些，`0.4.x` 基本可以收口：

```text
1. 搜索结果稳定。
2. 语义能力不抢关键词精确命中。
3. 排序可调且可解释。
4. 用户偏好能参与但不失控。
5. 调试工具足够排查问题。
```

到这里，`0.4.x` 就可以进入“效果优化”和“边界功能打磨”，而不是继续扩搜索主链路。
