# SeekMind 0.3.x 第一批开发顺序

本文档把 [0.3.x 执行任务清单](./5-0.3x执行任务清单(2026-05-23).md) 再压缩成第一批最小可执行迭代，目标是先做“命中可见”，再做“结果可读”，最后再扩展到“聚合和排序”。

---

## 1. 第一批目标

第一批只解决一个问题：

```text
用户搜到结果后，能立刻看清楚关键词命中了哪里。
```

对应能力：

```text
1. 命中高亮
2. 更好的 snippet
3. 命中字段标签
4. 结果页调试信息
```

这一批不碰：

```text
1. 按文档聚合
2. 排序权重重构
3. 搜索历史
4. 收藏
5. 最近文档
```

原因很简单：

```text
先把“看得清”做出来，再做“看得顺”。
```

---

## 2. 第一批范围

### 2.1 后端

只做最小必要改动：

```text
1. 搜索结果返回命中字段来源。
2. 搜索结果返回 snippet 位置索引。
3. 搜索结果返回命中片段的原始文本。
4. 保留当前 chunk 级结果结构，不引入聚合层。
```

建议优先改动的模块：

```text
src-tauri/src/seekmind/storage/fulltext.rs
src-tauri/src/seekmind/storage/db.rs
src-tauri/src/seekmind/commands.rs
src-tauri/src/seekmind/models.rs
src-tauri/src/seekmind/search.rs
```

### 2.2 前端

只做结果展示相关改动：

```text
1. 搜索结果卡片支持高亮。
2. 右侧预览支持高亮。
3. snippet 支持展开与截断。
4. 显示命中文本来自标题、文件名还是正文。
```

建议优先改动的页面：

```text
src/pages/AppSearch.vue
src/components/seekmind/SeekMindSearchResultCard.vue
src/components/seekmind/SeekMindPageShell.vue
```

## 3. 任务拆分

### 3.1 任务 A：搜索结果高亮

目标：

```text
关键词一出现，用户就能在结果里看到。
```

后端任务：

```text
1. 输出命中字段。
2. 输出命中片段位置。
3. 输出原始片段文本。
```

前端任务：

```text
1. 按命中位置渲染高亮。
2. 支持多关键词高亮。
3. 保证高亮和深浅背景都能看清。
```

验收标准：

```text
搜索“马斯克”时，结果卡片里直接能看到“马斯克”被高亮，而不是只看到一段普通文本。
```

---

### 3.2 任务 B：snippet 改进

目标：

```text
snippet 不再只是截断文本，而是优先展示命中前后上下文。
```

后端任务：

```text
1. 从 chunk 文本中提取命中前后上下文。
2. 对多命中场景取命中密度更高的片段。
3. 返回可直接展示的 snippet。
```

前端任务：

```text
1. snippet 默认显示一段更短、更聚焦的上下文。
2. 长内容支持展开。
3. 与高亮样式一致。
```

验收标准：

```text
用户不用打开结果，也能大致看出该结果讲的是不是自己要找的内容。
```

---

### 3.3 任务 C：命中字段标签

目标：

```text
让用户知道命中来自哪里。
```

后端任务：

```text
1. 标记命中来源为文件名、标题或正文。
2. 返回一个稳定的字段标签。
```

前端任务：

```text
1. 在结果卡片上显示“文件名命中 / 标题命中 / 正文命中”。
2. 在多字段命中时显示优先命中字段。
```

验收标准：

```text
用户能快速判断是“标题里提到过”还是“正文里真正讲过”。
```

---

### 3.4 任务 D：最小调试信息

目标：

```text
保留可解释性，避免高亮做完之后不好排查。
```

后端任务：

```text
1. 保留当前 debug report。
2. 增加命中字段和 snippet 起止位置。
3. 保留命中数和查询归一化信息。
```

前端任务：

```text
1. 搜索页显示最小调试卡片。
2. 保留 Tantivy 文档数和命中数。
3. 在调试面板中显示当前查询归一化结果。
```

验收标准：

```text
出现“搜不到”或“高亮不对”时，能在调试面板里快速确认问题在哪一层。
```

---

## 4. 第一批不做的事

这一批明确不做：

```text
1. 按文档聚合
2. 同文档折叠展开
3. 排序权重调整
4. 搜索历史
5. 收藏
6. 常用目录入口
7. 最近查看文档
```

原因：

```text
这些能力都会增加信息量和交互复杂度，应该放到第二批。
```

## 6. 第一批验收测试点

### 6.1 视觉验收

```text
1. 结果卡片中的命中词可见。
2. 高亮在浅色背景下仍然清晰。
3. snippet 内容能直接看出上下文。
```

### 6.2 功能验收

```text
1. 搜索一个明确存在的关键词能看到高亮。
2. 搜索多个关键词时高亮仍然正确。
3. 调试面板可辅助判断命中来源。
```

### 6.3 回归验收

```text
1. 现有搜索结果仍然能打开原文件。
2. 切片页不受影响。
3. 索引状态页不受影响。
```

---

## 7. 推荐落地顺序

建议按下面顺序推进：

```text
1. 后端返回命中字段和位置
2. 前端实现高亮渲染
3. 调整 snippet 生成
4. 补最小调试信息
5. 回归验证
```

这条顺序的好处是：

```text
先把数据链路接通，再打磨展示层，不容易返工。
```

---

## 8. 第一批结束标准

第一批完成时，应满足：

```text
1. 用户看到结果时，能立即知道关键词命中了哪里。
2. snippet 更像“命中摘要”，而不是随机截断文本。
3. 调试面板还能解释为什么命中或不命中。
4. 不影响现有索引和切片功能。
```

达到这个标准后，再进入第二批：

```text
按文档聚合 + 折叠展示 + 排序增强。
```
