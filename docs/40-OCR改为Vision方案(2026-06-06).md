# DocMind OCR 改为 Vision 方案（2026-06-06）

## 背景

当前 DocMind 的 PDF OCR 已经接入了 sandbox 版打包链路，但底层仍然依赖 `tesseract` 可执行文件和语言包。对于 macOS 沙盒应用，这种外部二进制依赖在开发环境可用、在打包环境不稳定，且不符合“尽量使用系统能力”的方向。

macOS 自带 Vision 框架提供了本地文本识别能力，可以直接在应用内完成 OCR，不依赖系统外部进程，也不依赖 tessdata 语言包，更适合保留 sandbox 的桌面应用。

## 目标

1. 保留 macOS sandbox，不再依赖外部 `tesseract` 进程。
2. 将 PDF OCR 切换为 macOS Vision 框架。
3. OCR 仍然保持现有的 Python 侧解析编排和 PDF 扫描页队列逻辑。
4. OCR 结果、错误、进度、状态页提示保持可观测。
5. 不改变当前 PDF chunk、OCR 任务表、RAG、索引入库的主链路。

## 方案概述

采用“**macOS Vision OCR helper binary**”的方式实现：

1. 由 Rust 编译一个 macOS-only 的 OCR helper，可执行文件随 App 一起打包。
2. helper 内部调用 Vision 框架识别图片文字。
3. Python sidecar 继续负责 PDF 解析、扫描页判定、OCR 队列组织。
4. Python 在需要 OCR 时调用 helper binary，而不是调用系统 `tesseract`。
5. Rust 侧状态探测也切到 Vision helper 的可用性探测。

这样可以保持：

- sandbox 不关闭
- 不调用外部系统进程
- 现有 PDF 解析流程变化最小
- OCR 能力与 App 一起交付

## 运行时链路

### 1. PDF 解析

Python sidecar 继续执行：

- 文本页直接切块
- 扫描候选页进入 OCR 队列
- 继续保存 `pdf_ocr_tasks`
- 继续回传进度事件给 Rust

### 2. OCR 执行

Python 对每个 OCR 任务调用 bundled helper：

- 输入：图片文件路径
- 输出：识别文本，必要时附带语言、耗时、状态
- 不再调用 `tesseract`

### 3. Rust 状态探测

Rust 在状态页里探测 bundled Vision OCR helper 是否可用：

- helper 是否存在
- helper 是否能运行
- OCR 可用时展示明确状态

### 4. 打包

macOS 构建时：

- 编译 Vision OCR helper
- 将 helper 复制到 `app-resources` 或 bundle 资源目录
- Python sidecar 启动时通过环境变量拿到 helper 路径

## 接口建议

### 环境变量

建议继续沿用“由 Rust 注入给 Python”的模式，但改为：

- `DOCMIND_VISION_OCR_BIN`
- `DOCMIND_VISION_OCR_ENABLED`

如果后续需要，也可以附加：

- `DOCMIND_VISION_OCR_LANGS`

### helper 命令行

建议 helper 支持最少一条命令：

```bash
docmind-vision-ocr --image /path/to/page.png --langs chi_sim+eng
```

输出建议为纯文本或 JSON。优先建议 JSON，便于后续扩展：

```json
{
  "ok": true,
  "text": "识别结果",
  "elapsed_ms": 123,
  "provider": "vision"
}
```

## 实现顺序

1. 新增 macOS Vision OCR helper binary。
2. 将 helper 复制到打包资源目录。
3. Python parser 改为优先调用 Vision helper。
4. Rust 状态探测改为 Vision helper 探测。
5. 删除 `tesseract` / `tessdata` 相关打包逻辑与状态文案。
6. 验证 PDF OCR、扫描版 PDF、OCR 任务重跑、状态页提示。

## 完成标准

- 打包后的 macOS sandbox 应用中，PDF OCR 不再依赖系统外部 `tesseract`。
- 开发环境和打包环境都可以正常识别扫描版 PDF。
- OCR 队列、进度、状态页、任务重跑功能保持可用。
- 代码中不再出现 OCR 依赖外部进程的主路径。

## 不纳入本次改造的内容

- 不改 PDF chunk / RAG 主流程。
- 不改 OCR 任务表结构。
- 不改前端 PDF 解析页面布局。
- 不引入新的后端服务进程。
