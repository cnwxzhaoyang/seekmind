# SeekMind UI 原型对齐实施方案

## 1. 背景

当前项目已经具备完整功能，包括统一搜索、AI 问答、知识库、切片浏览、索引状态和设置等能力，但整体 UI 仍然偏“功能型面板”，与 `ui-prototype/seekmind-v2.html` 的桌面原型在视觉统一性、层级组织和操作密度上存在差异。

本次改造目标不是重做功能，而是把现有功能壳层统一到原型风格下，同时保留现有路由、数据流、事件处理和 i18n 能力。

## 2. 改造目标

1. 视觉上对齐原型：
   - 顶部 macOS 风格标题栏
   - 左侧固定导航
   - 中央工作区采用分栏/卡片式布局
   - 右侧详情面板采用轻量信息展示
   - 统一使用更接近原型的圆角、阴影、留白和按钮密度

2. 功能上不丢失：
   - 现有路由全部保留
   - 现有搜索、问答、知识库、切片、状态、设置功能不删除
   - 现有 API 调用、事件监听、收藏/历史/目录树/上下文菜单等交互继续可用

3. 结构上更易维护：
   - 避免把原型直接“拍扁”进单个页面文件
   - 用公共壳和可复用面板组件承接页面变化
   - 保持 Vue 文件体量可控，避免单文件继续膨胀

4. 国际化保持可用：
   - 继续保留中英文切换
   - 原型中出现的文案、按钮提示、空状态都要提供 i18n 键

## 3. 参考原型的核心特征

原型 `ui-prototype/seekmind-v2.html` 的核心特征可以拆成四层：

1. 壳层：
   - 顶部标题栏
   - 左侧导航
   - 中央主内容区
   - 右侧信息区

2. 搜索层：
   - 圆角搜索胶囊
   - 类型切换 segmented control
   - 筛选按钮组
   - 搜索结果卡片强调“匹配度、来源、快捷动作”

3. 详情层：
   - 右侧预览面板
   - 高亮上下文
   - 快捷操作按钮
   - 轻量信息卡片

4. 桌面层：
   - 窗口感更强
   - 视觉更轻，边框更少
   - 主色统一为蓝色系
   - 图标风格统一，可直接复用原型中的图标命名

## 4. 现有功能保留范围

以下能力必须原样保留，改造只改 UI 结构，不动业务能力：

1. 路由与页面：
   - `/` 统一搜索
   - `/qa` AI 问答
   - `/collections` 知识库
   - `/chunks` 切片浏览
   - `/status` 索引状态
   - `/settings` 设置

2. 搜索能力：
   - 文档/片段/会话/集合混合搜索
   - 结果筛选、收藏、加入集合、基于结果提问
   - 调试面板、上下文预览、引用复制、快速打开

3. 问答能力：
   - 会话管理
   - 回答流式展示
   - 引用来源展示
   - 会话导出/复制/重命名/删除

4. 知识库能力：
   - 集合管理
   - 条目管理
   - 标签、备注、最近访问
   - 上下文菜单和详情编辑

5. 切片与索引能力：
   - 目录树
   - 文档列表
   - 切片预览
   - 重新索引、OCR 刷新、打开、快速查看
   - 索引进度与任务状态

6. 设置能力：
   - 索引规则
   - 语义检索参数
   - AI 回答配置
   - 网络代理
   - 主题与语言切换

## 5. UI 改造总方案

### 5.1 应用壳层重构

建议把当前 `SeekMindPageShell` 改为“原型式桌面壳”：

1. 顶部区域：
   - 增加 macOS 风格标题栏
   - 左侧保留窗口装饰点
   - 中间显示当前模块标题
   - 右侧放全局动作图标，例如快捷命令、通知、全局新增入口

2. 左侧区域：
   - 保留导航入口，但视觉上贴近原型
   - 采用固定宽度导航
   - 当前高亮使用原型的蓝色强调方式

3. 中央区域：
   - 所有页面都在统一壳中渲染
   - 页面内部只负责内容，不再各自重复一套壳层

4. 底部日志/状态：
   - 继续保留日志面板
   - 但视觉上要更轻，默认不抢占主体视觉

### 5.2 页面布局策略

1. 统一搜索页：
   - 最接近原型，应优先作为风格基准页
   - 左侧结果列表 + 右侧详情预览
   - 顶部搜索胶囊、segmented、筛选按钮、统计卡片全部对齐原型风格

2. AI 问答页：
   - 保留聊天流式布局
   - 视觉语言跟原型统一，减少厚重边框
   - 右侧引用/来源/会话信息面板采用轻量卡片

3. 知识库页：
   - 继续三栏或双栏布局
   - 统一卡片样式、列表密度和操作按钮位置
   - 重点统一集合卡片、条目卡片、标签展示

4. 切片页：
   - 用“目录树 + 文档列表 + 预览详情”组织内容
   - 视觉上更像原型里的索引/预览区，而不是传统管理后台

5. 状态页：
   - 保留统计与异常摘要
   - 变成原型中的“信息看板”风格

6. 设置页：
   - 继续使用分区设置
   - 头部、分组标题、表单密度统一到原型风格

## 6. 组件拆分方案

为了避免 UI 改造后页面文件过大，建议同步做组件层拆分：

1. 新增公共壳组件：
   - `SeekMindTopBar`
   - `SeekMindAppNav`
   - `SeekMindWorkspaceShell`
   - `SeekMindPrototypeCard`
   - `SeekMindToolbarButton`
   - `SeekMindSegmentedControl`

2. 现有复用组件保留并统一样式：
   - `SeekMindBadge`
   - `SeekMindContextMenu`
   - `SeekMindIndexTree`
   - `SeekMindSearchResultCard`
   - `SeekMindSearchResultGroupCard`
   - `SeekMindFileIcon`
   - `SeekMindPreviewBlockRenderer`

3. 页面内部继续拆分：
   - 搜索页的结果列表、详情面板、调试面板拆开
   - 问答页的聊天区、来源区、会话区拆开
   - 设置页的分组继续保持组件化

## 7. 视觉规范

### 7.1 色彩

1. 主色：
   - 继续使用蓝色作为主强调色
   - 不引入新的冲突主题色

2. 背景：
   - 采用浅灰蓝/浅灰白的桌面背景
   - 内容区与壳层区分但不做强分割

3. 卡片：
   - 低对比边框
   - 轻阴影
   - 更大的圆角

### 7.2 字体与排版

1. 保持中英文混排友好
2. 统一标题、正文、元信息字号
3. 列表和树形文字长度超出后保留 tooltip
4. 关键标题和副标题尽量同一行显示，避免上下堆叠

### 7.3 图标

1. 可直接复用原型中的 icon 命名和图标风格
2. 统一为轻线性图标
3. 按钮图标必须语义明确，贴近功能主题

## 8. 交互规范

1. 尽量减少显式按钮堆叠：
   - 常用二级操作优先放右击菜单
   - 页面主按钮保留 1 到 2 个

2. 列表与卡片：
   - 支持选中态
   - 支持 hover 态
   - 支持键盘可达

3. 详情区：
   - 采用右侧上下文预览方式
   - 让“选中对象”与“详细信息”天然分离

4. 空状态和加载态：
   - 统一风格
   - 不能出现硬切换和布局抖动

## 9. 国际化方案

当前项目已有中英文切换能力，本次 UI 改造需要同步补齐：

1. 所有新增标题、按钮、空态、提示文案必须放入 i18n
2. 原型文案如果只是视觉占位，也要映射到真实业务文案
3. 新增组件不得写死中文
4. 若新增图标按钮仅用于装饰，也要提供 `title`/`aria-label`

## 10. 分阶段实施建议

### 第一阶段：壳层对齐

1. 重做顶栏
2. 重做左侧导航
3. 统一全局背景、边框、圆角、阴影
4. 保持现有路由可正常切换

### 第二阶段：搜索页优先对齐

1. 统一搜索页变成原型式主页面
2. 结果列表、右侧预览、筛选区重排
3. 结果卡片改造成更接近原型的视觉

### 第三阶段：其余页面统一

1. 问答页
2. 知识库页
3. 切片页
4. 状态页
5. 设置页

### 第四阶段：细节打磨

1. 空状态
2. 加载态
3. tooltip
4. 右击菜单
5. 中英文文案统一

## 11. 风险与约束

1. 风险：原型偏静态，现有页面偏功能型
   - 处理：以原型为视觉参照，以当前业务页面为交互骨架

2. 风险：页面文件可能继续变大
   - 处理：必须拆公共壳和局部组件，避免直接堆在单页面里

3. 风险：部分功能入口在视觉上被弱化
   - 处理：保留功能入口，只调整层级和布局位置

4. 风险：国际化遗漏
   - 处理：凡是新增 UI 文案都必须先补 i18n 键

5. 风险：切换原型风格后影响现有事件绑定
   - 处理：优先复用现有业务组件，只替换外层容器和样式

## 12. 验收标准

1. UI 整体风格与 `ui-prototype/seekmind-v2.html` 明显一致
2. 现有页面和功能入口全部可用
3. 中英文切换正常
4. 搜索、问答、集合、切片、状态、设置页都能正常访问
5. 右击菜单、快捷操作、预览、复制、打开等交互未丢失
6. 页面没有因为改造出现单文件过长不可维护的问题

## 13. 建议的执行顺序

1. 先改公共壳层
2. 再改统一搜索页
3. 再统一可复用卡片与按钮
4. 最后逐页迁移其余功能页

## 14. 结论

本次改造应采用“壳层统一 + 页面分步迁移 + 功能不动”的方式推进。这样既能把项目外观拉到原型层级，也能避免一次性重构带来的功能回退风险。
