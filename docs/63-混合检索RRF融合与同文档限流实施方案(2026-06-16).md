# 混合检索 RRF 融合与同文档限流实施方案

## 1. 背景

当前 SeekMind 的混合检索已经具备两条召回链路：

- `Tantivy` 关键词召回
- 语义向量召回

但现有合并方式仍然偏向“关键词结果先入队，再追加语义结果”，本质上更接近拼接去重，而不是严格的 rank fusion。  
当关键词结果很多时，语义专属命中的 chunk 会被压到很后面，最终页面 TopN 看不到，容易误判为“语义失效”。

从开发期 trace 日志也可以看到：

- `keyword_top_k` 有结果
- `vector_top_k` 有结果
- `merged` 里也有 vector-only 候选
- 但最终展示仍会被 keyword 主导

因此，本次要做的不是扩大召回，而是把“合并顺序”改成“融合排序”。

## 2. 目标

1. 将混合检索从“关键词优先拼接”改成真正的 RRF 融合排序。
2. 保留 keyword-only、vector-only、keyword+vector 三类候选。
3. 在最终 TopN 中增加同文档限流，避免单个文档刷屏。
4. 保留并强化开发期 trace 日志，便于验证融合前后差异。
5. 不改变前端交互，只调整后端检索排序与 debug 输出。

## 3. 当前问题

### 3.1 合并逻辑偏关键词优先

当前代码流程大致是：

```text
keyword_hits -> chunk_ids
semantic_hits -> append vector-only chunk_ids
fetch rows
sort by current score
```

这会导致：

- 关键词结果天然排在前面
- vector-only 结果即使召回到了，也会被排到后面
- 文档量变大后，语义命中的 chunk 更容易被挤出最终结果

### 3.2 同一文档占用过多前排

关键词命中往往集中在同一文档的多个 chunk 上。  
如果不做同文档限流，Top10 / Top20 可能被同一文档占满，用户会觉得结果不够多样。

### 3.3 当前“总分”并不等于可解释的融合分

现有 `score` 里混入了：

- keyword / semantic 基础分
- chunk 权重
- 标题/文件名加分
- 偏好加分
- 时间加分

这种分数可以用于排序，但不适合直接当作“融合策略”的最终解释。  
需要在 trace 中额外打印 `keyword_rank`、`vector_rank`、`rrf_score`、`doc_limit` 命中情况。

## 4. 方案

### 4.1 召回阶段不变

保留两路召回：

- 关键词召回：`Tantivy`
- 语义召回：embedding cosine 相似度

召回 TopK 可以继续保留较大的候选池，确保不因为 TopK 太小而截断语义结果。

### 4.2 候选阶段改为按 chunk_id 合并

对 keyword / vector 两路结果按 `chunk_id` 合并成统一候选池，每个候选保留：

- `keyword_rank`
- `keyword_score`
- `vector_rank`
- `vector_score`
- `match_source`

其中：

- 只命中 keyword：`keyword-only`
- 只命中 vector：`vector-only`
- 两边都命中：`keyword+vector`

### 4.3 排序阶段使用 RRF

建议使用 Reciprocal Rank Fusion（RRF）作为主融合策略：

```text
rrf_score = keyword_weight * 1 / (k + keyword_rank)
          + vector_weight * 1 / (k + vector_rank)
```

建议初始参数：

- `k = 60`
- `keyword_weight = 0.7`
- `vector_weight = 1.0`

如果后续希望语义更强，可再调高 `vector_weight`。  
RRF 的优点是：

- 不依赖 keyword score / vector score 的量纲一致性
- 只看 rank，稳定性更好
- 能让 vector-only 的高排名结果自然进入前排

### 4.4 最终展示增加同文档限流

在最终 TopN 输出前，对同一 `document_id` 做限流，建议默认：

- 每个文档最多保留 2 到 3 个 chunk

这样可以：

- 避免单个文档刷屏
- 提升结果多样性
- 更容易让语义命中的其他文档露出

如果某个文档的多个 chunk 都很强，可以保留最强的前几个，其余作为候补。

### 4.5 trace 日志增加 final_ranked

在现有日志基础上新增：

- `final_ranked:`

每条输出建议包含：

- `chunk_id`
- `source`
- `keyword_rank`
- `vector_rank`
- `rrf_score`
- `final_score`
- `doc_id`

这样可以直接看到：

- 候选是怎么被融合的
- 为什么某个 vector-only chunk 没进前排
- 同文档限流是否生效

## 5. 实施顺序

### 5.1 第一阶段：融合排序

优先把 `merged` 改成真正的候选融合，不再让 keyword 顺序天然决定 final 顺序。

### 5.2 第二阶段：同文档限流

在 final 阶段增加文档级 cap，保证结果多样性。

### 5.3 第三阶段：日志增强

补充 `final_ranked` 和同文档分布信息，便于开发期回归验证。

## 6. 验收标准

以下情况都应满足：

1. 关键词和语义都能独立召回。
2. vector-only 的高 rank 结果不会被 keyword 结果完全压住。
3. 同一文档不会在 TopN 里无限刷屏。
4. trace 日志能清楚打印 `keyword_top_k`、`vector_top_k`、`merged`、`after_filter`、`final_ranked`。
5. 搜索结果在大文档库下仍保持可解释、多样且稳定。

## 7. 非目标

本次不做以下事情：

- 不改前端搜索面板结构
- 不改语义模型或 embedding 生成方式
- 不改文档切片和索引流程
- 不改用户可见的搜索文案
