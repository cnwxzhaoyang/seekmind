# DocMind 0.4.x 第一批开发顺序

本文档把 [0.4.x 语义搜索实施拆解](./7-0.4x语义搜索实施拆解(2026-05-23).md) 再压缩成第一批最小可执行迭代，目标不是立刻把语义搜索接入主链路，而是先把 embedding 生成、向量存储、模型状态和调试闭环打通。

---

## 1. 第一批目标

第一批只解决一个问题：

```text
DocMind 能稳定把 chunk 变成向量，并且能看到当前语义能力到底有没有准备好。
```

对应能力：

```text
1. 本地 embedding 模型状态展示
2. chunk embedding 生成
3. 向量存储和删除
4. query embedding 调试
5. 最小语义调试面板
```

这一批不做主搜索融合，原因很直接：

```text
先把“向量能稳定生成、稳定存、稳定查”做出来，再把它接进关键词检索。
```

---

## 2. 第一批范围

### 2.1 后端

只做最小必要改动：

```text
1. 增加本地 embedding 模型状态模型。
2. 增加 chunk embedding 生成与缓存。
3. 增加向量索引存储抽象。
4. 增加 query embedding 调试接口。
5. 增加语义调试报告接口。
6. 不改主搜索链路，只做独立语义能力底座。
```

建议优先改动的模块：

```text
src-tauri/src/docmind/semantic/mod.rs
src-tauri/src/docmind/semantic/models.rs
src-tauri/src/docmind/semantic/embedding.rs
src-tauri/src/docmind/semantic/vector_store.rs
src-tauri/src/docmind/semantic/commands.rs
src-tauri/src/docmind/storage/db.rs
src-tauri/src/docmind/commands.rs
src-tauri/src/lib.rs
```

### 2.2 前端

只做语义状态和调试展示：

```text
1. 设置页展示当前 embedding 模型状态。
2. 索引状态页展示语义索引是否就绪。
3. 增加语义调试面板。
4. 保留全文搜索主流程，不切换主搜索结果。
```

建议优先改动的页面：

```text

```

## 3. 任务拆分

### 3.1 任务 A：本地 embedding 模型状态

目标：

```text
用户一眼能看出当前语义搜索用的是哪个模型，模型是否可用，是否需要重建向量。
```

后端任务：

```text
1. 增加模型元数据结构。
2. 记录模型名称、路径、维度、启用状态。
3. 暴露模型可用性检查接口。
4. 暴露当前默认模型接口。
5. 暴露模型切换后的重建提示状态。
```

前端任务：

```text
1. 设置页展示当前 embedding 模型。
2. 展示模型维度和可用性。
3. 展示是否需要重建向量。
4. 保留切换入口但不强制联动主搜索。
```

数据库任务：

```text
1. 新增 embedding_models 表。
2. 记录默认模型和可用状态。
```

验收标准：

```text
用户在设置页能看到“当前语义模型是什么、是否可用、是否需要重建”。
```

---

### 3.2 任务 B：chunk embedding 生成与缓存

目标：

```text
把已有 chunk 稳定转成向量，并且同样内容不会重复生成。
```

后端任务：

```text
1. 从 chunk 表读取待编码文本。
2. 对 chunk 内容生成 embedding。
3. 记录 text_hash，避免重复计算。
4. 支持单 chunk 重建。
5. 支持失败重试。
```

前端任务：

```text
1. 在索引状态页显示语义索引进度。
2. 显示 chunk embedding 成功数和失败数。
3. 对单个文档显示是否已经生成向量。
```

数据库任务：

```text
1. 新增 chunk_embeddings 表。
2. 记录 chunk_id、model_id、vector_blob、dimension、text_hash、status。
```

验收标准：

```text
相同内容不会重复生成 embedding；改动 chunk 内容后能重新生成。
```

---

### 3.3 任务 C：向量存储最小闭环

目标：

```text
让语义向量可检索、可删除、可重建。
```

后端任务：

```text
1. 选定本地向量存储实现。
2. 提供 add / upsert / delete / search 接口。
3. 和 chunk id 一一对应。
4. 目录删除、文档删除时同步删除向量。
5. 支持整库重建。
```

前端任务：

```text
1. 索引状态页显示向量存储是否就绪。
2. 显示向量索引的 chunk 数量。
3. 显示最近一次向量重建时间。
```

数据库任务：

```text
1. 新增 vector_index_meta 表。
2. 记录向量索引状态、chunk 数、最后错误。
```

验收标准：

```text
删除文档后，向量存储里不会残留旧 chunk；重建后数量能对上。
```

---

### 3.4 任务 D：query embedding 调试

目标：

```text
先验证 query 向量能稳定生成，不直接接入主搜索链路。
```

后端任务：

```text
1. 复用同一 embedding 模型生成 query 向量。
2. 增加 query embedding 调试接口。
3. 返回 query 归一化结果、向量维度、是否成功。
4. 失败时只影响语义调试，不阻断全文搜索。
```

前端任务：

```text
1. 语义调试面板显示 query embedding 是否成功。
2. 显示 query 向量维度。
3. 显示 query 归一化后的文本。
```

验收标准：

```text
输入一个查询词，能看到它被转成了向量，并且全文搜索不受影响。
```

---

### 3.5 任务 E：最小语义调试面板

目标：

```text
让当前语义能力状态可见、可解释、可排查。
```

后端任务：

```text
1. 增加语义调试报告接口。
2. 返回模型状态、embedding 计数、向量索引计数、query embedding 结果。
3. 保留最小错误信息。
```

前端任务：

```text
1. 在状态页或设置页增加语义调试卡片。
2. 展示模型、向量数、最近一次错误、query 生成结果。
3. 提供重建提示。
```

验收标准：

```text
一眼能看出语义层是不是可用，以及问题卡在模型、embedding 还是向量存储。
```

## 4. 第一批不做的事

这一批明确不做：

```text
1. 语义检索接入主搜索结果。
2. 关键词 + 语义混合排序。
3. rerank。
4. query 改写。
5. 仅语义模式。
6. 多模型自动切换。
7. 云端 embedding API。
8. 本地问答。
```

原因：

```text
先把底座打通，再谈搜索体验。语义层没稳定之前，贸然接入主搜索只会放大问题。
```

---

## 5. 第一批推荐顺序

建议按下面顺序执行：

```text
1. 本地 embedding 模型状态
2. chunk embedding 生成与缓存
3. 向量存储最小闭环
4. query embedding 调试
5. 最小语义调试面板
```

这个顺序的原因：

```text
先确认模型能用，再确认 chunk 能编码，再确认向量能存，最后才看 query 和调试。
```

---

## 6. 第一批验收清单

第一批完成后，至少应满足：

```text
1. 能看到当前 embedding 模型状态。
2. 能把已有 chunk 生成向量。
3. 向量数据能持久化。
4. 删除文档后向量可同步删除。
5. query 能独立生成向量。
6. 能看到语义调试信息。
7. 全文搜索主流程不受影响。
```

---

## 7. 风险控制

第一批最主要的风险是：

```text
1. 模型太大导致本地跑不动。
2. 向量存储和 chunk id 对不齐。
3. 语义调试信息不足，问题不好定位。
4. 过早接入主搜索导致结果质量下降。
```

规避方式：

```text
1. 默认选轻量模型，优先 384 维。
2. 先做独立调试接口，再接主搜索。
3. 保留全文检索作为保底。
4. 语义能力未就绪前不要默认开启混合检索。
```

---

## 8. 阶段收口判断

如果第一批做到下面这几项，就可以进入 `0.4.x` 第二批：

```text
1. 模型状态可见。
2. chunk embedding 可生成。
3. 向量可持久化。
4. query 可调试。
5. 语义层可观测。
```

进入第二批后，再开始做：

```text
1. 语义检索接入主搜索。
2. 混合排序。
3. 语义开关。
4. 语义结果解释。
```
